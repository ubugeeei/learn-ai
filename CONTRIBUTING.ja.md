# コントリビューションガイド

## 1 コミットにつき 1 つの学習概念

Conventional Commit のタイトルを使います。1 つのコミットは、独立して
コンパイル・テスト・説明できる 1 つの概念を導入するべきです。実装、テスト、
対応するハンズオン章を一緒に保ちます。

例:

```text
feat(autodiff): implement scalar reverse-mode differentiation
test(attention): verify future tokens receive no prefix gradient
docs(inference): derive nucleus sampling
```

## ドキュメンテーションコメント

公開型と自明でない公開操作には、メソッド名の言い換えではなく、契約を
説明する英語の Scaladoc を付けるべきです。関連する項目を含めます。

- 数学的な意味と入出力の形状
- 構築によって確立される不変条件
- ミューテーションと所有権のルール
- 決定性と乱数状態の所有権
- 失敗条件と、返されるエラーチャネル
- 設計に影響する場合の、計算量やアロケーションの挙動
- プロダクションシステムと比較した教育上の簡略化

実装コメントは、数値、セキュリティ、インデックスに関する選択がなぜ存在
するのかを説明すべきです。コードの 1 行を単に散文へ言い換えるだけの
コメントは避けます。

## ハンズオン章の深さ

実装済みの章は `docs/00-guide/04-chapter-anatomy.md` に従います。概念的な
概要だけでは、完全なドキュメントではありません。コードの変更とともに、
次を保ちます。

- 手計算できる実例
- 数式/型/形状の対応表
- ソースファイルと実行パスのウォークスルー
- 明示的な不変条件、所有権、ミューテーション、エラー挙動
- テストオラクルの説明とデバッグチェックリスト
- 出力の解釈を伴う実行可能なコマンド
- 限界、プロダクションとの違い、次の依存先

幅広い背景説明の散文よりも、実際の Scala コードの正確な説明を優先します。
学習者が、説明されていないループをリバースエンジニアリングすることなく、
章から実装を再構築できるべきです。

`DocumentationStructureSuite` は、実装済みの章のディレクトリに対して、
共通のウォークスルー、読解、デバッグの各セクションを強制します。これは
リグレッションの最低ラインであり、技術的な説明のレビューの代わりには
なりません。

## テスト

メインの学習パスでは、外部のテストライブラリを使いません。新しいスイート
はすべて `learnai.testing.AllTests` に登録し、ハッピーパス以外もカバーします。

概念に応じて、次を含めます。

- 手計算できる例
- 空、ゼロ、最大、不正な入力
- 有用なメッセージを伴う形状/範囲/型のエラー
- 固定シードの下での決定性
- 代数的性質とラウンドトリップ
- 有限差分による勾配 (gradient) チェック
- 最適化実装と参照実装の等価性
- 破損、タイムアウト、予算、権限の失敗
- 焦点を絞った単体テストを置き換えない、エンドツーエンドの統合テスト

説明のつかない数値差を隠すためだけに、許容誤差を緩めてはいけません。
予想される誤差の原因を記録し、計算のスケールから導かれた許容誤差を選び
ます。

## 必須チェック

ピン留めされた環境からリポジトリを実行します。

```console
nix develop -c sbt check
git diff --check
```

実行可能な章を変更する場合は、その `runMain` コマンドも実行し、観察された
損失 (loss)、出力、指標を確認します。ウォームアップ済みのベンチマークと
記録された環境なしに、性能を主張してはいけません。

## 依存関係ポリシー

コアのハンズオンは、Scala 3 と JDK の API のみを使います。まず仕組みを
実装します。外部ライブラリは、そのコスト、隠れた挙動、バージョンのピン留め
を文書化した上で、後から分離された比較用アダプタとして登場することが
あります。
