# 23 — 温度・top-k・top-p サンプリング

## この章で作るもの

温度 (temperature)、top-k、nucleus top-p のフィルタリングを適用し、確率を
再正規化し、呼び出し側が所有する乱数生成器からサンプリングする、独立した
デコーディングポリシーを実装します。ソース:
`src/main/scala/learnai/lm/Sampling.scala`。

## デコーディングは独立した意思決定の層

モデルは次トークンのロジットを出力します。実際のトークンを選ぶのは
デコーディングポリシー (decoding policy) です。同じモデルと同じプロンプト
でも、ポリシーが異なれば、決定性、多様性、繰り返し、失敗率が異なる結果に
なり得ます。

```text
logits
 -> temperature
 -> softmax
 -> top-k
 -> renormalize
 -> top-p
 -> renormalize
 -> sample
```

ポリシーとシードは、生成の再現性の一部です。

## 貪欲デコーディング

$$
x_{t+1}=\operatorname*{arg\,max}_i z_i
$$

貪欲 (greedy) な出力は決定的ですが、繰り返しに陥ったり、局所的に尤もらしい
だけの悪い続きを選んだりすることがあります。`k=1` の top-k は貪欲
デコーディングと等価です。

## 温度

$$
p_i=\operatorname{softmax}(z_i/\tau),\quad\tau>0
$$

- 低い温度はロジットの差を拡大し、エントロピーを下げます。
- `1` はモデルの分布をそのまま保ちます。
- 高い温度は分布を平坦にし、多様性を高めます。

温度ゼロは未定義です。ゼロで割るのではなく、貪欲デコーディングを明示的に
表現してください。

## Top-k

確率の高い上位 $k$ 個のトークンを残し、残りをゼロにして再正規化します。

$$
\tilde p_i=\begin{cases}
p_i/Z&i\in K\\
0&\text{otherwise}
\end{cases}
$$

確信度にかかわらず候補数は固定になります。`k` が語彙サイズを超える場合は、
すべてのトークンが残ります。確率が等しい場合は、決定的なタイブレークとして
小さいトークン ID を使います。

## Nucleus top-p

確率でソートし、累積確率が閾値 $p$ に達する最小の接頭辞を保持します。

```text
probabilities: [0.60,0.25,0.10,0.05]
top-p = 0.80 -> keep [0.60,0.25], cumulative 0.85
```

確信度の高い分布では候補が少なく、平坦な分布では多く残ります。少なくとも
1 つのトークンは常に保持されます。

## フィルタリングの順序は契約の一部

この実装では温度、次に top-k、次に top-p を適用します。top-k は nucleus
フィルタリングの前に再正規化されます。操作を逆順にすると結果が変わり得ます。

サービング API を比較するときは、次の点を確認してください。

- top-p/top-k の順序。
- 境界値の包含。
- ロジットバイアスとペナルティの順序。
- 保持されるトークンの最小数。
- 乱数生成器とシードのセマンティクス。

## 評価

モデルの尤度と、サンプリングされた 1 つの応答は分けて測定します。生成
タスクには複数のシードと、正しさ、多様性、繰り返し、レイテンシ、危険な
出力の割合といった指標が必要です。

厳密さが求められるタスクには貪欲や低い温度が向くことがあります。創造的な
タスクには適度なエントロピーが有効なことがあります。影響の大きいツール
操作には、デコーディングにかかわらずスキーマとポリシーゲートが必要です。

## 実装ウォークスルー

`Sampling.distribution` は、温度、softmax、top-k、再正規化、top-p、最後の
再正規化という固定の順序でポリシーを適用します。順序を変えると分布が変わる
ため、この順序は公開された挙動の一部です。

ロジット `[3,2,1,0]` に対して、top-k `2` はトークン ID `0` と `1` を残し
ます。それ以外の ID の確率はゼロになり、生き残ったものが再正規化されます。
一方 top-p は、確率とインデックスの組を降順にソートし、閾値に達するまで
累積し、閾値を越える最小の接頭辞を保持します。閾値が非常に小さくても、
少なくとも 1 つのトークンは常に保持されます。

温度は softmax の前にロジットを割ります。1 未満の値は差を拡大して
エントロピーを下げ、1 より大きい値は差を平坦にします。温度は有限かつ正で
なければなりません — ゼロは貪欲デコーディングの省略記法としてはサポート
されていません。決定的な貪欲の挙動には top-k を 1 にしてください。

同順位の順序付けには小さいトークン ID を使います。この第 2 キーがないと、
等しい確率がコレクション実装の詳細によって順序付けられ、固定シードの
サンプリングが揺らいでしまいます。`normalize` は正の残存質量を要求し、その
合計でスケールします。

`Sampling.sample` は乱数の状態を一切持ちません。`RandomGenerator` を
受け取り、フィルタリング済みの `Categorical` を構築し、そこに 1 つの ID を
求めます。したがって再現性は、ポリシーと、それ以前の RNG 呼び出しの両方に
依存します。

## テストの読み方

top-k が 1 の場合には、決定的な argmax のオラクルがあります。語彙より
大きい top-k はクランプを検査します。作為的に構築した確率ベクトルは、
手計算で分かる nucleus を与えます。エントロピーの比較は温度の方向を検証
します。等しいロジットの同順位は ID の順序を検証します。等しいシードは、
フィルタリングを含むサンプリング経路全体を検証します。無効なポリシー値は
`SamplingConfig` の構築時に失敗します。

## デバッグチェックリスト

1. 温度適用後、softmax 前のロジットを出力してください。
2. 各フィルタの後に、保持された ID と確率質量を出力してください。
3. トークンを取り除いたら再正規化してください。
4. 決定的な同順位の順序を指定してください。
5. シード付きサンプルが揺らぐ場合は、ポリシーの順序と RNG の呼び出し回数を
   比較してください。

## 演習

1. ロジット `[0,1,2]` について、いくつかの温度で確率とエントロピーを計算
   してください。
2. top-k と top-p を入れ替えると候補が変わるケースを見つけてください。
3. 10,000 サンプルの頻度を理論的な確率と比較してください。
4. 繰り返しペナルティを設計し、その操作の順序を明記してください。
5. 固定のプロンプトとシードで、貪欲、top-k、top-p を比較してください。

## 完了基準

- モデルのロジットとデコーディングポリシーを分離できる。
- 温度がエントロピーに与える効果を説明できる。
- top-k と top-p の選択を比較できる。
- フィルタリングに再正規化が必要な理由を説明できる。
- 再現のために操作の順序とシードを記録できる。
- `SamplingSuite` が通る。
