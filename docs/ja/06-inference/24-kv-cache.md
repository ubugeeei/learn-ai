# 24 — KV キャッシュ

## この章で作るもの

訓練側の実装は、すべてのトークンが 1 つの並列な損失に寄与するため、
シーケンス全体を評価します。自己回帰的な推論は形が違います。トークン `t`
をサンプリングした後、モデルに必要なのはトークン `t + 1` のロジットだけ
です。それまでのすべてのキーと値を再計算するのは正しいものの、無駄が
あります。

この章では次のことを行います。

- 注意機構 (attention) のどの値が再利用できるかを導出します。
- 各 Transformer 層に固定容量の KV キャッシュ (KV cache) を実装します。
- リクエストが所有する `MiniGptInferenceSession` を構築します。
- 独立した参照経路で、キャッシュありとフル接頭辞の等価性を証明します。
- コンテキストウィンドウがずれるときに、学習された絶対位置の挙動を保ち
  ます。
- 決定的な作業量の勘定と、実測のウォールクロック結果を分離します。

## 前提知識

第 18〜23 章、特に因果的自己注意機構における形状と、ロジットと
サンプリングされたトークン ID の違いを理解している必要があります。

## 1. 参照デコーダが繰り返すもの

時刻 `t` のある 1 つのヘッドについて、因果的注意機構は次のとおりです。

$$
q_t = x_t W_Q,\qquad
k_i = x_i W_K,\qquad
v_i = x_i W_V
$$

$$
a_{t,i} = \operatorname{softmax}_i
\left(\frac{q_t k_i^\top}{\sqrt{d_h}}\right),
\qquad 0 \le i \le t
$$

$$
o_t = \sum_{i=0}^{t} a_{t,i}v_i
$$

一度 `k_i` と `v_i` が計算されると、その入力とパラメータはリクエストの間
変化しません。次のデコーディングステップには新しい `q_t`、`k_t`、`v_t` が
必要ですが、それまでのキーと値はすべて再利用できます。

キャッシュはクエリを保持**しません**。過去のクエリが再び使われることは
ないからです。

## 2. 形状と所有権

層数 `L`、コンテキスト容量 `T`、モデル幅 `C` に対して、この実装は次を
確保します。

```text
per layer keys:    [T, C]
per layer values:  [T, C]
all cache payload: L * 2 * T * C * sizeof(Double)
```

ヘッド次元はチャネル範囲へのビューなので、キャッシュはヘッドごとの
オブジェクトではなく、結合された 1 つの `[T, C]` 行列を保持します。

`AttentionKeyValueCache` はミュータブルで、1 つのリクエストに属します。
モデルの重みは共有されたまま変化しません。この区別は運用上重要です。

```text
shared across requests: MiniGPT parameters
owned by one request:   token history, RNG, KV caches, work counters
```

ミュータブルなキャッシュを並行するリクエスト間で共有すると、それらの
コンテキストが混ざってしまいます。そのため API はリクエストごとに明示的な
`MiniGptInferenceSession` を作成し、スレッドセーフではないことを
ドキュメント化しています。

## 3. キャッシュを使う 1 トークンの注意機構

`CausalSelfAttention.forwardCached` は、ちょうど `[1, C]` の入力を受け取り
ます。

1. 現在の行をクエリ、キー、値へ射影します。
2. キー/値の行を固定配列に追記します。
3. 現在のクエリとキャッシュ位置 `[0, length)` の間のスコアを計算します。
4. 最大スコアを引くことで、数値的に安定な softmax を適用します。
5. 重み付けした値の行を 1 つ作り、出力射影を適用します。

この経路には三角マスクがありません。キャッシュには過去と現在の位置しか
含まれておらず、未来の位置はまだ存在しないからです。

## 4. 層ごとの状態

各 Transformer ブロックには別々のキャッシュが必要です。第 1 層のキーは
第 1 層の入力の射影であり、第 2 層のキーは第 2 層の入力の射影です。1 つの
キャッシュを層をまたいで使い回すのは、形状としては整合するものの数学的
には誤りであるバグです。

位置 `t` のトークンに対して、セッションは次を評価します。

```text
token[t] + position[t]
  -> block[0](cache[0])
  -> block[1](cache[1])
  -> ...
  -> final RMSNorm
  -> tied token-embedding transpose
  -> logits[t]
```

## 5. 等価性オラクル

もっともらしいテキストを生成するというだけでは、最適化が正しいことには
なりません。テストは、キャッシュを使ったすべての結果を、既存のフル接頭辞
経路と比較します。

```text
cached(token[t], cache(prefix))
    approximately equals
fullForward(prefix :+ token[t]).lastRow
```

この比較は、まず注意機構単体に対して、次に MiniGPT 全体に対して行われ
ます。これにより、誤ったヘッドのオフセット、注意機構の後に保存された
キー/値、層をまたぐキャッシュ共有、不正な位置 ID、不安定な softmax コード
を捕まえられます。

許容誤差が考慮するのは浮動小数点演算の順序だけです。系統誤差を隠すために
緩めることはありません。

## 6. コンテキスト容量と絶対位置

MiniGPT は現時点で、学習された絶対位置埋め込みを使っています。参照
デコーダは、長すぎるコンテキストを左から切り詰め、保持した位置 ID をゼロ
から振り直します。

```text
old context positions: 0 1 2 3
append at capacity:      1 2 3 4
reference window IDs:    0 1 2 3
```

単に最も古いキャッシュ行を追い出すだけでは、保持された隠れ状態が位置
`1, 2, 3` でエンコードされたままになります。それはモデルの挙動を静かに
変えてしまいます。

そのため `generateCached` は、容量が一杯になったときに保持ウィンドウを
再構築します。これで参照実装のセマンティクスが正確に保たれますが、
再構築の間はキャッシュの作業量の優位が一時的に失われます。後の章では、
RoPE と明示的な長コンテキストポリシーによって、このトレードオフを偶発的な
ものではなく設定可能なものにします。

## 7. 決定的な作業量の勘定

プロンプトが `P` トークンで、容量に達することなくさらに `N` トークン生成
するとします。参照経路は

$$
P + (P+1) + \cdots + (P+N-1)
= NP + \frac{N(N-1)}{2}
$$

個のトークン行を評価します。キャッシュ経路が評価するのは

$$
P + (N-1)
$$

行です。プロンプトを一度プリフィル (prefill) し、その後サンプリングした
各トークンを評価しますが、最後のトークンはそのロジットが不要なため評価
しません。

これらは決定的な、アルゴリズム上の作業量の勘定です。アロケーション、JVM
のコンパイル、小さい行列のオーバーヘッド、メモリアクセスも影響するため、
実行時間の高速化を証明するものではありません。

## 8. ラボの実行

```console
$ nix develop -c sbt 'runMain learnai.transformer.runKvCacheLab'
```

このラボは次を行います。

- 計測の前に、サンプリングされたトークン ID が同一であることを検証します。
- JIT のウォームアップを行います。
- 参照経路とキャッシュ経路の計測を繰り返します。
- 最小、中央値、最大の時間を報告します。
- JVM、OS、CPU、設定、シード、ワークロードを記録します。
- 出力が観測可能であり続けるように、チェックサムを出力します。
- トークンの作業量とキャッシュのペイロードを分けて報告します。

これは JMH の代替ではなく、教育用の透明なマイクロベンチマークです。その
計測値を一般的な性能主張に流用しないでください。対象のハードウェアで
再実行し、環境の完全な出力を保存してください。

## 9. 実装の解説

`AttentionKeyValueCache` は `capacity * channels` の固定配列を 2 つ確保
します。`currentLength` が唯一の論理サイズの変更点です。追記時には
`[1,C]` のキー/値テンソルと残り容量を検証し、オフセット
`currentLength * channels` に 1 行をコピーしてから長さを増やします。
クリアは再確保もバイトの消去も行わず、長さをゼロにします。アクセス不能に
なった古い行は、後の追記で上書きされます。

`MiniGptInferenceSession` はブロックごとにちょうど 1 つのキャッシュを作り
ます。`append` は `currentLength` の位置でトークンを埋め込み、1 行の
テンソルを、各ブロックとそのキャッシュの組で順に畳み込み、最終ノルムと
共有 (tied) 射影を適用してから、セッション長と作業量カウンタを増やします。
長さはすべてのブロックが成功した後にのみ増えるため、各層のキャッシュと
セッションは常に揃っています。

`generateCached` には微妙なループ境界があります。プリフィルは最後の
プロンプトトークンの後のロジットを計算します。各イテレーションはその
ロジットからサンプリングします。サンプリングした最後のトークンは、それ
以上トークンをサンプリングしないなら、もう一度の順伝播を必要としません。
これが、キャッシュの作業量が `P + N` ではなく `P + (N-1)` である理由です。

容量に余裕があれば、サンプリングしたトークンを一度だけ追記します。容量に
達したら、コードは保持ウィンドウを構成して `rebuild` を呼びます。これは
各層のキャッシュをクリアしますが、累積の作業量カウンタは意図的に保持
します。参照側の作業量は、実際のセッション評価とは独立に数えられます。

ベンチマークは、1 つの固定モデルと同一シードのワークロードを構築し、計測
の前にトークンの一致を確認し、両経路をウォームアップし、計測を繰り返し、
チェックサムを出力します。環境検出は JVM、OS、CPU を記録します。それでも
これはマイクロベンチマークであり、JMH ほど厳密には JVM のコンパイルを制御
しません。

## 10. テストの読み方

注意機構レベルの等価性は、キャッシュのインデックス計算をブロック/モデルの
挙動から切り離します。モデルレベルの接頭辞比較は、位置と層の所有権の誤り
を捕まえます。容量をまたぐ生成は、再構築のセマンティクスを検証します。
決定的な作業量の勘定のテストは、小さいケースで参照 `9` 行対キャッシュ
`4` 行を主張します。これはタイミングの主張ではありません。オーバーフロー/
リセットのテストは、変更の境界を定義します。

## 11. デバッグチェックリスト

1. 各トークンの後に、すべての層キャッシュの長さがセッション長と等しいこと
   を確認してください。
2. 注意機構の前に、1 つの層の射影済みキー/値の行を比較してください。
3. キャッシュのスコアを、フル接頭辞のスコアの最終行と比較してください。
4. 注意を計算する前に、現在の K/V が追記されていることを検証してください。
5. ウィンドウがずれたら、絶対位置 ID を比較し、ゼロから再構築してください。
6. トークンの作業量の勘定、キャッシュのペイロード、ウォールクロック計測を
   分離してください。

## 12. テストすべき失敗モード

- ゼロまたは負のキャッシュ次元。
- 固定容量を超えた追記。
- 不正なキー/値の形状。
- キャッシュとモデルのチャネル数の不一致。
- 空のプリフィル。
- コンテキスト容量より長いプリフィル。
- 1 つのキャッシュが誤ってリクエスト間や層間で共有される。
- キャッシュの出力が、独立したフル接頭辞経路と異なる。
- ウィンドウの追い出しが絶対位置のセマンティクスを変えてしまう。
- 比較の中でサンプリングが異なるシードやフィルタリングポリシーを使って
  いる。

## 演習

1. 注意機構の結果を変えずに、ページ化キャッシュの抽象を追加してください。
2. キャッシュのペイロードを `Float` で保存し、ロジットの誤差とメモリを
   測定してください。
3. バッチ次元を追加し、バッチ内のリクエストが異なる論理長を持ち得る理由を
   説明してください。
4. リクエストのキャンセル後のキャッシュのコンパクションを実装してください。
5. RoPE を実装した後、再構築、追い出し、位置オフセットの各ポリシーを
   明示的に比較してください。

## 完了基準

次のことができれば完了です。

- キーと値は再利用できるのにクエリは再利用できない理由を導出できる。
- 層数、コンテキスト、チャネル数からキャッシュのペイロードサイズを計算
  できる。
- リクエストの所有権と並行性の境界を説明できる。
- 注意機構レベルとモデルレベルで、キャッシュありとフル接頭辞の等価性を
  証明できる。
- 学習された絶対位置がウィンドウのずれで再構築を必要とする理由を説明
  できる。
- 作業量の勘定を、条件を明記したタイミング実験とは分けて報告できる。
