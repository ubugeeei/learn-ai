# 25 — バージョン付きで安全なチェックポイント形式

## この章で作るもの

MiniGPT の設定、安定したパラメータラベル、形状、行優先 (row-major) の
`Double` 値をシリアライズします。ロードの前に SHA-256 を検証し、保存済み
ファイルをアトミックに置き換えます。ソース:
`src/main/scala/learnai/io/MiniGptCheckpoint.scala`。

## チェックポイントは配列の集まり以上のもの

生の値だけでは、その意味を特定できません。有用な形式は次を含みます。

- マジックバイトと形式バージョン。
- アーキテクチャ設定。
- 安定したパラメータ名。
- ランク、形状、要素数、dtype、バイトオーダー。
- 値。
- 整合性チェックサム。

デプロイ可能なバンドルには、トークナイザの語彙とマージも必要です。この
教育用形式は意図的に推論用モデルだけを保存し、その制限をドキュメント化
しています。

## ファイルレイアウト

```text
magic "LAIGPT01"                 8 bytes
format version                   Int32
MiniGptConfig                    fixed fields
parameter tensor count          Int32
repeat:
  UTF-8 label length + bytes
  rank + dimensions
  scalar count
  Float64 values, big-endian
SHA-256(payload)                 32 bytes
```

JDK のデータストリームは、ビッグエンディアンの数値エンコーディングを使い
ます。この形式は、JVM のオブジェクトシリアライゼーションにもメモリ内の
オブジェクトレイアウトにも依存しません。

## ラベルと順序

`blocks.0.attention.query.weight` のような名前は、意味上の所属を識別
します。ローダは期待するモデルを構築し、代入の前に個数、順序、ラベル、
形状、要素数を確認します。

形状だけでは、同じ形状の 2 つの重みの入れ替わりを検出できません。保存は
重複したラベルを拒否します。

## 整合性と真正性

SHA-256 は偶発的な破損と不完全な転送を検出します。誰でも再計算できるため、
発行者を認証するものではありません。

- 整合性 (integrity): チェックサム。
- 真正性 (authenticity): 信頼されたデジタル署名、発行者の同一性、来歴。

本番の配布には両方が必要です。

## パースの前に検証する

ローダはフィールドを解釈する前に、ファイルサイズの上限とチェックサムを
確認します。さらに次にも上限を設けます。

- ファイルの総バイト数。
- 設定から導かれるスカラー数。
- ラベルのバイト数とランク。
- 期待されるテンソルの形状と個数。

検証前に信頼できない長さでアロケーションを行うと、メモリ枯渇攻撃が可能に
なります。

## 全か無かの代入

`Tensor.assignParameterValues` は、要素を 1 つでも変更する前に、個数と
すべての値の有限性を検証します。ローダが変更するのは新しく構築したモデル
なので、ロードの失敗が現在サービング中のモデルを壊すことはありません。

本番のロールアウトでは、新しいモデルを検証し、スモーク推論を実行してから、
参照をアトミックに切り替えるべきです。

## アトミックな保存

正常なチェックポイントの上に直接書き込むと、クラッシュやディスクフルの後
にそれを破壊しかねません。代わりに次のようにします。

1. 同じディレクトリに、完全な一時ファイルを書き込みます。
2. そのチェックサムを含めます。
3. 宛先の上にアトミックに移動します。
4. アトミックな移動が未サポートの場合にのみ、置き換えの移動にフォール
   バックします。

より強い永続性には、ファイルとディレクトリの sync も必要です。

## 推論と再開可能な訓練

推論に必要なのはモデルの設定と重みです。訓練を正確に再開するには、さらに
次が必要です。

- AdamW のモーメントとステップ。
- スケジューラと損失スケーラの状態。
- RNG の状態。
- データサンプラ/シャッフルの位置。
- トークナイザ、データ、コード、ビルドのリビジョン。

重みだけをロードすると、オプティマイザのダイナミクスは最初からやり直しに
なり、元の軌跡を再現できません。

## なぜオブジェクトシリアライゼーションを使わないのか?

汎用のオブジェクトデシリアライゼーションはクラスのフックを実行し得るうえ、
ファイルを実装の詳細に縛り付けます。信頼できないモデルファイルがコード
実行のリスクになり得るのです。この形式は明示的なプリミティブフィールド
だけを読み取り、許可されたモデル構造を構築します。Safetensors も同じ
「コードではなくデータ」の原則に従っています。

## テスト

- ラウンドトリップ後の設定、ラベル、値、ロジットのビット単位の一致。
- 1 バイトの破損の拒否。
- 切り詰められたファイルの拒否。
- 既存ファイルのアトミックな置き換え。
- 無効な代入は古い値を変更しない。

## 実装の解説

`MiniGptCheckpoint.save` は、まず設定とすべてのパラメータを決定的な
ペイロードに変換します。パラメータの順序は `model.parameters` に由来
します。ラベル、形状、値が書き込まれるため、ロード側は偶然の一致で代入
するのではなく、構造的に非互換なデータを拒否できます。

外側のファイルは、マジック識別子、形式バージョン、ペイロード長、
ペイロード、チェックサムを持ちます。整数は明示的なバイトオーダーを使い
ます。文字列とベクトルは長さの前置き付きなので、リーダーがバイナリデータ
の中でデリミタのバイトを探すことはありません。

保存は一時的な同ディレクトリのファイルに書き込み、フラッシュして
クローズしてから、宛先をアトミックに置き換えます。置き換え前のクラッシュ
では、以前のチェックポイントが無傷で残ります。直接の上書きでは、最新に
見える切り詰められたファイルが残りかねません。

ロードは信頼境界を逆向きに、防御的な順序でたどります。

```text
read bounded bytes
  -> validate magic/version/declared length
  -> verify checksum
  -> parse config and metadata
  -> construct a model with required shapes
  -> validate all parameter records
  -> assign only after complete validation
```

`assignParameterValues` は、要素を 1 つでも変更する前に、入力全体の個数と
有限性を検証します。この全か無かの性質が、後方の不正な値によって
パラメータが半分だけロードされた状態で残ることを防ぎます。

## テストの読み方

保存/ロードは、ファイルのバイト列だけでなく、設定、ラベル、パラメータ、
ロジットを比較します。1 バイトの破損は、パースの前にチェックサムで失敗
しなければなりません。切り詰めは、部分的なモデルではなくエラーを返さ
なければなりません。既存パスの置き換えは、アトミックなワークフローを検証
します。テンソルの代入は、拒否された入力が以前の値を変更しないことを明示
的にテストします。

## デバッグチェックリスト

1. モデルのフィールドより先に、マジック、バージョン、宣言されたペイロード
   長を検査してください。
2. 値をデコードする前に、生のペイロードのバイト列に対してチェックサムを
   検証してください。
3. 最初の不一致の箇所で、パラメータのラベルと形状を比較してください。
4. 後方のレコードをまだ検証している間に、部分的な代入をしないでください。
5. 一時パスへの書き込みを再現し、アトミックな移動のサポートを調べて
   ください。
6. ファイル由来のすべてのサイズを信頼できないものとして扱い、上限を設けて
   ください。

## 演習

1. 16 進ビューアでマジックとバージョンを調べてください。
2. バージョン 2 への移行を設計してください。
3. トークナイザのメタデータとチェックサムをバンドルのマニフェストに追加
   してください。
4. 推論のロードでスキップできる、オプションのオプティマイザ状態を追加して
   ください。
5. Ed25519 署名と検証ステップを配置してください。
6. 公開時のロードテストと、固定ロジットのスモークテストを設計してください。

## 完了基準

- 設定、ラベル、形状が必要な理由を説明できる。
- チェックサムと署名を区別できる。
- 信頼できない長さの検証を説明できる。
- アトミックな置き換えが以前のファイルを守る仕組みを説明できる。
- 再開可能な訓練に必要な追加の状態を列挙できる。
- `MiniGptCheckpointSuite` が通る。
