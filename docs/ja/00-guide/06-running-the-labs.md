# 迷わずラボを実行する

## 最短経路

リポジトリのディレクトリから、次を実行します。

```console
$ ./learn-ai model
```

このラッパーは、何が実行されるのか、どの出力を確認すべきか、その実験が
何を証明しないのか、そして内部で呼び出す低レベルのコマンドを説明します。

続いて、次を実行します。

```console
$ ./learn-ai training
$ ./learn-ai diagnostics
$ ./learn-ai agent
```

`./learn-ai help` を使うと、ガイド付きコマンドの一覧を表示できます。

## コマンドの各層の意味

次の低レベルコマンドを考えます。

```console
$ nix develop -c sbt 'runMain learnai.Main model'
```

これは 4 つの層からなります。

### `nix develop`

Nix は `flake.nix` と `flake.lock` を読み、ピン留めされた JDK と sbt の
ツールを取得し、一時的な開発環境を作成します。モデルの訓練は行いません。

目的は再現性です。別のマシンに異なるグローバルパッケージがあるからと
いって、このコマンドが黙って別の Java や Scala のツールチェーンを選択する
ことがあってはなりません。

### `-c`

これは Nix に、その環境の中で残りのコマンドを実行してから終了するよう
指示します。`-c` がない場合、`nix develop` は対話シェルを開き、あなたが
別のコマンドを入力するのを待ちます。

### `sbt`

sbt は Scala のビルドツールです。`build.sbt` を読み、Scala のソースファイル
をコンパイルし、クラスパスを構築し、選択されたプログラムを起動します。

sbt はビルド基盤であり、AI モデルではありません。モデルは
`src/main/scala/learnai` 以下の Scala コードです。

### `runMain ...`

`runMain` は、唯一の本番エントリポイント `learnai.Main` の実行を sbt に
依頼します。その後ろの `model` はラボを選ぶ引数です。

```text
learnai.Main model
| entrypoint | lesson |
```

`Main.scala` の小さな一覧が、`model` や `agent` を各パッケージの普通の実行関数へ
対応付けます。入口から実装までの対応を一か所で読めます。

## どのラボを実行すべきか

| コマンド | モデルの重みを変更するか | リモート API を使うか | 主な問い |
| --- | --- | --- | --- |
| `./learn-ai foundations` | いいえ | いいえ | Scala の制御フローを読めるか? |
| `./learn-ai xor` | はい | いいえ | スカラー自動微分は非線形関数を学習できるか? |
| `./learn-ai bigram` | はい | いいえ | Transformer 以前の次トークン訓練とは何か? |
| `./learn-ai model` | はい | いいえ | 完全な Transformer グラフは学習と生成ができるか? |
| `./learn-ai training` | はい | いいえ | バッチ、検証、スケジューリング、指標はどう相互作用するか? |
| `./learn-ai diagnostics` | いいえ | いいえ | 勾配 (gradient) は正しいか、ローカルの計時はどう報告されるか? |
| `./learn-ai cache` | いいえ | いいえ | キャッシュ付きデコーディングは、計算を減らしつつ参照結果を保存するか? |
| `./learn-ai quantization` | いいえ | いいえ | int8 のバイト数と数値誤差はどうトレードオフされるか? |
| `./learn-ai agent` | いいえ | いいえ | ランタイムはツールと権限の契約を強制するか? |
| `./learn-ai planning` | いいえ | いいえ | 依存関係のあるエージェントタスクは、どうリトライされ復旧されるか? |
| `./learn-ai test` | 多くのフィクスチャをテストする | いいえ | 実装済みのすべての契約はまだ成立しているか? |

現在のガイド付きコマンドは、すべてローカルの CPU 上で実行されます。
エージェントのラボは、状態遷移が再現可能になるよう、スクリプト化された
フェイクモデルを使います。OpenAI などのプロバイダにプロンプトを送信する
ことはありません。

## `./learn-ai model` の読み方

このラボは次を行います。

1. `to be or not to be` を繰り返したコーパスを構築する
2. 小さな BPE トークナイザ (tokenizer) を訓練する
3. 入力/次トークンターゲットのウィンドウを 1 つ作成する
4. 固定シードから 1 層の MiniGPT を初期化する
5. 120 回の AdamW 更新を実行する
6. 固定のプロンプトとシードから、30 個の新しいトークンをサンプリングする

重要な出力:

```text
parameters:   4584
initial loss: 5.514108
final loss:   0.000220
generated:    ...
```

- `parameters` は、訓練可能なスカラー値の個数です。
- `initial loss` は、更新前の次トークン誤差を測ったものです。
- `final loss` は、更新後に同じ固定の訓練ウィンドウで測ったものです。
- `generated` は、訓練済みのトイモデルからサンプリングされたものです。

コーパスとウィンドウが極小で反復的なため、非常に低い損失 (loss) になるのは
想定どおりです。モデルはパターンを記憶しています。これは、埋め込み
(embedding)、attention、フィードフォワード、損失、逆伝播、
オプティマイザの経路が正しくつながっていることを検証するのに有用です。
一般的な言語能力の証拠ではありません。

## `./learn-ai training` の読み方

こちらは、より現実的な学習実験です。次を行います。

1. トークナイザの訓練が検証テキストをリークできないよう、バイトトークンを
   使う
2. ウィンドウを構築する前に、生のトークンを分割する
3. 訓練領域のみから訓練バッチをサンプリングする
4. 各バッチをマイクロバッチに分割し、勾配を累積する
5. クリッピングとウォームアップ/コサイン学習率を伴う AdamW を適用する
6. 5 回の更新ごとに、固定の検証プレフィックスを評価する
7. 正準な実験 ID とトークン数を記録する

出力の 1 行は次のようになります。

```text
update= 10 tokens= 320 lr=0.019392 train=3.667246 \
validation=3.377318 gradient=1.3141 clip=0.7610
```

- `update`: 完了したオプティマイザ更新の回数であり、個々の事例数では
  ありません。
- `tokens`: 累積の訓練ターゲットトークン数。検証は含まれません。
- `lr`: その更新に使われた学習率。
- `train`: 更新前の、サンプリングされたバッチでの損失。
- `validation`: 更新後のホールドアウト損失。
- `gradient`: クリッピング前のグローバル勾配ノルム。
- `clip`: 勾配に適用された係数。`1.0` はクリッピングなしを意味します。

バッチはサンプリングされるため、訓練損失は行の間で上下することがあります。
検証は同じホールドアウト事例を再利用するため、より安定した比較になります。

## `./learn-ai diagnostics` の読み方

このコマンドはモデルを最適化しません。実装を検査します。

### パラメータ一覧

ラベルと形状は、どのコンポーネントが各 Tensor を所有しているかを示します。
密なペイロードはパラメータ値のみを数え、勾配、オプティマイザのモーメント、
活性化、オブジェクト、JVM のオーバーヘッドは含みません。

### 勾配プローブ

逆方向モードの勾配を、サンプリングされたパラメータ座標において、中心有限
差分と比較します。`gradient passed: true` は、指定された絶対/相対の許容誤差
の下で両者が一致したことを意味します。すべての座標やあらゆる入力について
証明するものではありません。

### ベンチマーク

`median`、`p95`、`variation` は、ウォームアップ後にローカル CPU で順伝播を
繰り返した結果を表します。ランタイム行には、JVM、OS、アーキテクチャ、
プロセッサ数が記録されます。この数値を GPU サービスと比較したり、統制
されたベンチマーク環境なしに公表したりしないでください。

## `./learn-ai agent` の読み方

エージェントのラボには、3 つの固定シナリオがあります。

1. ツールを使わずに、正確な答えを返す
2. 許可された読み取り専用の検索を 1 つ要求し、実行する
3. ホストが実行前に拒否しなければならない書き込みを要求する

言語モデルは、あらかじめ決められた判断を返すスクリプト化されたフェイク
です。これによって、ランタイムの正しさをモデルの品質から分離できます。

最も重要な区別は次のとおりです。

```text
requested tool != authorized tool != physically attempted tool
```

拒否された書き込みは、要求されたツールには現れるかもしれませんが、物理的な
試行イベントに現れてはなりません。

## 出力が異なるとき

ラボのシードとデータは固定されているため、損失と生成テキストは決定的で
あるはずです。JVM のウォームアップ、CPU、OS の負荷、ガベージコレクション
が異なるため、ローカルの計時は異なることがあります。

コマンドが失敗した場合:

1. 最後の sbt のサマリだけでなく、最初のエラーを読みます。
2. `./learn-ai test` を実行します。
3. `git status --short` で、作業中のソース変更がないか確認します。
4. `nix --version` が成功することを確認します。
5. 許容誤差やハイパーパラメータを変更する前に、その章のデバッグ
   チェックリストを使います。

コミットされていない Git ツリーに関する Nix の警告は、情報提供にすぎません。
現在のソースが最後のコミットと異なることを示しており、それは再現性にとって
重要ですが、それだけでプログラムが失敗したことにはなりません。

## 実行後にソースを読む

エントリポイントから実装へと、プログラムをたどります。

```text
./learn-ai command
  -> learnai.Main and lesson name
  -> ordinary lesson function in its package
  -> model/runtime public methods
  -> Tensor/tool primitives
  -> colocated *Suite.scala
  -> corresponding chapter in docs
```

まず実行し、出力フィールドを 1 つ予測してから、それを担当するメソッドに
print やブレークポイントを置きます。ラッパーの目的はコマンドラインの
当て推量をなくすことであり、実装への道筋を隠すことではありません。
