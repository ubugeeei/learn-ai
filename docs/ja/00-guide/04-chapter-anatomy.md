# 完全な実装章の構造

## この基準が存在する理由

短い概念的な要約は、すでにその主題を知っている人の役には立ちますが、
前提知識ゼロのハンズオンにはなりません。このリポジトリでは、完成した章は、
学習者がその名前を認識できるだけでなく、実装を再構築できるように助ける
ものでなければなりません。

本文、数式、コード、テスト、実行可能な実験は、1つの仕組みに対する異なる
視点です。

```text
intuition
  -> concrete numbers
  -> symbols and shapes
  -> API contract
  -> loop/index implementation
  -> tests and independent oracle
  -> observed behavior
  -> limitations and next design
```

矢印が1つでも欠けていると、学習者はコードがなぜ動くのかを推測しなければ
なりません。

## 必須の学習レイヤー

### 1. 平易な言葉による動機

技法の名前を出す前に、問題を述べます。新しいコンポーネントなしでは何が
失敗するのか、そのコンポーネントがどの責務を担うのかを説明します。

悪い例:

> softmax を実装せよ。

役に立つ例:

> ロジットは制約のないスコアですが、カテゴリカルサンプリングには合計が
> 1 になる非負の値が必要です。softmax はスコアの順序を保ったままその変換を
> 行います。最大値の減算は、結果を変えずにオーバーフローを防ぎます。

### 2. 手計算できる例

重要な挙動を確かめられる最小の数値を使います。可能な範囲で、すべての
中間値を示します。例は、紙と電卓で検算できるほど小さくあるべきです。

行列演算であれば、実際の行、列、積、和を書き出します。エージェントの
状態機械であれば、正確な状態とイベントを列挙します。直列化であれば、
正確なバイトレイアウトを描きます。

### 3. 記号、型、形状

すべての数式は、実行可能な名前へ結び付けなければなりません。

| 数学 | Scala | 意味 |
| --- | --- | --- |
| $X\in\mathbb{R}^{T\times C}$ | `input: Tensor` | トークンごとに 1 行の隠れ状態 |
| $W_Q\in\mathbb{R}^{C\times C}$ | `queryProjection.weight` | クエリのパラメータ |
| $Q=XW_Q$ | `queryProjection(input)` | 射影されたクエリ |

アルゴリズムを説明する前に、事前条件と出力の形状を述べます。形状は
正しさの一部であり、任意の補足説明ではありません。

### GitHub が描画する数式デリミタ

このリポジトリは GitHub Markdown を対象とします。インライン数式は
`$x_i$` のように、両側を 1 つのドル記号で区切ります。ディスプレイ数式は、
ダブルドルの行の間に置きます。

```markdown
$$
y = Wx + b
$$
```

バックスラッシュと丸括弧、またはバックスラッシュと角括弧によるデリミタは
使わないでください。LaTeX エンジンはそれらを認識しますが、GitHub は式を
数式レンダラーに渡さず、文字どおりのテキストとして表示することがあります。

### 4. ソースマップ

プロダクションコードとテストのファイルを名指しします。公開エントリポイント
と、それらが委譲するプライベートなヘルパーを特定します。学習者は、
ウォークスルーを読む前に、どこにブレークポイントを置けばよいか分かるべき
です。

### 5. 実装ウォークスルー

実行順序に従います。自明でないブロックごとに、次を説明します。

- どの不変条件がすでに成り立っているか
- どのデータが読み取られるか
- どのデータが割り当てられ、または変更されるか
- ループ/インデックスの順序がなぜ数式と一致するのか
- その後にどの不変条件が新たに成り立つか
- 失敗がどこで報告されるか

ソースファイル全体を貼り付けてはいけません。必要最小限のコード片を引用し、
各行の責務を説明して、記号へと結び付け直します。

### 6. 設計の代替案

もっともらしい代替案を少なくとも 1 つ説明し、その章でまだそれを使わない
理由を述べます。例としては、イミュータブルなバッファと可変バッファ、例外と
`Either`、スカラーグラフとテンソルカーネル、完全な再計算とキャッシュなどが
あります。

教育上の簡略化にはラベルを付けなければなりません。さもないと、学習者が
それをプロダクションのベストプラクティスだと誤解するかもしれません。

### 7. テストの読み方

テストは実行可能な説明です。重要なテストカテゴリごとに、次を述べます。

- そのテストが捕まえることを意図しているバグ
- 期待値が実装から独立している理由
- 失敗が通常は何を意味するか
- 許容誤差やシードが適切である理由

正常系、境界、不正入力、そして関連するプロパティ/数値のケースを含めます。
エンドツーエンドの成功テストは、焦点を絞った単体テストの代わりには
なりません。

### 8. デバッグチェックリスト

症状から始まる判断の道筋を示します。

```text
NaN loss
  -> inspect logits before softmax
  -> verify maximum subtraction
  -> inspect target range
  -> inspect gradient norm
  -> reduce learning rate only after locating the first invalid value
```

チェックリストは調査のしかたを教えるべきであり、パラメータの手当たり
次第の変更を促すものであってはなりません。

### 9. 実行可能な観察

正確な Nix コマンドを提供し、その出力が何を証明するのか、そして何を証明
しないのかを説明します。訓練の損失 (loss) が下がることは、そのデータ上での
グラフの結合と最適化を証明しますが、汎化を証明するものではありません。

### 10. 限界と次への接続

計算量、メモリ、数値、並行性、安全性、スケールの限界を、関連する範囲で
述べます。最後に、後のどの章がその新しいコンポーネントを利用するのかを
示します。

## ドキュメントの完了の定義

実装済みの技術章については、レビュアーが次の質問すべてに「はい」と答え
られるべきです。

- 初学者が、技法の名前を見る前に問題を述べられるか?
- 手計算できる例があるか?
- 数式が Scala の名前と形状に対応付けられているか?
- ウォークスルーは実際の実行パスに従っているか?
- ミューテーション、所有権、エラーチャネルが明示されているか?
- テストには、説明された独立のオラクルがあるか?
- 症状駆動のデバッグセクションがあるか?
- 実行コマンドは、その出力が何を実証するのかを述べているか?
- 教育上の簡略化がプロダクションシステムと区別されているか?
- 研究由来の仕組みが関わる箇所で、一次情報源がリンクされているか?

## 章の学び方

一度だけ読むのではなく、4 回のパスを使います。

1. **予測:** 問題と手計算の例を読み、API を予測します。
2. **トレース:** ウォークスルーに沿って、1 つの入力をソースの中で追跡します。
3. **反証:** 各テストを読み、それが捕まえるバグを名指しします。
4. **再構築:** ソースを閉じ、契約、数式、例からコンポーネントを実装します。

章を理解したと言えるのは、それを再構築してデバッグできるときであり、
完成したコードが単に見覚えのあるものになったときではありません。

テストスイートは、実装済みの章に対して意図的に控えめな最低ラインを強制
します。すなわち、実装ウォークスルー、テストの読解/オラクルのセクション、
デバッグチェックリスト、そして説明を展開するのに十分な分量です。この
構造チェックに合格しても教育品質が確立されるわけではありません。詳細な
章がいつの間にか短い概要へと縮んでしまうのを防ぐだけです。
