# このリポジトリでの学び方

## 前提知識

プログラミング、ターミナル、Git、Scala、高度な数学の知識は前提としません。
読むだけでは十分ではないため、すべての章は同じ実験ループに従います。

1. コードを実行する前に、結果や学習曲線を**予測**します。
2. すべての記号を型付きの値に対応付けて、数式を**実装**します。
3. テスト、誤差、時間、メモリを**測定**します。
4. なぜその結果になったのかを、自分の言葉で**説明**します。
5. **変数を1つだけ変え**、観察結果を自分の予測と比較します。

## 章の構成

各章は一貫したフォーマットを使います。

- **何を作るか**: 実行可能な成果物
- **なぜ重要か**: LLM やエージェントの中での位置づけ
- **理論**: 数式、形状、直観
- **実装**: 小さくテスト可能なステップ
- **検証**: コマンドと期待される性質
- **観察**: 確認する価値のある値
- **演習**: 焦点を絞ったバリエーション
- **完了基準**: 次へ進む前に必要な証拠

## 数式の読み方

数学の記法は、曖昧さを取り除くためのコンパクトな言語です。このコースに
登場するすべての新しい記号は、その読み方、型、形状、コードに結び付けられます。

たとえば、ドット積は次のように書かれます。

$$
y = \boldsymbol{w}\cdot\boldsymbol{x}
  = \sum_{i=1}^{n} w_i x_i
$$

| 記号 | 意味 | 型 | コード |
| --- | --- | --- | --- |
| $n$ | 要素数 | 正の整数 | `x.size` |
| $\boldsymbol{x}$ | 入力ベクトル | $\mathbb{R}^n$ | `VectorD` |
| $\boldsymbol{w}$ | 重みベクトル | $\mathbb{R}^n$ | `VectorD` |
| $y$ | ドット積の結果 | スカラー | `w.dot(x)` |
| $\sum$ | 範囲にわたる総和 | 演算 | ループまたは `foldLeft` |

数式を見たら、まず入力と出力の型と形状を書き出してください。形状の
トレースは、Transformer のデバッグ手法として最も効果的なものの1つです。

## 実験のルール

- 例をコピーしたら、それを隠してもう一度自分で書きます。
- 2つの条件を比較するときは乱数シードを固定します。
- 一度に変える条件は1つだけにします。
- 期待される結果が分からないままコードを育てないでください。
- 浮動小数点値の比較には根拠のある許容誤差を使います。
- 失敗した実験も、その仮説と観察結果とともに記録します。

## 所要時間の目安

全 40 章のパスとキャップストーンは、演習を含めるとおよそ 250〜400 時間
かかると見込まれます。経験や任意の拡張によって、この範囲は大きく変わり
ます。進捗は、次の問いに答えられるかどうかで定義されます。

> この関数の入力、出力、不変条件、計算量、誤差の原因は何か?

## デバッグの順序

1. 最後に連鎖したエラーではなく、最初の原因となるエラーを読みます。
2. 形状、値の範囲、`NaN`、`Infinity` を調べます。
3. 入力を1要素または2トークンまで減らします。
4. 手計算できる参照値と比較します。
5. 直近に変更した1つの条件を元に戻します。

小さな入力は、大きなモデルの実行結果から推測するよりも確実に、壊れた
不変条件を明らかにします。

## 拡張された実装章の読み方

詳細な章の規約は
[完全な実装章の構造](04-chapter-anatomy.md)で定義されています。完成した
ソースを先頭から順に読むことから始めないでください。次の順序を使います。

1. まずコードを使わずに具体的な数値を手で追います。
2. すべての記号に Scala 名と形状を注記します。
3. 公開 API の事前条件と失敗時の挙動を予測します。
4. 実行順序に沿って実装をトレースします。
5. 各テストを実行する前に、その失敗のしかたを予測します。
6. 焦点を絞ったスイートを実行し、意図的に不変条件を1つ壊します。
7. 実装を元に戻し、観察された失敗を説明します。

これによって、コードリーディングが実験になります。また、章がまだ薄すぎる
場合にそれが明白になります。本文がトレースを導けず、テストの結果を予測
できないなら、そのドキュメントは不完全です。
