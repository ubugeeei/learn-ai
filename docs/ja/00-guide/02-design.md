# 設計原則

これらの原則は、コードと解説の両方に適用されます。観測可能性と正しさは、
便利な抽象化よりも優先されます。

## 1. 教えている仕組みを隠さない

メインパスは Scala 3 と JDK の API のみを使います。BLAS、機械学習、
トークナイザ (tokenizer)、JSON、HTTP のヘルパーライブラリから始めることは
しません。データレイアウト、計算量、アロケーション、数値誤差、失敗時の
挙動を見えるままに保ちます。

外部ライブラリは、その仕組みが実装されテストされた後で、分離された比較用
アダプタとして登場することがあります。

## 2. 不正な状態はコンストラクタと型付き境界で拒否する

- ベクトルと行列の形状不一致は即座に拒否します。
- トークン ID、ツール呼び出しなど、意味を持つ値には別個の型を使います。
- 予期される I/O の失敗は、例外ではなく `Either` で表現します。
- ミューテーション(可変更新)は、訓練ステップ、オプティマイザ更新、
  エージェントループの状態に限定します。

数値カーネルは性能のために配列を所有してもかまいませんが、可変ストレージ
を呼び出し側に公開することはありません。

## 3. 決定性をデフォルトにする

乱数シード、パラメータ初期化、データの順序は明示的です。同じ環境と同じ
シードであれば、テストと章の観察結果が再現されるはずです。非決定的な
並列性を導入する章では、そのトレードオフを測定しなければなりません。

## 4. 小さな入力で正しさを証明する

- 手計算できる単体テスト
- 参照実装と最適化実装の等価性
- 数値微分と自動微分の勾配 (gradient) チェック
- 最適化の前後でのキャッシュ/量子化の等価性
- 直列化のラウンドトリップと破損テスト

大きなエンドツーエンドテストは、焦点を絞った不変条件の代わりにはなりません。

## 5. 形状と単位を含める

すべての Tensor の数式には形状情報を含めます。時間は継続時間として表し、
データはバイト単位で測定し、確率には明示的な範囲を持たせます。

埋め込み (embedding) ルックアップの例:

```text
tokenIds:      [batch, time]
embedding:     [vocabulary, channels]
output:        [batch, time, channels]
```

## 6. 依存の方向を一方向に保つ

```text
apps and chapter experiments
  -> agent / language model / neural network
  -> math and tensor primitives
  -> Scala/JDK standard library
```

下位レイヤーが上位レイヤーに依存することは決してありません。章番号は
教える順序を表すものであり、再利用可能なパッケージ名には漏れ出しません。

## 7. 教育上の簡略化とプロダクションの主張を区別する

各章では次を明らかにします。

- 何を簡略化したか
- どの性質が失われるか
- プロダクション実装が通常は何を追加するか
- 採用の判断をどの測定に基づいて行うべきか

正しい参照実装が、自動的に高速で、安全で、プロダクショントラフィックに
耐えられるようになるわけではありません。

## 8. エージェントの権限は明示的に渡す

エージェントランタイムは、暗黙のファイルシステム、シェル、ネットワークの
権限を持ちません。許可されたツールのリストを受け取り、その入力を検証し、
予算とタイムアウトの範囲内で実行し、構造化された観察結果を記録します。
停止条件、冪等性キー、監査イベントは、プロンプト上の提案ではなく
プロトコルの機能です。

## 9. 一度に1つの学習概念をコミットする

Conventional Commit のタイトルを使います。実装、テスト、対応する章を、
同じまとまりのあるコミットに入れます。

```text
feat(math): implement immutable vectors
feat(autodiff): add scalar reverse-mode differentiation
docs(attention): derive scaled dot-product attention
test(agent): cover malformed tool arguments
```
