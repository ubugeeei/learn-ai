# 34 — JSON、スキーマ、ツールケイパビリティ

## この章で作るもの

依存ライブラリなしの厳格な JSON パーサ、トップレベルオブジェクトのスキーマ
(schema)、明示的な `Tool` ケイパビリティ (capability)、そして構造化された
成功・失敗の観測 (observation) を実装します。

- JSON: `src/main/scala/learnai/json/Json.scala`
- ツールプロトコル: `src/main/scala/learnai/agent/Protocol.scala`

## モデルのツール呼び出しは信頼できない提案

生成された JSON は、決してそのまま実行してよい命令ではありません。

```text
model output
 -> strict JSON parse
 -> lookup in granted tool registry
 -> schema validation
 -> authorization and domain policy
 -> timeout-bounded execution
 -> typed observation
```

ユーザーのテキスト、モデル出力、検索 (retrieval) で取得した文書、ツール出力は、
いずれも信頼できないデータです。

## 依存ライブラリなしの厳格な JSON

このパーサはオブジェクト、配列、文字列、数値、真偽値、null をサポートし、
次を拒否します。

- 末尾の余分なデータやカンマ。
- オブジェクトフィールドの重複。
- 不正な数値文法。
- 不正なエスケープと制御文字。
- 対になっていない UTF-16 サロゲート。
- 入力サイズとネスト深さの制限違反。

重複フィールドは、バリデータが最初の出現を読み、実行側が最後の出現を読むような
場合に危険です。どちらを採用するかを選ぶのではなく、拒否してください。

レンダラは、スナップショットや監査ハッシュを決定的にするために挿入順を保持
します。ただし、アプリケーションのセマンティクスが JSON のフィールド順に
依存してはいけません。

## スキーマ検証

`ToolSchema` は必須フィールド、トップレベルの JSON 型、追加フィールドの
ポリシーを検証 (validation) します。

```scala
ToolSchema(Vector(
  ToolField("message", JsonFieldType.StringValue, "Text to echo", required = true)
))
```

本番のスキーマではさらに、長さ、パターン、範囲、列挙値、ネストの制限、
フィールド間ルールが必要です。スキーマ検証はドメイン検証の代わりには
なりません。

## ケイパビリティベースの設計

実行できるのは、ランタイムの明示的な `Vector[Tool]` に含まれるツールだけです。
モデルには同じ許可済みの定義が見えます。未知の名前は `unknown_tool` を返し、
リフレクションやシェルへのフォールバックはありません。

```text
agent with [weather.read] cannot call filesystem.write
agent with [draft.create] cannot call message.send
```

万能なシェル 1 つよりも、小さなドメイン操作を選んでください。読み取り/書き込み、
下書き/送信、プレビュー/コミットを分離することで、承認と監査が意味を持つように
なります。

## モデルの引数とホストのコンテキストを分離する

モデルが制御する JSON は `arguments` です。ホストが制御する呼び出し ID、
ステップ、プリンシパル、テナント、デッドライン、トレース ID は `ToolContext`
に属し、そこではモデルが認可メタデータを偽造することはできません。

## 構造化された失敗

ツールは `Either[ToolError,JsonValue]` を返し、次を伴います。

- 安定した機械可読コード。
- 安全なメッセージ。
- リトライ可能フラグ。

未知のツール、不正な引数、タイムアウトも観測になります。モデルは呼び出しを
修正したり、失敗を説明したりできます。シークレット、スタックトレース、内部
パスを露出してはいけません。

## プロンプトインジェクション (prompt injection) とツール出力

検索で取得したページには、それまでのポリシーを無視せよという指示が含まれて
いる可能性があります。それでもデータのままです。モデルがその指示に従うかも
しれないと想定し、ランタイムによる強制をプロンプトの外側に保ってください。

- ケイパビリティの許可リスト。
- パス・ドメイン・引数の許可リスト。
- 読み取りと書き込みの分離。
- 影響の大きい操作への承認。
- 出力のサイズ・内容の制限。
- 不変の監査イベント。

プロンプトはセキュリティ境界ではありません。

## 実装ウォークスルー

この章に 2 つのソースファイルがあるのは、2 つの信頼境界をまたぐためです。
`Json.scala` は信頼できないテキストを JSON 構文木に変換します。`Protocol.scala`
は、その木が許可されたケイパビリティの引数コントラクトを満たすかどうかを
確認します。

### 1. JSON ドキュメントをちょうど 1 つだけパースする

`JsonParser.parse` はまず大きすぎる入力を拒否し、次に可変の `Cursor` を作成
します。カーソルは 1 つの整数 `index` を持ち、これは常に次の未消費文字を指し
ます。`parseDocument` は 3 つの操作を行います。

```scala
skipWhitespace()
val value = parseValue(depth = 0)
skipWhitespace()
if index != input.length then fail("unexpected trailing character ...")
```

最後のチェックは不可欠です。これがないと、有効な前半部分のパースに成功した
という理由で `{"path":"safe"} malicious` が受理されてしまう可能性があります。

`parseValue` は先頭文字でディスパッチします。配列とオブジェクトは `depth + 1`
を受け取りますが、スカラ値は受け取りません。チェックは再び降りていく前に
行われるため、深くネストした入力はスタックを使い果たすことなく `Left` で
終了します。

### 2. オブジェクトのループを理解する

`{"x":1,"ok":true}` に対して、`parseObject` は `{` を読み進めた後、次を
繰り返します。

1. 引用符付きのフィールド名をパースする。
2. 可変の `names` 集合にすでに含まれていれば拒否する。
3. `:` を要求する。
4. 値を再帰的にパースする。
5. `}` または `,` を要求する。

カンマの直後の `}` は末尾カンマとして拒否されます。ビルダーが不変の `Vector`
に変換されるのは、オブジェクト全体が有効になった後だけです。`JsonObject` は
独立に一意性をチェックするため、Scala で直接構築されたオブジェクトも、パース
されたオブジェクトと同じ不変条件に従います。

### 3. 便宜的なパースではなく、文字列と数値の文法に従う

`parseString` は JSON の各エスケープを明示的に処理します。`\u` の上位
サロゲートの後には下位サロゲートが続かなければならず、単独の下位サロゲートは
失敗します。だからこそ、2 つの UTF-16 コードユニットとしてエンコードされた
絵文字は、不正なテキストではなく 1 つの有効な Unicode スカラ値になります。

`parseNumber` は `BigDecimal` を呼び出す前に、符号、整数部、省略可能な小数部、
省略可能な指数部を認識します。`BigDecimal` だけでは文法になりません。パーサは
先頭のゼロ、欠落した小数桁、不完全な指数部を明示的に拒否します。

### 4. 何も実行せずにパース済みオブジェクトを検証する

`ToolSchema.validate` は名前からフィールドへのマップを構築し、欠落している
必須フィールドをチェックした後、与えられたすべてのフィールドを走査します。
そして*すべての*問題を返します。

```text
missing required field 'query'; unknown field 'limit';
field 'exact' must be BooleanValue, got JsonString
```

ベクタを返すことで診断とテストが改善されます。それでも実行を認可するわけでは
ありません。構文的に有効な `{"path":"../../secret"}` は文字列スキーマには
一致しても、パスドメインのポリシーには違反する可能性があります。

`IntegerValue` が `JsonNumber` を受理するのは `BigDecimal.isWhole` が true の
ときだけです。したがって `2.0` は数学的には整数ですが、`2.5` はそうでは
ありません。ツールを公開する前に、それが望むワイヤコントラクトかどうかを
判断してください。

### 5. 探索、検証、認可、実行の順序を保つ

`AgentRuntime.executeCall` は信頼パイプラインを実装します。

```text
toolsByName.get(call.name)
  -> definition.schema.validate(call.arguments)
  -> authorize(tool, call, step)
  -> invokeWithRetries(tool, call, step)
```

未知の名前がスキーマのコードに到達することはありません。不正な引数が承認や
ツールのコードに到達することもありません。拒否された呼び出しが物理的な試行を
生むこともありません。この順序は観測可能なセキュリティ特性であり、単なる
最適化ではありません。

## テストの読み方

`JsonSuite` は文法の順に読んでください。ハッピーパスのテストが JSON の値集合
全体を確立し、後続のテストが重複フィールド、カンマ、数値文法、Unicode、
リソース制限をそれぞれ切り出して検証します。パーサの拒否テストは `isLeft`
だけでなく、該当するエラーカテゴリをアサートすべきです。そうすれば、誤った
拒否経路が偶然テストを通過することはありません。

`AgentRuntimeSuite` の「schema errors are returned to the model without
invoking the tool」は、実行回数を数えるフェイクツールを使います。重要な
オラクルは、`invalid_arguments` の観測*と*実行回数ゼロの両方です。最後の
スキーマテストは、欠落・未知・型の問題が最初の 1 件で打ち切られるのではなく
蓄積されることを確認します。

## デバッグチェックリスト

- 有効な JSON の後に続く余分なテキストが報告されない場合は、個々の値パーサ
  ではなく `parseDocument` を調べてください。
- ネストした入力でクラッシュする場合は、オブジェクトと配列の再帰呼び出しが
  深さカウンタを進めているか、制限値が正であるかを確認してください。
- 絵文字が失敗する場合は、入力にリテラルの Unicode が含まれるのか、エスケープ
  された UTF-16 サロゲートコードユニットが含まれるのかを判別してください。
- スキーマ的に有効な入力がドメインエラーを起こす場合も、両方のチェックを
  維持してください。スキーマは「形は正しいか」に答え、ドメインポリシーは
  「許可されているか」に答えます。
- 不正な呼び出しがツールを実行してしまう場合は、`executeCall` 内の順序を調べ、
  リグレッションテストに副作用カウンタを追加してください。
- 実行のたびにスナップショットが変わる場合は、オブジェクトの挿入順を確認し、
  順序が保証されないマップからプロトコル JSON を構築しないようにしてください。

## 演習

1. 整数の範囲制約と文字列長の制約を追加してください。
2. シンボリックリンクによるトラバーサルに耐えるファイル読み取りルートの
   ポリシーを設計してください。
3. 下書きと送信を分離し、送信にのみ承認を要求してください。
4. 出力 JSON のサイズと深さの制限を追加してください。
5. パーサに対するプロパティテスト・ファズテストを作成してください。

## 完了基準

- ツール呼び出しが信頼できない提案である理由を説明できる。
- パース、スキーマ、ドメイン、認可の各検証を区別できる。
- JSON の重複フィールドを拒否する理由を説明できる。
- プロンプトインジェクションがあってもケイパビリティが必要であり続ける理由を
  説明できる。
- 構造化された失敗を観測として返せる。
- `JsonSuite` とツールスキーマのテストが通る。
