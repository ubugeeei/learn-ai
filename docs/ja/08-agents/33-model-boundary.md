# 33 — プロバイダ非依存のモデル境界

## この章で作るもの

ベンダー SDK やワイヤフォーマットをエージェント (agent) のロジックから分離する
`LanguageModel` プロトコルを定義します。モデルは型付けされた履歴と許可された
ツール定義を受け取り、最終回答またはツール呼び出し (tool calling) を返します。

ソースコード: `src/main/scala/learnai/agent/Protocol.scala`。

## モデルとエージェントは別のコンポーネント

言語モデルはトークンを予測します。エージェントは、モデル出力を解釈し、ツールを
実行し、観測 (observation) をモデルに返すホストプログラムです。

```text
agent runtime
  -> ModelRequest
  -> provider adapter
  -> local or remote model
  <- ModelDecision
```

モデル自体はファイルシステムやネットワークへの権限を持ちません。外部への作用は、
ホストが供与するケイパビリティ (capability) を通じてのみ存在します。

## 型付けされた会話履歴

履歴は次の項目を区別します。

- `TextMessage(System|User|Assistant, content)`;
- `AssistantToolCalls(calls)`;
- `ToolObservation(callId, toolName, outcome)`.

すべてのツール結果は、その呼び出し ID と名前を参照します。複数の呼び出しが
位置の順序だけに依存することはありません。

## モデルの決定

プロバイダアダプタはレスポンスを次の形へ正規化します。

```scala
FinalAnswer(text, usage)
RequestTools(calls, usage)
```

API によってはテキストと呼び出しが混在することがあります。プロバイダ固有の
条件分岐をランタイム全体に散在させるのではなく、そのマッピングはアダプタ側で
定義しなければなりません。

## エラーは型付けされたデータ

ネットワーク、認証、レート制限、レスポンス解析の失敗は次の形になります。

```scala
ModelError(code, message, retryable)
```

ツールの失敗は観測として返せるため、モデルはリカバリ方法を選択できます。
モデルの失敗は次の決定を不可能にするため、このランタイムでは実行を終了します。
将来リトライポリシーを導入する場合は、回数、バックオフ、ジッタ、デッドラインを
定義しなければなりません。

## 使用量の集計

各決定は入力・出力トークン数を報告します。実行 (run) はそれらを累積し、コスト、
レイテンシ、コンテキスト上限、暴走ループの分析に使います。プロバイダ固有の
キャッシュトークンや推論トークンといったカテゴリは、追加のメトリクスとして
公開してもかまいません。

## フェイクモデルでテストする

`ScriptedModel` は単体テストでのリモート呼び出しを回避します。

- 決定は事前に固定されています。
- 受け取った履歴とツール定義を検査できます。
- 不正な呼び出しを容易に生成できます。
- テストはネットワーク、レート制限、モデルのランダム性に依存しません。

ランタイムの正しさとモデルの品質は、別々の評価レイヤです。

## アダプタの責務

実際のプロバイダアダプタは次を扱います。

- 内部履歴からプロバイダのメッセージ形式への変換。
- ツールスキーマ (schema) からプロバイダがサポートする JSON Schema サブセットへの変換。
- 認証と HTTP タイムアウト。
- ストリーミングチャンクの組み立て。
- エラーと使用量のマッピング。
- レスポンスのサイズ・深さ制限と厳格なパース。

シークレットはシークレットプロバイダからトランスポートのヘッダへ直接渡し、
プロンプト、履歴、通常のログには決して入れません。

## 実装ウォークスルー

`Protocol.scala` を開き、上から順に読んでください。宣言は、1 回のエージェント
ステップでデータが流れる順序と同じ順に並んでいます。

### 1. 履歴を直和型で表現する

`ConversationItem` は `sealed trait` です。その 3 つの実装は、混同してはならない
3 つのイベントを表します。

```scala
TextMessage(MessageRole.User, "What is in the index?")
AssistantToolCalls(Vector(ToolCall("call-1", "search_documents", arguments)))
ToolObservation("call-1", "search_documents", ToolSucceeded(result))
```

階層が sealed であるため、プロバイダアダプタはすべてのケースをパターンマッチ
でき、新しいケースが処理されていないとコンパイラが警告します。単なる
`Vector[String]` では、ロール、呼び出しと結果の対応関係、データメッセージと
制御メッセージの違いが失われてしまいます。

呼び出し ID は結合キーです。モデルが 1 ターンで 2 つのツールを要求し、それらが
逆の順序で完了したとしましょう。位置による対応付けでは誤った結果を紐付けて
しまう可能性があります。`callId` があれば、ランタイムは実行順序に関係なく
論理的な対応関係を保てます。

### 2. 提案と結果を分離する

`ToolCall` はモデルが制御するデータ、すなわち名前と JSON 引数を含みます。
`ToolObservation` は、何が起きたかというホストが制御する事実を含みます。
モデルは `delete_file` を提案できますが、その成功の観測を作れるのはホスト
だけです。この区別は第 34 章と第 38 章でセキュリティ境界になります。

`ToolOutcome` ももう 1 つの直和型です。

```scala
ToolSucceeded(JsonString("result"))
ToolFailed(ToolError("approval_denied", "...", retryable = false))
```

失敗は例外を投げるのではなく通常の値です。そのため、シリアライズやテストが
可能で、秘匿処理 (redaction) を施した後にモデルへ安全に返せます。

### 3. モデルに見せるのはツールのメタデータだけにする

`ToolDefinition` は名前、説明、スキーマ、作用クラスを含みます。`ModelRequest`
が受け取るのは定義だけであり、実行可能な `Tool` オブジェクトは決して渡され
ません。したがってプロバイダは、操作を実際に行う Scala の参照を受け取ることなく、
ケイパビリティを提示できます。

ホストは実行可能な実装を別に保持します。

```scala
trait Tool:
  def definition: ToolDefinition
  def execute(arguments: JsonObject, context: ToolContext)
      : Either[ToolError, JsonValue]
```

`ToolContext` はランタイムが生成します。モデルが自分の JSON にフィールドを
追加しても、エージェントステップや試行回数を変更することはできません。

### 4. 1 回のモデルターンを正規化する

`LanguageModel.complete` はプロバイダ非依存の単一のシグネチャを持ちます。

```scala
def complete(request: ModelRequest): Either[ModelError, ModelDecision]
```

外側の `Either` は「決定を取得できたか」に答えます。内側の `ModelDecision` は
「モデルが何を決定したか」に答えます。これらは別々の軸です。HTTP タイムアウトは
`Left(ModelError(...))` であり、失敗するツールへの正当なリクエストであっても
`Right(RequestTools(...))` です。

`ModelUsage.+` は成分ごとの加算を行います。ステップ 0 が `(120, 15)` トークン、
ステップ 1 が `(180, 25)` を報告した場合、実行全体の合計は `(300, 40)` になり、
`totalTokens` は `340` です。プロバイダは入力と出力で価格を分けていることが
多いため、両者を分けて保持することが重要です。

### 5. 実際のアダプタはプロトコルの外側に実装する

プロバイダアダプタは次の順序に従うべきです。

1. 型付けされた履歴とツール定義をプロバイダのリクエストに変換する。
2. トランスポートレベルのデッドラインを付けて送信する。
3. サイズとネストの制限のもとでレスポンスをパースする。
4. プロバイダの停止理由を `FinalAnswer` または `RequestTools` にマッピングする。
5. トークンカウンタを `ModelUsage` にマッピングする。
6. 失敗を安定した `ModelError` コードにマッピングする。

プロバイダのレスポンスクラスを `AgentRuntime` に追加してはいけません。
プロバイダの切り替えにループの修正が必要になるなら、境界が漏れています。

## テストの読み方

`AgentRuntimeSuite` は、決定が事前に固定されたスクリプト化モデルを定義して
います。まず「a final model answer completes without executing tools」から
始めてください。これは最小のプロトコルトレースであり、1 つのリクエスト、
1 つの決定、1 つの終端イベントから成ります。次に「a valid tool call becomes
an observation for the next model step」を読みます。このテストは 2 回目の
リクエストを検査し、型付けされた呼び出しと観測が往復を通じて維持されたことを
証明します。

アダプタを追加するときは、記録済みのプロバイダペイロードをアダプタに与え、
得られる `ModelDecision` をアサートするコントラクトスイートを作成してください。
これらのテストはランタイムのテストとは分離し、ワイヤフォーマットの変更が
エージェントループ (agent loop) のリグレッションと誤認されないようにします。

## デバッグチェックリスト

- ツール結果が誤った呼び出しに届く場合は、ベクタ内の位置ではなく `callId` の
  値を比較してください。
- 履歴をシリアライズできない場合は、すべての `ConversationItem` をパターン
  マッチし、網羅性チェックの警告を有効にしてください。
- モデルの失敗がツールの失敗として現れる場合は、アダプタが
  `Either[ModelError, ModelDecision]` のどちら側を返したかを調べてください。
- トークン合計が誤っている場合は、決定ごとの `ModelUsage` をログに出力し、
  成分ごとに加算してください。文字列長から使用量を推測してはいけません。
- プロバイダ固有の条件分岐が `AgentRuntime` に現れた場合は、アダプタに戻し、
  プロトコルレベルのテストを追加してください。

## 演習

1. 2 回目のモデルステップで送信される完全な履歴を図示してください。
2. ストリーミングイベントと最終決定の組み立てを設計してください。
3. レート制限エラーに対する上限付きリトライを設計してください。
4. 2 つのプロバイダアダプタで共有する 1 つのコントラクトスイートを書いてください。

## 完了基準

- モデルとエージェントの責務を分離できる。
- テキスト・呼び出し・観測の対応関係を説明できる。
- プロバイダの詳細がアダプタに置かれる理由を説明できる。
- フェイクモデルで決定的なテストを書ける。
- 使用量と終端のモデルエラーがトレースに現れることを確認できる。
