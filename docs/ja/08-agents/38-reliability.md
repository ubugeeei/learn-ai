# 38 — 信頼性、承認、リトライ

## この章で作るもの

モデルはアクションを提案できますが、そのアクションが認可されているかどうかを決めてはなりません。実用的なエージェントは、3 つの境界を分離します。

```text
model proposal -> schema/capability check -> host authorization -> execution
```

この章では、上限付きランタイムを次のように拡張します。

- すべてのツールに対する明示的な効果 (effect) 分類
- 効果を持つ (effectful) 呼び出しに対する、フェイルクローズド (fail-closed) なホスト承認
- すべての試行に対するタイムアウトと例外の隔離
- エラーと操作の両方が許す場合にのみ行われる、上限付きリトライ
- 論理的な呼び出し ID と物理的な試行番号の区別
- 認可、試行、リトライ、抑制に対する監査イベント

## 1. ケイパビリティは認可ではない

ツール定義をモデルに渡すことは、次を意味します。

> このケイパビリティ (capability) は存在し、モデルはそれに対する引数を提案してよい。

しかし、次を意味するわけでは**ありません**。

> 提案されたすべての呼び出しが承認されている。

ファイル書き込みツールは、ワークフローが時々それを必要とするために利用可能になっているかもしれませんが、特定のパス、差分、タイミングには、依然として人間またはポリシーによる承認が必要です。そのためランタイムは、モデルの提案を検証したうえで、効果を持つ呼び出しの前にホスト所有の `ToolApprover` に問い合わせます。

デフォルトの承認者 (approver) は、効果を持つツールを拒否します。呼び出し側は明示的にオプトインしなければなりません。

## 2. 運用上の効果を分類する

`ToolEffect` には 3 つのケースがあります。

| 効果 | 承認 | 自動リトライ |
| --- | --- | --- |
| `ReadOnly` | 自動 | 許可 |
| `IdempotentWrite` | 必須 | 許可 |
| `NonIdempotentWrite` | 必須 | 禁止 |

これらのカテゴリは、意図ではなく挙動を記述します。

- 公開ドキュメントの読み取りは、通常 `ReadOnly` です。
- リソースを指定された最終値に設定する操作は、`IdempotentWrite` になり得ます。
- カードへの課金、メールの送信、レコードの追記は、通常 `NonIdempotentWrite` です。

操作を誤って冪等 (idempotent) と分類すると、外部への影響が重複する可能性があります。分類はホスト/ツール実装に属するものであり、決してモデルの引数に属するものではありません。

## 3. 承認はフェイルクローズドである

効果を持つ呼び出しに対して、ランタイムは次のように問い合わせます。

```scala
approver.decide(call, definition, ToolContext(call.id, step))
```

この判断には理由が含まれ、`ToolAuthorizationChecked` として記録されます。拒否は、モデルから見える `approval_denied` という観測になり、ツールは呼び出されません。

承認者が例外を投げた場合、アダプター内でタイムアウトした場合、あるいは利用不能な場合、ランタイムはその失敗を拒否に変換します。承認システムの障害を許可として扱うことは、可用性インシデントをセキュリティインシデントへと変えてしまいます。

読み取り専用の操作は、対話的な承認なしで許可されますが、そのポリシー判断もトレースに表現されます。

## 4. 論理呼び出しと物理試行

1 つの `ToolCall.id` は、モデルの論理的なリクエストを識別します。リトライは、その同じリクエストに対する別の物理的な試行です。

```text
call ID = search-17
  attempt 1 -> retryable timeout
  attempt 2 -> success
```

`ToolContext.attempt` は 1 から始まります。`executedToolCalls` はエージェント予算に対して論理呼び出しを数え、一方 `ToolAttemptStarted` と `ToolAttemptFinished` は物理試行を露出させます。

この区別は、コスト、レイテンシ、デバッグ、冪等性キーにとって重要です。HTTP アダプターは、安定した呼び出し ID を冪等性キーとして転送し、試行番号は診断用メタデータとしてのみ転送するとよいでしょう。

## 5. リトライには 2 つの独立した許可が必要である

ランタイムがリトライするのは、次の場合だけです。

$$
\text{error.retryable}
\land
\text{effect.safeToRetry}
\land
\text{attempt} < \text{maximumToolAttempts}
$$

エラーは、観測された失敗を記述します。効果は、操作を記述します。どちらか一方の事実だけでは不十分です。

例えば、ネットワークタイムアウトは多くの場合リトライ可能ですが、決済リクエストは、レスポンスが失われる前にコミットされていたかもしれません。非冪等な決済をリトライすると、二重に課金してしまう可能性があります。ランタイムは代わりに `ToolRetrySuppressed` を発行します。

`maximumToolAttempts` には最初の試行が含まれます。値が 1 なら、自動リトライは無効になります。

## 6. タイムアウトはロールバックを証明しない

既存のランタイムは、試行が締め切りを超えると仮想スレッドに割り込みます。これはローカルの活性 (liveness) を守りますが、割り込みは、リモートシステムがロールバックしたことを証明しません。

```text
local timeout
  != remote cancellation acknowledgement
  != proof that no side effect happened
```

タイムアウトエラーがリトライ可能とマークされながらも、リトライポリシーが依然としてツールの効果をチェックするのは、このためです。プロダクション向けアダプターは、サービスが対応している場合、リモートの冪等性キー、デッドライン、操作ステータスの照会も使うべきです。

## 7. 重複する呼び出し ID

ランタイムは、呼び出し ID ごとに最終的な観測をキャッシュします。

- 同一の呼び出しを繰り返すと、キャッシュされた観測が返されます。
- 内容を変えて ID を再利用すると、`conflicting_call_id` が返されます。
- キャッシュ済みの同一呼び出しに対して、認可と実行は繰り返されません。

これは、モデルが自身の会話状態を見失った後にリクエストを繰り返すことから保護します。プロセスクラッシュに対するリモート冪等性の代わりにはなりません。永続的な呼び出し記録は、後の拡張です。

## 8. 監査イベント列

成功した効果を持つ呼び出しは、次のようなトレースを生成します。

```text
ToolStarted
ToolAuthorizationChecked(approved=true)
ToolAttemptStarted(attempt=1)
ToolAttemptFinished(attempt=1, retryable failure)
ToolRetryScheduled(nextAttempt=2)
ToolAttemptStarted(attempt=2)
ToolAttemptFinished(attempt=2, success)
ToolFinished
```

外側のイベントは、1 つの論理呼び出しを記述します。内側のイベントは、認可と物理試行を記述します。終端の `AgentStopped` は、すべての実行に存在し続けます。

## 実装ウォークスルー

信頼性 (reliability) ポリシーは、`Protocol.scala` の宣言と `AgentRuntime.scala` の制御フローとに分かれています。両方を読んでください。enum が正しくても、分岐条件が間違っていれば、それは依然として安全でないランタイムです。

### 1. 効果ポリシーをホスト所有のメタデータにエンコードする

各 `ToolEffect` ケースは 2 つの値を持ちます。

```scala
enum ToolEffect(val requiresApproval: Boolean, val safeToRetry: Boolean)
```

モデルは、その結果として得られる `ToolDefinition` を受け取りますが、新しい実行可能な `Tool` を作り出したり、ホスト実装に保存された効果を変更したりすることはできません。`ToolApprover.denyEffectful` が `AgentRuntime` のコンストラクタデフォルトであるため、設定を省略しても書き込みが許可されることはありません。

### 2. `executeCall` をセキュリティパイプラインとして追う

既知のツールと有効な引数に対して、`executeCall` は `authorize` を呼び出し、直ちに `ToolAuthorizationChecked` を記録します。拒否は、リトライ不能な観測に変換されます。

```scala
ToolFailed(ToolError("approval_denied", reason, retryable = false))
```

`ToolAttemptStarted` イベントは生成されません。この「不在」は、実装コードより前に認可が行われたことの、テスト可能な保証です。

読み取り専用の呼び出しは、外部の承認者を呼び出さずに内部的な `ApprovalGranted` を生成します。効果を持つ呼び出しは、ホストが作成した `ToolContext` とともに `approver.decide` を呼び出します。`try/catch` は、承認者のすべての例外を `ApprovalDenied` に変換します。フォールバックが許可になることは決してありません。

### 3. リトライ述語を文字どおりに実行する

`invokeWithRetries` は試行 1 から始まり、すべての物理呼び出しの前後で開始/終了を記録します。失敗した結果の後、次を評価します。

```scala
error.retryable &&
attempt < config.maximumToolAttempts &&
tool.definition.effect.safeToRetry
```

これが真のとき、`attempt` をインクリメントし、次の番号とともに `ToolRetryScheduled` を記録します。効果の条件だけが偽のときは、`ToolRetrySuppressed` を記録します。エラーがリトライ不能であるか、試行予算を使い果たしたときは、単に最後の結果で終了します。

`maximumToolAttempts = 3` が許可するのは試行 1、2、3 であって、最初の呼び出しの後に 3 回のリトライを許可するわけではないことに注意してください。

### 4. 安定した論理的アイデンティティを保つ

すべてのリトライは `ToolContext(call.id, step, attempt)` を受け取ります。`call.id` は安定しており、`attempt` だけが変わります。サービスが対応している場合、外部アダプターは呼び出し ID を冪等性キーとして送信すべきです。リトライごとに新しい冪等性キーを生成すると、ランタイムの論理的アイデンティティが台無しになります。

実行レベルの `cached` マップは `(ToolCall, ToolObservation)` を保存します。等価性は、ケースクラスのすべてのフィールドをチェックします。同一の繰り返しリクエストは最終的な観測を再利用し、名前や引数が変わった同じ ID は拒否されます。キャッシュの再利用は、論理呼び出し予算より前、そして認可より前に行われます。

### 5. タイムアウトと例外の結果を正確に解釈する

`invokeWithTimeout` には 3 つの結果チャネルがあります。

| ツールの挙動 | ランタイムの結果 |
| --- | --- |
| `Right(value)` を返す | `ToolSucceeded(value)` |
| `Left(error)` を返す | `ToolFailed(error)` |
| 締め切りを超過する | リトライ可能な `tool_timeout` |
| 例外を投げる | リトライ不能な `tool_exception` |

`ExecutionException` はワーカースレッドからの throwable をラップするため、コードは安全なエラーを作成する前に、その原因 (cause) をアンラップします。プロダクションでは、分類と秘匿化 (redaction) が済むまで、生の例外メッセージを露出させないでください。

エグゼキューターは、すべての分岐において `finally` でシャットダウンされます。これはリソースのクリーンアップであり、リモートのキャンセルやトランザクションのロールバックの証明ではありません。

## 9. テストを実行可能なポリシーとして読む

テストスイートは次を検証します。

- 効果を持つツールがデフォルトで拒否されること
- 明示的に承認された冪等な書き込みが実行されること
- 承認者の例外がフェイルクローズドになること
- 読み取り専用のリトライ可能な失敗が、試行上限内で回復すること
- 試行が単調増加する番号を受け取ること
- 非冪等な書き込みが自動リトライを抑制すること
- タイムアウトがリトライ可能な観測として見えたままであること
- 重複する呼び出し ID が、影響を繰り返す代わりに結果を再利用すること
- 矛盾する呼び出し ID が拒否されること

これらはポリシーテストであり、単なるコードカバレッジテストではありません。認可やリトライのルールを変える変更は、明示的な期待値も変えなければなりません。

`AgentRuntimeSuite` は、カウンターと記録された試行番号を持つツールを使います。拒否のテストでは、実行回数がゼロであることと、`ToolAttemptStarted` イベントがゼロ件であることを検証してください。リトライについては、最終的な成功だけでなく、正確な試行ベクターとイベント順序を検証してください。非冪等なタイムアウトについては、`ToolRetrySuppressed` と、実行が 1 回だけであることを検証してください。

このテストスタイルは、最終回答の検証では見逃されるポリシーの回帰を捕捉します。拒否イベントの前に安全でないコードが呼び出されていたとしても、モデルは拒否をそつなく説明してしまうかもしれません。だからこそ、イベント順序と物理カウンターはオラクルの一部なのです。

## 10. 残されたプロダクション境界

この実装は、いくつかの関心事を意図的に見えるままにしています。

- 承認者インターフェースが同期的であること
- 承認判断が永続的に保存されないこと
- リトライに遅延、ジッター、グローバルな締め切りがないこと
- ツールがまだリソースごとのスコープを露出していないこと
- リモートキャンセルの確認応答がアダプター固有であること
- シークレットと個人を特定できるデータには、別途の取り扱いが必要なこと
- マルチプロセスの冪等性には、永続的な共有ストアが必要なこと

この小さなランタイムは、これらのメカニズムが解決済みであるかのように装うことなく、それらがどこに属するのかを確立します。

## デバッグチェックリスト

- 書き込みが承認なしで実行される場合、ツールのホスト所有の `ToolEffect` を調べ、ランタイムが誤って `allowAll` を受け取っていないことを確認してください。
- 承認者の障害が作業を許可してしまう場合、すべての例外パスが `ApprovalDenied` を構築し、試行イベントを一切生成しないことを確認してください。
- リトライ回数が 1 つずれている場合、`maximumToolAttempts` には試行 1 が含まれることを思い出し、インクリメント前の比較を調べてください。
- 決済のような呼び出しがリトライされてしまう場合、まず効果の分類を修正してください。エラーの `retryable` フラグだけを変えるのでは、ポリシーとして不十分です。
- 重複した呼び出しが予算を消費してしまう場合、キャッシュの照会が、論理呼び出し上限のチェックとインクリメントより前に行われることを確認してください。
- タイムアウトがリモート状態を重複させてしまう場合、外部の冪等性とステータス照会を調べてください。ローカルのスレッド割り込みでは、コミット済みのリモート影響を修復できません。
- 監査の順序が分かりにくい場合、イベントを呼び出し ID でグループ化し、外側の論理呼び出しイベントと番号付きの試行イベントとを分けてください。

## 演習

1. テストのために決定的に注入された時間を使う、指数バックオフを追加してください。
2. 試行間で共有される、呼び出し全体の締め切りを追加してください。
3. ファイルツールに対する、パスにスコープを絞った承認を定義してください。
4. 呼び出し ID と最終結果を、追記専用ジャーナルに永続化してください。
5. 引数をより安全なサブセットに書き換える承認判断を追加し、書き換えられた呼び出しに独自のアイデンティティが必要な理由を説明してください。
6. リモートコミット後のタイムアウトをシミュレートし、非冪等な操作が繰り返されないことを検証してください。

## 完了基準

次のことができれば完了です。

- モデルのケイパビリティ、ホストの認可、実行を区別できる
- 読み取り専用、冪等、非冪等な操作を分類できる
- 3 つの条件からなるリトライ述語を導出できる
- ローカルのタイムアウトがリモートのロールバックを証明しない理由を説明できる
- 1 つの論理呼び出しを、複数の物理試行にまたがって追跡できる
- どの保証が永続的な外部ストアを必要とするかを説明できる
