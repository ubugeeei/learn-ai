# 35 — 上限付きエージェントループと監査トレース

## この章で作るもの

`model -> tool -> observation -> model` を、最終回答、ハードリミット、または
型付けされた失敗によって終了が保証される形で実装します。ソースコード:
`src/main/scala/learnai/agent/AgentRuntime.scala`。

## ステートマシンとしてのエージェントループ (agent loop)

```text
InvokeModel
  | FinalAnswer ----------------------> Completed
  | RequestTools
  v
ValidateAndExecute
  | append observations
  +-----------------------------------> InvokeModel

Any state -> model error / step limit / tool limit -> terminal failure
```

明示的な状態、カウンタ、終端理由は、曖昧なテキストバッファ上の無制限ループ
よりも安全です。

## 1 回のイテレーション

1. 履歴と許可済みツールから `ModelRequest` を構築する。
2. 使用量を累積する。
3. 最終回答なら、アシスタントのテキストを追加して終了する。
4. ツール呼び出し (tool calling) なら、アシスタントの呼び出し項目を追加する。
5. ケイパビリティ、スキーマ、予算を検証 (validation) する。
6. タイムアウト内で実行する。
7. 構造化された観測 (observation) を追加する。
8. 次のモデルステップを開始する。

モデルが作用を知るのは観測を通じてだけです。

## ハードリミット

`AgentConfig` には次が含まれます。

- `maximumModelSteps`;
- `maximumToolCalls`;
- `toolTimeoutMillis`.

これらは活性 (liveness) と安全性の不変条件であって、品質のための推奨値では
ありません。同じ出力を繰り返すモデル、ハングするツール、不正呼び出しの修正
ループでも、必ず終端状態に到達します。本番システムではさらに、実行全体の
デッドラインとコスト・トークン予算が必要です。

## タイムアウトとキャンセル

ツールは JDK の仮想スレッド上で実行され、タイムアウト付きで待機されます。
タイムアウト時にはタスクに割り込みがかけられ、リトライ可能な `tool_timeout`
の観測が返されます。

割り込みは協調的なものであり、強制終了が保証されるわけではありません。ツール
のコードとネットワーククライアントには、それぞれ独自のデッドラインが必要です。
タイムアウト直前に副作用が成功している可能性があるため、無条件のリトライは
副作用を重複させかねません。

## 冪等性 (idempotency)

呼び出し ID は冪等キーとして機能します。

- 同じ ID かつ同じ呼び出し: キャッシュされた結果を再利用する。
- 同じ ID で内容が異なる: 競合として拒否する。

決済、送信、作成の操作では、同じキーを外部サービスにも渡してください。
プロセスクラッシュをまたぐには、永続的な重複排除が必要です。

## 監査イベント

すべての実行は、次の情報とともに成功または失敗を返します。

- モデルの呼び出し・決定と使用量。
- ツールの開始・完了・再利用。
- 終端ステータスと理由。

本番のイベントには、所要時間、トレース ID、プリンシパル、ポリシー判断、
秘匿処理済みの引数ハッシュを追加してもかまいません。シークレットや不要な
個人データをログに残してはいけません。

## 終端状態は区別される

- `Completed`;
- `ModelFailed`;
- `ModelStepLimitExceeded`;
- `ToolCallLimitExceeded`.

上限の使い切りが成功した回答として偽装されることはありません。呼び出し側は、
リトライ、予算の追加、ユーザー承認、手動への引き継ぎを選択できます。

## 軌跡 (trajectory) テスト

テストは最終テキスト以上のものを検査します。

- 有効な呼び出しが次のリクエストの観測になる。
- スキーマ違反ではツールが決して実行されない。
- 未知のケイパビリティは決して実行されない。
- 同一 ID の呼び出しは副作用を 1 回だけ生む。
- 競合とタイムアウトは型付けされたまま保たれる。
- 予算が次の実行を防ぐ。
- 失敗しても完全なトレースが保持される。

実モデルでの評価では、最終的な正しさ、ツール選択、引数の正確さ、ステップ数、
コスト、レイテンシ、安全でない試行を、それぞれ別々に測定すべきです。

## 実装ウォークスルー

`AgentRuntime.run` は、小さなステートマシンのインタプリタとして読んでください。
内部ではローカルな可変変数を使いますが、停止時に公開するのは不変の `AgentRun`
スナップショットだけです。

### 1. 実行状態の全体を把握する

ループは 6 つの変化する値を所有します。

| 変数 | 意味 |
| --- | --- |
| `history` | モデルから見える会話項目 |
| `events` | ホストから見える監査上の軌跡 |
| `usage` | 返されたすべてのモデル決定の合計 |
| `executedToolCalls` | 予算に計上された論理的な新規呼び出し ID |
| `step` | 完了したモデル/ツールのイテレーション数 |
| `cached` | 呼び出し ID から元の提案と観測への対応 |

`stop` はこれらの値をクロージャとして捕捉します。そのため、失敗を含むすべての
リターンパスが同じ完全な形を生成します。トレースを破棄してしまう、例外だけの
結果形式は存在しません。

### 2. 成功するツール往復をトレースする

初期履歴にユーザーメッセージが 1 つ含まれ、モデルが最初に
`RequestTools(Vector(call), usage0)` を返すと仮定します。

1. `ModelInvoked(0, 1)` が追加される。
2. `ModelRequest(history, definitions, 0)` が送信される。
3. `usage0` が実行全体の合計に加算される。
4. `ModelReturned(0, "tool_calls", usage0)` と
   `AssistantToolCalls` が追加される。
5. 呼び出し ID が `cached` に存在しないため、論理予算がチェックされ、
   インクリメントされる。
6. `executeCall` が 1 つの観測と認可・試行イベントを返す。
7. 観測が履歴と ID キャッシュの両方に入る。
8. `step` が 1 になり、次のリクエストにはユーザーメッセージ、アシスタントの
   呼び出し、ツールの観測が含まれる。

2 回目の決定が `FinalAnswer` なら、ランタイムはアシスタントのテキストを追加
して `Completed` を返します。したがって、2 回の決定によるやり取りではモデルが
2 回呼び出されますが、実行される論理ツール呼び出しは 1 回です。

### 3. 上限のチェック位置が異なる理由を理解する

モデルステップの上限は外側の `while` 条件です。ツール呼び出しの上限は新しい
呼び出し ID に対してのみ、物理的な実行の直前にチェックされます。キャッシュ済み
の重複は、追加の論理呼び出しも物理的な試行も消費しません。競合する重複も実行は
されず、型付けされた `conflicting_call_id` の観測になります。

この配置は off-by-one エラーを防ぎます。`maximumToolCalls = 1` の場合、最初の
新規呼び出しは実行され、2 番目の新規呼び出しは副作用が生じる前に実行を停止
させます。

### 4. 論理呼び出しと物理的試行を分離する

`executedToolCalls` は受理された新規 ID を数えます。`ToolAttemptStarted` は
リトライを含む実際の呼び出し回数を数えます。したがって、1 回の論理的な読み取り
呼び出しが 3 回の物理的試行を生むこともあります。この区別は、課金、冪等性の
分析、評価に必要です。

### 5. タイムアウト境界を追う

`invokeWithTimeout` は仮想スレッドのエグゼキュータを作成してツールをサブミット
し、`future.get(timeout, MILLISECONDS)` で待機します。正常な `Right` は
`ToolSucceeded` に、正常な `Left` は `ToolFailed` になります。タイムアウトは
割り込みでキャンセルし、リトライ可能な `tool_timeout` を返します。ワーカーから
の例外は `ExecutionException` からアンラップされ、リトライ不能な
`tool_exception` になります。`finally` ブロックは常にエグゼキュータを
シャットダウンします。

これで抑え込めるのは待機中のスレッドであって、外部の世界ではありません。HTTP
リクエストやデータベーストランザクションには、それぞれ独自のデッドラインと
冪等キーが必要です。

## テストの読み方

`AgentRuntimeSuite` は軌跡の仕様書として使ってください。各フェイクには 1 つの
目的があります。スクリプト化されたモデル決定は経路を制御し、実行回数を数える
ツールは実行の有無を証明し、ブロックするツールはタイムアウトの挙動を検証します。

新しい終端パスを追加するたびに、4 つの層をアサートしてください。

1. `AgentStatus` と最終回答。
2. 履歴の厳密な対応。
3. 関連する監査イベントの順序。
4. 使用量と論理/物理カウンタ。

モデルステップ上限のテストは特に有用です。放置すれば無限に続く修正ループが
終了し、それでも `AgentStopped` が発行されることを証明します。

## デバッグチェックリスト

- 2 回目のモデルリクエストに結果が含まれていない場合は、観測が `events` だけ
  でなく `history` にも追加されているか確認してください。
- 使用量が少なすぎる場合は、決定の種類でマッチする前に使用量を加算し、最終回答
  の決定とツールの決定が同じ集計経路を通るようにしてください。
- 上限を超えて副作用が 1 回余分に許される場合は、インクリメントおよび
  `executeCall` に対する比較の位置を特定してください。
- 重複が 2 回実行される場合は、ID や名前だけでなく、キャッシュされた `ToolCall`
  全体を比較してください。
- タイムアウトした操作が動き続ける場合は、ツール内部に協調的な割り込みと
  トランスポートのデッドラインを追加してください。エグゼキュータのキャンセル
  だけではロールバック機構になりません。
- テストが最終テキストしか確認していない場合は、イベントトレースも検査して
  ください。もっともらしい回答の裏に、安全でない試行が隠れていることがあります。

## 演習

1. 実行全体のデッドラインとトークン予算を追加してください。
2. 指数バックオフとフェイククロックを使った上限付きリトライを追加してください。
3. 観測の順序が決定的になる並列読み取り専用呼び出しを設計してください。
4. 書き込みツールに人間の承認を挟む状態を追加してください。
5. イベントから成功率、ステップ数の p95、ツールエラー率を集計してください。
6. 永続的な冪等性ストアを設計してください。

## 完了基準

- ループを有限ステートマシンとして説明できる。
- 上限が活性をどのように保証するか説明できる。
- タイムアウトと副作用の成功を区別できる。
- エンドツーエンドの冪等性を説明できる。
- 最終回答以外の軌跡メトリクスを列挙できる。
- `AgentRuntimeSuite` が通る。
