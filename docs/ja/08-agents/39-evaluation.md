# 39 — エージェントの評価

## この章で作るもの

流暢な最終回答だけでは、エージェントが正しく動作したことを示すには不十分です。実用的な評価 (evaluation) は、結果、トラジェクトリ (trajectory)、安全境界、運用コストを検査しなければなりません。

この章では、次を記録する決定的なハーネスを構築します。

- 終端ステータスと厳密な結果チェック
- 必須および禁止された、ツールのリクエストまたは実行試行
- 承認拒否など、期待されるポリシー上の失敗
- トークン、モデルステップ、論理ツール呼び出しの予算
- ケースごとのエンドツーエンドのレイテンシ
- 明示的な価格スナップショットから推定されるトークンコスト
- トレースを捨てない、チェック単位の失敗と集計メトリクス

## 1. 評価の単位を定義する

`AgentEvalCase` は次を含みます。

```text
stable case name
fresh LanguageModel factory
initial conversation state
named checks over the final AgentRun
```

モデルは、ケースごとに新しく作成されます。状態を持つフェイクやプロバイダーセッションを再利用すると、判断、キャッシュ、乱数状態がケース間で漏れる可能性があります。

ケース名とチェック名は一意でなければなりません。回帰の履歴、失敗のグループ化、リビジョン間のレポート比較には、安定したアイデンティティが必要です。

## 2. 結果チェック

`ExpectedStatusCheck` は終端状態を検証します。

```text
Completed
ModelFailed
ModelStepLimitExceeded
ToolCallLimitExceeded
```

`ExactAnswerCheck` は、決定的なプロトコル結果をバイト単位で比較します。固定文字列、ID、構造化された出力、既知のフェイクモデルの回答に適しています。

完全一致は、自由記述の自然言語に対しては通常厳しすぎます。将来のセマンティックなチェックやモデル採点によるチェックは、不確実性が文書化された、独立してバージョン管理されるコンポーネントであるべきです。決定的なオラクルを、黙ってジャッジモデルに置き換えてはなりません。

## 3. トラジェクトリチェック

似たように聞こえる 2 つのツールに関する問いは、セキュリティ上の意味が異なります。

### モデルはツールをリクエストしたか?

`ForbiddenToolRequestsCheck` は `AssistantToolCalls` を検査します。ランタイムが実行を拒否した場合でも失敗します。

これは、次のような行動要件に使います。

> 読み取り専用のタスクで、モデルはメッセージ送信を提案してはならない。

### ランタイムはツールを物理的に試行したか?

`ForbiddenToolAttemptsCheck` と `RequiredToolAttemptsCheck` は、`ToolAttemptStarted` イベントを検査します。

これは、次のような効果要件に使います。

> 承認なしに、いかなる書き込みもその実装に到達してはならない。

したがって、拒否された書き込みは、禁止試行チェックには合格しながら、禁止リクエストチェックには不合格になり得ます。両方を保持することで、安全なランタイムが貧弱なモデルポリシーを隠すことも、行儀の良いモデルが安全でないランタイムを隠すことも防げます。

## 4. 期待される失敗のチェック

制御された失敗を生み出すことで成功となるシナリオもあります。例えば、モデルが効果を持つツールを提案し、ホストがそれを拒否しなければならない場合です。

`RequiredToolErrorCheck("approval_denied")` は、モデルが期待されるポリシー結果を観測したことを検証します。同様のケースで、次をテストできます。

- 無効なスキーマ引数
- 利用できないケイパビリティ
- タイムアウト
- 矛盾する冪等性 (idempotency) ID
- リトライの使い果たし

最終ステータスが `Completed` であることだけをチェックするのでは、安全メカニズムが実際に作動したかどうかを見逃してしまいます。

## 5. リソースチェック

ハーネスには次が含まれます。

```text
MaximumTokensCheck
MaximumToolCallsCheck
MaximumModelStepsCheck
```

エージェントにとって、予算は正しさの一部です。無制限にトークンを消費したり、ツールをループし続けたりする回答は、上限内に収まった回答と運用上等価ではありません。

論理ツール呼び出し数は、物理試行数とは異なります。第 38 章では、1 つの論理呼び出しが複数回リトライされることがあります。シナリオがリトライ挙動を特に制約しなければならない場合は、試行回数チェックを追加してください。

## 6. コストはバージョン管理された推定値である

`TokenPricing` は、100 万トークンあたりの入力・出力のドル額を `BigDecimal` として保存します。コストは次のとおりです。

$$
\text{cost} =
\frac{n_{in}p_{in} + n_{out}p_{out}}{10^6}
$$

価格は時間とともに変化し、プロバイダー、モデル、キャッシュ状態、バッチ、サービスティアによって異なります。そのためハーネスは、呼び出し側が提供するスナップショットを受け取り、デフォルトはゼロです。レポートが再現可能なのは、価格の出典と適用日を、純粋な計算の外側に記録している場合だけです。

現時点のベンダー価格を、学習用のコアにハードコードしてはなりません。

## 7. レイテンシは観測値であり、ベンチマークの主張ではない

すべてのケースは、経過した `System.nanoTime` を記録します。これは固定された環境内で大きな回帰を見つけるのに役立ちますが、1 つのサンプルはベンチマークではありません。

性能を主張するには、次も制御して報告してください。

- ウォームアップと反復測定
- 平均だけでなく、パーセンタイルまたはばらつき
- モデル/プロバイダーのリビジョン
- ネットワーク上の位置とサービスティア
- ツール/環境のリセット時間
- 並行度とレート制限
- ハードウェアと JVM
- プロンプト、シード、キャッシュ状態

レポートはケースごとの生の所要時間を保持しているため、別のベンチマークランナーがこれらの制御を追加できます。

## 8. ケースを隠さずに集計する

`AgentEvalReport` は次を計算します。

- 合格したケース数と合格率
- 入力/出力トークン使用量の合計
- モデルステップの合計
- 論理ツール呼び出しの合計
- 経過時間の合計
- 推定トークンコストの合計
- 安定した名前でグループ化された、失敗チェックの件数

また、すべての `AgentEvalResult`、完全な `AgentRun`、チェックの詳細も保持します。集計された合格率は、ケースが失敗した理由を説明できません。トレースなら説明できます。

少なくとも次のスライスは必ず確認してください。

```text
task category
tool/effect category
terminal status
failure check
model/provider revision
prompt or policy revision
latency and token bucket
```

## 実装ウォークスルー

`Evaluation.scala` は、小さく合成可能なチェックから評価を組み立てます。ハーネスは、どの回答が普遍的に良いかを知りません。各ケースが独自のオラクルを宣言します。

### 1. チェックを名前付きの純粋関数として扱う

`AgentRunCheck` は 1 つのメソッドを持ちます。

```scala
def evaluate(run: AgentRun): EvalCheckResult
```

各実装は、イミュータブルな実行からブール値と説明的な詳細を導出します。例えば、`ExpectedStatusCheck` は enum 値を比較し、`MaximumTokensCheck` は `run.usage.totalTokens` を固定された上限と比較します。

1 つのケース内のチェックは、一意な名前を持たなければなりません。この不変条件がなければ、`failedChecksByName("maximum_tokens")` のような集計は、同名の 2 つのエントリが偶発的な重複なのか、それとも別個の要件なのかを識別できません。

### 2. リクエスト集合と試行集合を異なる成果物から導出する

`AgentEvaluation.requestedToolNames` は、モデルから見える履歴を走査します。

```scala
run.history.collect {
  case AssistantToolCalls(calls) => calls.map(_.name)
}.flatten.toSet
```

`attemptedToolNames` はホストの監査イベントを走査し、`ToolAttemptStarted` を選択します。拒否された呼び出しは、リクエスト集合にのみ現れます。リトライされた呼び出しは、イベントには複数回現れますが、返される集合には一度だけ現れます。正確なリトライ回数が重要な場合は、この集合ヘルパーではなく、イベントベクターに対するチェックを書いてください。

### 3. 各ケースを隔離して評価する

`AgentEvaluator.evaluate` はまず、空の入力と重複するケース名を拒否します。各ケースについて、次を行います。

1. 単調増加する開始時刻を記録します
2. `modelFactory()` を呼び出して新しい状態を取得します
3. 設定された `AgentRuntime` を実行します
4. 非負の経過ナノ秒を記録します
5. 同じ完了済みの実行に対して、すべてのチェックを評価します
6. その実行の使用量からコストを推定します

ケースの順序はベクターの順序です。失敗の後にスイートを打ち切るケースは存在しないため、1 つの不良シナリオが後続の結果を隠すことはありません。

### 4. 価格スナップショットを正確に計算する

100 万トークンあたり 3 ドルの入力トークン 2,000 個と、100 万トークンあたり 12 ドルの出力トークン 500 個の場合は、次のようになります。

$$
\frac{2000(3)+500(12)}{1{,}000{,}000}
=\frac{12{,}000}{1{,}000{,}000}
=0.012\text{ USD}
$$

`BigDecimal` によって、この単純な会計計算に二進浮動小数点の誤差を持ち込むことを避けられます。使用量カテゴリと価格スナップショットが不完全である可能性があるため、この数値はあくまで推定値です。

### 5. 証拠を消さずに集計する

`AgentEvalReport` は fold とカウントで合計を計算しますが、そのコンストラクタ引数は完全な結果ベクターのままです。`failedChecksByName` は `groupMapReduce` を使って、失敗したチェックのカテゴリを数えます。

```scala
results
  .flatMap(_.checks.filter(!_.passed).map(_.name))
  .groupMapReduce(identity)(_ => 1)(_ + _)
```

3 つのケースが `maximum_tokens` に失敗した場合、マップは `3` を報告しますが、個々の結果は依然として、実際の上限、使用量、履歴、イベントトレースを保持しています。

### 6. 環境オラクルで拡張する

ファイル状態のチェックは `AgentRunCheck` を実装できますが、セットアップとティアダウンは、純粋な `evaluate` メソッドの中には属しません。ケースの実行をフィクスチャレイヤーで包み、隔離されたディレクトリをリセットし、エージェントを実行し、その後、観測されたファイルハッシュを持つチェックを構築してください。これにより、1 つのケースの副作用が、次のケースの初期状態になることを防げます。

## 9. ライブプロバイダーの前に決定的なフェイクを

ランタイムとポリシーの回帰テストは、スクリプト化されたモデルと決定的なツールを使います。これにより、オーケストレーションの挙動を、リモートモデルのドリフトから切り離せます。

ライブプロバイダーによる評価も依然として必要ですが、それは別の問いに答えるものです。

```text
fake-model eval: is the runtime/policy implementation correct?
live-model eval: how often does this model produce successful trajectories?
```

これらを 1 つのスイートに混ぜると、失敗の診断が難しくなります。

## 10. 環境状態オラクル

`AgentRunCheck` トレイトは拡張可能です。チェックは、実行後に制御されたテスト環境を検査できます。

- ファイルの内容またはハッシュ
- データベース行の状態
- 送信済みメッセージのアウトボックス
- ブラウザアプリケーションの状態
- テストスイートの終了コード
- 返された証拠に対する引用 ID

最終的な文章を採点するよりも、実行可能な状態オラクルを優先してください。すべてのケースの前に環境をリセットし、リセットそのものを検証してください。

## 11. ラボを実行する

```console
$ nix develop -c sbt 'runMain learnai.agent.runAgentEvaluationLab'
```

このラボは、3 つの決定的なシナリオを実行します。

1. 直接の完全一致回答
2. 読み取り専用の照会を物理的に実行しなければならない回答
3. 実行前に拒否されなければならない、提案された非冪等 (non-idempotent) な書き込み

各チェック、トークン、レイテンシ、集計された合格率、論理呼び出し数を出力します。

## 12. テストの読み方

テストスイートは次を検証します。

- 結果、必須ツール、リソース、使用量、コストの集計
- 失敗したチェックが、ケース別・カテゴリ別に見えたままであること
- リクエストされた禁止ツールと、試行された禁止ツールが区別されること
- ケースごとに新しいモデルが作成されること
- ケースの順序が決定的なままであること
- 重複するケース/チェックのアイデンティティが、実行前に失敗すること

`EvaluationSuite` は、失敗の可視性を軸に構成されています。最初のテストは、成功時の集計を証明します。2 番目は意図的にチェックを失敗させ、ケースごとの詳細とグループ化された件数の両方を検証します。3 番目は拒否された書き込みを作り、「リクエストされた」と「試行された」が異なる答えを返すことを証明します。最後のテスト群は、新しいモデルの構築、安定した順序、実行前の検証を確認します。

チェックを追加するときは、合格するケースと失敗するケースを 1 つずつ含めてください。`detail` 文字列に、再実行せずに失敗を診断できるだけの測定値と期待値のデータが含まれていることを検証してください。

## 13. 残されたプロダクション境界

- 永続的な結果ストレージとスキーマのバージョン管理
- プロバイダー/モデル/プロンプト/ポリシーのフィンガープリント
- 環境のリセットと隔離
- 反復した確率的サンプルに対する統計的信頼度
- セマンティック、引用忠実性、較正されたジャッジのチェック
- 敵対的なプロンプトインジェクションと汚染されたツールのスイート
- 計画レベルの DAG メトリクスとチェックポイント復旧の評価
- 人間によるレビューのプロトコルとアノテーター間一致
- 継続的なドリフト検出とリリースゲート

## デバッグチェックリスト

- 評価結果がケースの順序に依存する場合、すべてのモデル、フェイクツール、乱数生成器、環境フィクスチャがケースごとに再作成またはリセットされていることを検証してください。
- 拒否されたツールが試行済みと報告される場合、試行を `AssistantToolCalls` や `ToolStarted` からではなく、`ToolAttemptStarted` のみから導出してください。
- リトライがメトリクスから消えてしまう場合、集合ベースのケイパビリティチェックと、イベント件数のチェックとを区別してください。
- 合計使用量が間違っている場合、レポートの fold を変更する前に、すべての `AgentEvalResult.run.usage` を比較してください。
- コストがプロバイダーの請求書と異なる場合、価格の日付、キャッシュティア、推論トークンのルール、丸め、プロバイダーが報告する使用量カテゴリを記録してください。
- レイテンシの主張が揺らぐ場合、1 回の `nanoTime` 観測をベンチマークとして扱うのをやめ、ウォームアップ、反復、分布の報告を追加してください。
- クリティカルなケースが失敗しているのに合格率が健全に見える場合、ケースごとの結果を調べ、クリティカルなチェックを集計とは独立にゲートしてください。

## 演習

1. 物理的なツール試行回数とリトライ回数の予算を追加してください。
2. セットアップ、チェック、ティアダウンのフェーズを持つファイル状態オラクルを追加してください。
3. 確率的なケースをそれぞれ固定シードで実行し、信頼区間を計算してください。
4. 第 36 章のための、引用の適合率と再現率のチェックを追加してください。
5. ゴールをブロックしている最初のタスクを特定する、計画チェックを追加してください。
6. バージョン付きの JSON レポートを定義し、厳格なパーサーでラウンドトリップしてください。

## 完了基準

- 最終結果、リクエストのトラジェクトリ、物理的な影響を区別できる。
- 決定的なフェイクとライブプロバイダーの評価が、それぞれ異なる問いに答える理由を説明できる。
- トークン、ツール、ステップ、レイテンシ、価格に関する前提を明示できる。
- 集計メトリクスの背後に、ケースごとのトレースを保持できる。
- 機能的な正しさを確認できる場合には、環境状態オラクルを優先できる。
- どのチェックが決定的で、どのチェックが統計的または人間による不確実性の報告を必要とするかを特定できる。
