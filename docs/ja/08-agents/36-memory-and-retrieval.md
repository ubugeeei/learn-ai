# 36 — チャンク分割、ベクトル検索、引用付き検索

## この章で作るもの

文書をソースオフセット付きのチャンク (chunk) に分割し、決定的なハッシュ
ベースラインで埋め込み (embedding) を作り、コサイン類似度でランク付けし、
エージェントツールを通じて引用 (citation) メタデータを返します。ソースコード:
`src/main/scala/learnai/retrieval/Retrieval.scala`。

## コンテキストウィンドウと外部メモリ

モデルの重みは学習時のパターンを圧縮したものであり、正確で最新の、プライベート
文書のデータベースではありません。すべての文書をプロンプトに入れると、
コンテキスト、コスト、アテンションの計算量が増えます。関連しそうなスパンだけを
検索 (retrieval) で取得します。

```text
documents -> chunks -> embeddings -> index
query     -> embedding -> similarity -> top chunks -> model context
```

検索はモデルのパラメータを変更しません。検索で取得したテキストは信頼できない
ままであり、ツールの権限を付与するものでもありません。

## チャンクと来歴 (provenance)

各 `TextChunk` は次を保持します。

- 安定したチャンク ID と文書 ID。
- 文書タイトル。
- ソースの開始・終了オフセット。
- スパンの正確なテキスト。

回答はチャンク ID を引用でき、読者はソースのスパンを確認できます。来歴なしに
テキストをコピーすると、更新の追跡や引用の検証が困難になります。

固定サイズのチャンク分割は分かりやすい一方で、文、見出し、コードブロックを
分断することがあります。本番システムでは、構造を考慮したチャンク、トークン
上限、親子検索を比較します。

## オーバーラップ

オーバーラップは境界周辺のコンテキストを保持します。チャンクサイズ $C$、
オーバーラップ $O$ に対して、ステップサイズは $C-O$ です。オーバーラップを
増やすと再現率 (recall) が向上する可能性がありますが、インデックスサイズと
重複した結果が増えます。

## ハッシュ埋め込みベースライン

この教育用の埋め込み器はトークンを小文字化し、固定数のバケットにハッシュし、
符号ビットを使い、L2 正規化します。

$$
\hat v=v/\lVert v\rVert_2
$$

これはトークンの厳密な一致は捉えますが、意味的な同義性や語順は捉えません。
ハッシュ衝突は避けられませんが、符号付きハッシュによって系統的な正のバイアスが
軽減されます。

このベースラインを使えば、外部のニューラル埋め込みモデルを導入する前に、
インデックス作成、正規化、ランキング、引用、評価を確認できます。

## コサイン検索

クエリと文書のベクトルを正規化すると、コサイン類似度は内積になります。

$$
\cos(q,d)=\hat q\cdot\hat d
$$

結果はスコアの降順で並び、同点はチャンク ID で決定的に順序付けられます。
厳密なスキャンは、近似最近傍インデックスを追加する前の有用なオラクルです。

## 検索ツール

`SearchTool` は読み取り専用です。すべての結果に、スコア、テキスト、文書の
識別情報、ソースオフセットが含まれます。ホストは次を強制できます。

- すべての引用が返されたチャンクを参照する。
- 主張とともにソースのスパンを表示する。
- 証拠を捏造せず、検索の失敗を報告する。
- 取得テキスト内の指示はポリシーではなくデータのままにする。

## 検索の評価

検索は回答生成とは別に測定します。

- Recall@k: 正しい証拠が上位 k 件に現れるか。
- MRR: 最初の正解結果の逆順位。
- 引用の適合率とカバレッジ。
- 回答の忠実性 (faithfulness)。
- レイテンシとインデックスのバイト数。

固定のクエリ/証拠セットを使い、チャンクサイズ、オーバーラップ、埋め込み
モデル、結果件数を比較してください。

## 実装ウォークスルー

`Retrieval.scala` は意図的にパイプライン全体を 1 つのファイルに収めています。
コンポーネントを本番サービスに置き換える前に、1 つの文書をソーステキストから
ツール JSON まで追跡してください。

### 1. チャンク境界を手で導出する

10 文字の文書に対して `maximumCharacters = 4`、`overlapCharacters = 1` と
します。ステップは (4-1=3) なので、開始位置は `0, 3, 6, 9` で、半開区間の
スパンは次のようになります。

```text
[0,4)  [3,7)  [6,10)  [9,10)
```

`substring(start, end)` は同じ半開区間の規約を使います。隣接するチャンクは
境界で 1 つの UTF-16 コードユニットを共有します。ループ条件が
`start < text.length` であるため、最後の短いチャンクも保持されます。

このオフセットは JVM の `String` インデックスであり、Unicode コードポイントの
インデックスでも UTF-8 バイトオフセットでもありません。本番の引用 API は自身の
規約を明示しなければなりません。さもないと、BMP 外の文字の周辺で誤ったスパンを
ハイライトしてしまいます。

### 2. 決定的なハッシュベクトルを構築する

`HashingEmbedder.embed` は `Locale.ROOT` で小文字化し、英字でも数字でもない
文字で分割し、各トークンを処理します。

```scala
val hash = token.hashCode
val bucket = (hash & 0x7fffffff) % dimensions
val sign = if (hash & 0x80000000) == 0 then 1.0 else -1.0
values(bucket) += sign
```

符号ビットをマスクすることで、バケットのインデックスは非負になります。元の
符号ビットは引き続き `+1` か `-1` かを選びます。同じトークンが繰り返されれば
その分だけ加算されるので、これは符号付き bag-of-words の頻度ベクトルです。
最後に `raw.norm` で割ることで、トークンが 1 つもない場合を除き、単位長に
なります。

正規化が変えるのは大きさであって方向ではありません。ある語を 10 回含む文書が、
ベクトルが長いという理由だけで、より短い文書に自動的に勝つことはありません。

### 3. 一度だけインデックスし、厳密にスコアリングする

`VectorIndex.build` はまず文書 ID の一意性を検証します。各文書を独立にチャンク
分割し、すべてのチャンクを埋め込み、不変の `(TextChunk, VectorD)` エントリと
して保存します。検索はクエリを一度だけ埋め込み、ゼロベクトルを拒否し、エントリ
ごとに 1 回の内積を計算してから並べ替えます。

```scala
.sortBy(result => (-result.score, result.chunk.id))
.take(resultCount)
```

スコアを負にすることで、Scala の昇順ソートが関連度の降順になります。完全な
同点はチャンク ID で決着するため、スナップショットや評価がマップの反復順序に
よって変わることはありません。

厳密なスキャンのコストは、チャンク数 (N)、次元数 (D) に対して (O(ND)) です。
後で近似インデックスを導入する際には、正しさのオラクルとして残しておいて
ください。

### 4. 検索をケイパビリティにする

`SearchTool.definition` はちょうど 1 つの文字列フィールド `query` を許可し、
デフォルトで読み取り専用の作用になります。ランタイムは `execute` の前に
スキーマ (schema) を検証 (validation) するため、実装は検証済みの `JsonString`
を安全に取り出せます。

すべての結果は、`chunk_id`、`document_id`、タイトル、オフセット、スコア、
正確なテキストを含むオブジェクトとして返されます。スコアはデバッグに役立ち
ますが、確率ではありません。コサイン値 `0.8` は、回答が正しい確率が 80% で
あることを意味しません。

### 5. ラボを段階ごとに観察する

実験するときは、次を出力して確認してください。

1. チャンク ID、オフセット、正確な部分文字列。
2. ベクトルのノルムと非ゼロのバケット。
3. ランク付けされたチャンク ID と生のコサインスコア。
4. レンダリングされたツール JSON。
5. 回答レイヤが選択した引用。

こうすることで障害を局所化できます。正しい証拠が上位 k 件に一度も入らない
なら、プロンプトエンジニアリングでは取り戻せません。

## テストの読み方

`RetrievalSuite` はパイプラインの順序に従っています。チャンクのテストは
カバレッジ、オーバーラップ、文書の分離を証明します。埋め込み器のテストは
決定性、大文字小文字の正規化 (case-folding)、単位ノルムを証明します。
ランキングのテストは関連する結果とスコアの同点を扱います。失敗のテストは
空のインデックスとトークンのないクエリを扱います。最後のツールのテストは、
返されたテキストの一致だけでなく、構造化された引用メタデータをアサートします。

ニューラル埋め込み器を追加するときは、チャンクとツールのテストを維持して
ください。埋め込み固有のアサーションだけを置き換え、厳密なベースラインに
対して Recall@k を比較する固定の関連性フィクスチャを追加してください。

## デバッグチェックリスト

- 文字が消える場合は、すべての半開区間スパンを列挙し、その和が
  `[0, text.length)` を覆っているか確認してください。
- チャンクのループが終了しない場合は、`overlapCharacters < maximumCharacters`
  が成り立ち、ステップが厳密に正になっているか確認してください。
- 大文字小文字だけが異なる同等の入力から異なるベクトルが得られる場合は、
  マシンのデフォルトロケールではなく `Locale.ROOT` を使ってください。
- コサインスコアが想定範囲を超える場合は、ランキングの前に正規化とベクトルの
  次元を調べてください。
- 同点の順序が変わる場合は、第 2 キーが安定したチャンク ID であることを確認
  してください。
- 引用が誤ったテキストを指す場合は、保存しているオフセットの規約と、利用側の
  バイト/コードポイント/UTF-16 の規約を比較してください。
- 検索が良好なのに回答がハルシネーションを起こす場合は、引用の使い方と忠実性
  を、検索の再現率とは別に評価してください。

## 演習

1. 文境界を考慮したチャンク分割を実装してください。
2. ハッシュの次元数を変えて、衝突と Recall@k を測定してください。
3. BM25 を実装し、ハッシュによるコサインと比較してください。
4. 検索されるチャンクにプロンプトインジェクションを仕込み、権限が変わらない
   ことを確認してください。
5. 引用が返されたチャンク ID を参照していることを検証してください。
