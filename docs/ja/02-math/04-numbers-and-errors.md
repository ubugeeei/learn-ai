# 04 — 数・関数・浮動小数点誤差

## この章で作るもの

`Double` の挙動を明らかにする実験、近似比較、そして補正付き総和 (compensated
summation) を実装します。ソース:
`src/main/scala/learnai/math/Numerics.scala`。

## 数学の実数と計算機の数は異なる

実数 $\mathbb{R}$ は整数、有理数、$\sqrt{2}$、$\pi$ を含み、数直線上に隙間が
ありません。一方、計算機の記憶領域は有限です。Scala の `Double` は 64 ビットの
IEEE 754 binary64 形式で、符号・指数部・有限の仮数部を符号化します。

概念的には次の形です。

$$
(-1)^{\text{sign}}\times1.\text{fraction}\times2^{\text{exponent}}
$$

有限の二進展開を持たない値は丸められます。十進の `0.1` はその一つです。

```console
$ nix develop -c sbt 'runMain learnai.math.runFloatingPointLab'
0.1 + 0.2            = 0.30000000000000004
exactly equals 0.3   = false
approximately equal = true
```

これは有限表現の帰結であって、実行時の欠陥ではありません。ニューラルネットワーク
は膨大な数の加算と乗算を行うため、丸め・オーバーフロー・アンダーフローは
アーキテクチャ上の関心事となります。

## 絶対誤差と相対誤差

近似値 $a$ と基準値 $b$ に対して:

$$
E_{abs}=|a-b|
$$

$10^{-6}$ の誤差は、$10^{-8}$ の近くでは巨大ですが、$10^{12}$ の近くでは
微小です。相対誤差 (relative error) はスケールを考慮します。

$$
E_{rel}=\frac{|a-b|}{\max(|a|,|b|)}
$$

`approximatelyEqual` は、絶対許容誤差とスケールに応じた相対許容誤差の大きい方を
採用します。

```scala
val difference = math.abs(left - right)
val scale = math.max(math.abs(left), math.abs(right))
difference <= math.max(absoluteTolerance, relativeTolerance * scale)
```

ゼロ付近では絶対許容誤差が支配し、大きな値では相対許容誤差が支配します。

## 特殊な値

`Double` は次の値も表現します。

- 正と負の無限大。
- `NaN`。`0.0 / 0.0` のような未定義の数値演算から生じます。

`NaN` は自分自身と等しくありません。訓練 (training run) 中に一度発生すると、
たいていは後続の演算に伝播していきます。値を生成する境界で有限性を検証し、
失敗が原因の近くで起きるようにしてください。

## 加算の順序が有限精度の結果を変える

実数の加算は結合的です。

$$
(a+b)+c=a+(b+c)
$$

有限精度の演算では、非常に大きな数の隣で小さな加数が失われることがあります。
そのため、演算順序が異なる並列リダクションはビット単位では一致しないことが
あります。

Kahan の補正付き総和は、失われた下位ビットの推定値を追跡します。

```scala
val corrected = next - compensation
val updated = sum + corrected
compensation = (updated - sum) - corrected
sum = updated
```

これで演算が厳密になるわけではありませんが、大きさの混在する項に対しては誤差を
減らせることが多い手法です。`VectorD.sum` はこの実装を使っています。

## 数値的安定性

数学的に等価な式でも、有限精度での挙動は大きく異なりえます。たとえばソフト
マックスで `exp(logit)` を直接計算するとオーバーフローする可能性があります。
最大のロジットを引いても確率は一切変わりませんが、すべての指数関数の値は高々 1
に抑えられます。

あらゆる数値式について、次を自問してください。

1. 入力の範囲としてどのような値がありうるか。
2. 中間値がオーバーフローまたはアンダーフローしうるか。
3. 減算によって有効桁が打ち消されないか。
4. 演算回数とともに誤差はどう増大するか。
5. より安定な等価形が存在するか。

## 実装ウォークスルー

`Numerics.approximatelyEqual` は絶対と相対の両方の考え方を組み合わせます。
絶対のみのルールは大きな値で破綻し、相対のみのルールはゼロ付近で不適切な挙動を
します。この実装は、大きい方の絶対値からスケールを計算し、絶対許容誤差と
スケール付き相対許容誤差の大きい方を上限として差を受け入れます。

例を一つ計算してみましょう。`left = 1_000_000.0`、`right = 1_000_000.01`、
絶対許容誤差 `1e-9`、相対許容誤差 `1e-8` とします。

```text
difference     = 0.01
relative bound = 1_000_000.01 * 1e-8 = 0.0100000001
chosen bound   = max(1e-9, 0.0100000001)
result         = true
```

ゼロ付近の値では、絶対誤差の下限があることで、極小のスケールで割るようなロジック
が無意味になるのを防ぎます。`NaN` との比較はすべて false になるため、`NaN` は
減算より前に拒否されます。等しい無限大は明示的に処理され、符号が逆の無限大は
近似的にも等しくありません。

`compensatedSum` は Kahan 流の誤差補正を実装しています。各ステップで
`value - compensation` が、直前の丸められた加算で失われた下位の情報を復元
します。新しい補正値は、補正済みの値を加算したときに生じた丸め誤差を測ります。
これで厳密な実数演算が得られるわけではありませんが、非常に大きな項と非常に
小さな項が混在する総和を大幅に改善します。

`requireFinite` は、Tensor のデータ・パラメータ・更新値が `NaN` や無限大を
含んではならないという不変条件を一箇所に集約します。検証済みの値を返すことで、
別途チェックを重複させることなく代入の中で使いやすくなっています。

## テストの読み方

古典的な `0.1 + 0.2` の例は、厳密な二進の等価性が誤った判定基準 (oracle) である
理由を説明します。大きな値のテストは絶対のみの実装を防ぎます。`NaN` と無限大の
ケースは、不正な比較が偶然通ってしまうことを防ぎます。補正付き総和のテストは、
素朴な総和が情報を失うように大きな寄与と小さな寄与を意図的に並べています。
期待される合計は選んだ項から導出されており、補正付き実装から導いたものでは
ありません。

## デバッグチェックリスト

1. それを報告した最後の演算ではなく、最初に非有限値が現れた場所を特定する。
2. 許容誤差を選ぶ前に、値の大きさを出力する。
3. 減算がオーバーフローしたか、無限大を含んでいなかったかを確認する。
4. 不安定な総和については、順方向・逆方向・整列済み・補正付きの順序を比較する。
5. 表示用に丸めた ML の中間値を計算に戻さない。

## 演習

1. `1e16 + 1.0 == 1e16` を予想してから実行してください。
2. `Vector(1e16, 1.0, 1.0, -1e16)` を並べ替えて、両方の総和を比較してください。
3. `Double.MaxValue * 2.0` と `Double.MinPositiveValue / 2.0` を観察してください。
4. 過大な相対許容誤差によって隠される欠陥を作ってみてください。
