# 05 — ベクトル

## この章で作るもの

有限の `Double` 値を保持する不変な `VectorD` を作り、加算・減算・スカラー倍・
アダマール積・内積 (dot product)・ノルム (norm) を実装します。ソース:
`src/main/scala/learnai/math/VectorD.scala`。

## スカラーからベクトルへ

スカラーは一つの数です。ベクトルはスカラーの順序付きの集まりです。

$$
\boldsymbol{x}=
\begin{bmatrix}x_1\\x_2\\\vdots\\x_n\end{bmatrix}
\in\mathbb{R}^n
$$

`VectorD(2.0, -1.0, 3.0)` の形状 (shape) は `[3]` です。数学の記法では通常 1 から
インデックスを付けますが、Scala は 0 から数えます。

| 数学 | Scala | 値 |
| --- | --- | --- |
| $x_1$ | `x(0)` | `2.0` |
| $x_2$ | `x(1)` | `-1.0` |
| $n$ | `x.size` | `3` |

LLM は一つのトークンを数百から数千個の値、すなわち埋め込み (embedding)
ベクトルで表現します。

## 要素ごとの演算

同じサイズのベクトル同士は、対応する要素を加算します。

$$
\boldsymbol{x}+\boldsymbol{y}=
[x_1+y_1,\ldots,x_n+y_n]
$$

```scala
VectorD(1.0, 2.0) + VectorD(3.0, 4.0)
// VectorD(4.0, 6.0)
```

サイズが異なると有効な対応関係が存在しないため、`VectorD` は切り詰めや
パディングを行わず、演算そのものを拒否します。

スカラー倍とアダマール積は次のとおりです。

$$
a\boldsymbol{x}=[ax_1,\ldots,ax_n]
$$

$$
\boldsymbol{x}\odot\boldsymbol{y}=[x_1y_1,\ldots,x_ny_n]
$$

アダマール積の結果はベクトルのままです。内積はそうではありません。

## 内積

$$
\boldsymbol{x}\cdot\boldsymbol{y}=\sum_{i=1}^{n}x_i y_i
$$

```scala
VectorD(1.0, 2.0, 3.0).dot(VectorD(4.0, 5.0, 6.0))
// 1*4 + 2*5 + 3*6 = 32
```

形状: `[n] dot [n] -> scalar`。結果は、大きさを含めた方向の一致度を測ります。
後の章では、注意機構 (attention) がクエリとキーの内積を関連度スコアとして
使います。

## ノルム

ユークリッドノルム、すなわち L2 ノルムはベクトルの長さです。

$$
\lVert\boldsymbol{x}\rVert_2
=\sqrt{\sum_i x_i^2}
=\sqrt{\boldsymbol{x}\cdot\boldsymbol{x}}
$$

`VectorD(3.0, 4.0).norm` は `5.0` です。コサイン類似度は大きさの影響を取り除き
ます。

$$
\cos\theta=
\frac{\boldsymbol{x}\cdot\boldsymbol{y}}
{\lVert\boldsymbol{x}\rVert_2\lVert\boldsymbol{y}\rVert_2}
$$

検索 (retrieval) の章では、正規化した内積をベクトル検索に使います。

## なぜ内部配列を隠すのか

JVM の `Array[Double]` は連続したプリミティブ格納領域を提供しますが、可変です。
呼び出し側と共有すると、ベクトル API の関知しないところでベクトルの中身が
変わってしまう可能性があります。

`VectorD` は次の表現不変条件 (representation invariant) を守ります。

- 構築時に、すべての値が有限であることを検証する。
- 所有する配列を外部に公開しない。
- 更新と演算は新しいベクトルを返す。
- 演算の前に形状チェックを行う。

一度構築されれば、`VectorD` は不変で有限な値として扱えます。

## なぜカーネル内で `while` を使うのか

数値計算の内側のループでは、要素ごとのオブジェクト生成を避け、演算回数を見える
ようにするため、明示的なインデックスを使います。公開 API は関数型のままです。

- 加算: 時間 $O(n)$、出力メモリ $O(n)$。
- 内積: 時間 $O(n)$、結果メモリ $O(1)$。
- インデックスアクセス: 時間 $O(1)$。

## 実装ウォークスルー

`VectorD` は、効率的なインデックス付きループのためにプライベートな
`Array[Double]` を所有しつつ、値としてのセマンティクスを公開します。公開
コンストラクタは受け取った値を所有ストレージにコピーし、各要素を
`requireFinite` で検証します。`toVector` は別のコピーを返すため、構築後に
呼び出し側が内部配列を書き換えることはできません。

要素ごとの演算は `zipMap` を使います。出力を確保する前に `requireSameSize` を
呼び出します。これにより形状エラーはベクトルの境界で報告され、ループの途中で
配列インデックスの失敗が起きることはありません。出力配列は新規に確保されるため、
`a + b` が `a` や `b` を変更することはできません。

内積を手で追ってみましょう。

```text
a = [1, 2, 3]
b = [4, 5, 6]
products = [4, 10, 18]
sum = 32
```

実装はインデックスごとに積を生成し、`Numerics.compensatedSum` に渡します。
つまり内積は一つのスカラーへのリダクションであり、新しい公開ベクトルの確保では
ありません。`squaredNorm` は `dot(this)` を再利用し、`norm` はその平方根を
取ります。こうして数学的な定義がコードの中に見えるように保たれています。

`updated` は不変性の良い例です。内部配列をクローンし、クローンを変更し、
コンストラクタの経路で検証してから、新しい `VectorD` を返します。`mean`・
`max`・`argmax` は、空のベクトルに対しては結果が存在しないため `Either` を返し
ます。対照的に、空のベクトルの `sum` は加法単位元である `0.0` です。

## テストの読み方

加算のテストは、入力が元の値を保持していることも検証します。これは算術の誤り
だけでなくエイリアシングを捕捉するためです。サイズ不一致のテストは、部分的な
処理が行われる前に失敗が起きることを保証します。有限値のコンストラクタテストは、
以降のすべての演算が依拠する不変条件を守ります。空のリダクションのテストは、
有効な単位元を持つ演算と、未定義の選択操作を区別します。等価性とハッシュの
テストは多くの `Assert.equal` 呼び出しに暗黙に含まれており、オブジェクトの同一性
ではなく配列内容の比較に依存しています。

## デバッグチェックリスト

1. 算術を調べる前に、両方のサイズを出力する。
2. 入力が変化するなら、コンストラクタと結果配列の所有権を調べる。
3. ダイナミックレンジが大きいときだけ内積が誤るなら、素朴な総和と補正付き総和を
   比較する。
4. `argmax` がタイのときに一致しないなら、先頭インデックスを返す方針を文書化して
   テストする。
5. 後の処理で非有限値が現れたら、構築が安全でない内部経路を使っていないか確認
   する。
