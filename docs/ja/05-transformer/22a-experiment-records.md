# 22a — 実験同一性と再現性の記録

## この章で作るもの

同じラベルの付いた 2 つの訓練 (training run) が、必ずしも同じ実験であるとは限りません。シード、コーパスの 1 バイト、オプティマイザのオプション、ソースのリビジョン、Nix のリビジョンのいずれかが変われば、結果は変わり得ます。この章では、リポジトリに既にある厳密な JSON と JDK の暗号機能だけを使って、正準な実験仕様 (canonical experiment specification) と実行マニフェスト (execution manifest) を実装します。

ソース:

- `src/main/scala/learnai/experiment/ExperimentManifest.scala`
- `src/test/scala/learnai/experiment/ExperimentManifestSuite.scala`
- `src/main/scala/learnai/training/MiniGptTrainingLab.scala`

成果物はファイル名の命名規約ではありません。2 つの結果が比較可能かどうかを別のエンジニアが判断できるほど入力が明示された、決定的な実験同一性 (experiment identity) です。

## 1. 再現性は依存グラフである

訓練の結果はモデルアーキテクチャ以外にも多くのものに依存します:

```text
source data bytes
  -> tokenizer and token IDs
  -> split and examples
  -> batch order
  -> model initialization
  -> objective and optimizer state
  -> update schedule
  -> implementation revision
  -> runtime implementation and hardware
  -> observed metrics and artifacts
```

ノードが 1 つでも未知であれば、「同じ実験」は仮定にすぎなくなります。プロフェッショナルな記録は、少なくとも次のものを区別します:

- **論理的入力 (logical inputs):** 意図したデータ、モデル、訓練設定、シード、コード、環境定義
- **実行環境 (execution environment):** それらの入力を実行した JVM とマシン
- **実行状態 (run state):** チェックポイント、データカーソル、乱数状態、完了済みの更新
- **結果 (results):** メトリクス、サンプル、モデル成果物、評価レポート

この章では最初の 2 つを実装します。第 22b 章がメトリクスを生成します。第 22c 章で再開可能な実行状態を追加します。

## 2. 論理的同一性と観測された実行時環境

`ExperimentSpecification` は次を含みます:

| フィールド | 同一性がこれに依存する理由 |
| --- | --- |
| `modelSeed` | 初期化されるパラメータが変わる |
| `MiniGptConfig` | 形状と計算が変わる |
| `MiniGptTrainingConfig` | バッチ、オプティマイザ、スケジュール、検証が変わる |
| `CorpusFingerprint` | モデルに提示される証拠が変わる |
| `codeRevision` | 実装のセマンティクスが変わる |
| `environmentRevision` | ピン留めされたツールと依存関係が変わる |

その正準 JSON がハッシュ化されて `experimentId` になります。

`RuntimeFingerprint` は `ExperimentManifest` によって付加されますが、この論理 ID からは除外されます。同じ意図の実験を 2 つの JVM 上で繰り返し、移植性やパフォーマンスを調べることがあります。その場合、実験 ID は一致しつつ、完全な実行マニフェストは異なるべきです。

ハードウェアが特定の研究における論理的な処理条件の一部でなければならない場合は、仕様に明示的なハードウェアポリシーのフィールドを追加してください。データが公開された後に ID のセマンティクスを黙って変更してはいけません。マニフェストのスキーマをインクリメントしてください。

## 3. 説明ではなく正確なバイト列をハッシュする

SHA-256 は任意のバイト列を 256 ビットに写像します。16 進表現が小文字 64 桁になるのは、各桁が 4 ビットを表すためです:

$$
256/4=64
$$

テストは `abc` の UTF-8 バイト列をハッシュし、標準的な値を確認します:

```text
ba7816bf8f01cfea414140de5dae2223
b00361a396177a9cb410ff61f20015ad
```

1 バイトでも変われば、圧倒的な確率で異なるダイジェストが生成されます。ハッシュが確立するのはバイトの同一性であり、内容の真実性、品質、所有権、安全性、使用許可ではありません。

`CorpusFingerprint.fromText` は意図的に小さく作られています。ラボの正確な UTF-8 ソース文字列をハッシュし、トークン数・訓練用・検証 (validation) 用のカウントを記録します。プロダクションのコーパスビルダーであれば、不変なシャードファイル、ソート済みシャードマニフェスト、トークナイザ成果物、そしてすべての変換設定をハッシュすべきです。可変なディレクトリ名をハッシュしても再現性にはなりません。

## 4. 正準 JSON はプロトコルの一部である

メモリ上のケースクラスをハッシュしても、プロセス間で安定しません。JSON のハッシュが安定するのは、等価な内容がただ 1 つのバイト表現を持つ場合だけです。リポジトリの `JsonObject` は挿入順を保持し、重複フィールドを拒否します。コンパクトレンダラは数値の表記を正規化します。

仕様のエンコーダは固定のフィールド順を使います:

```text
name
model_seed
model
training
corpus
code_revision
environment_revision
```

ネストされたモデル、スケジュール、オプティマイザ、コーパスの各オブジェクトも固定順です。省略可能な勾配 (gradient) クリッピングは、数値または JSON の `null` としてエンコードされ、意味が変わり得る「省略されたフィールド」としては決してエンコードされません。

同一性は次のとおりです:

$$
id=\operatorname{SHA256}(\operatorname{UTF8}(canonicalJson))
$$

後からフィールド順を変更すると、意味的な値が同じでも ID が変わってしまいます。だからこそ実行マニフェストには `schema_version` があり、将来のデコーダはスキーマの進化を明示的に扱わなければなりません。

## 5. バリアントをタグ付きデータとして表現する

学習率ポリシーは sealed trait です。正準 JSON は、どのバリアントが選択されたかを保持します:

```json
{"kind":"constant","value":0.02}
```

または:

```json
{
  "kind":"warmup_cosine",
  "peak":0.02,
  "minimum":0.002,
  "warmup_updates":5
}
```

最初の学習率だけを記録すると、その後のスケジュールが失われます。任意の Scala の `toString` を保存すると、同一性がコンパイラ生成のフォーマットに結合してしまいます。タグ付きのプロトコルデータは、バリアントを明示的かつパース可能にします。

同じルールは、将来のオプティマイザ (optimizer)、精度モード、データサンプラ、分散戦略にも当てはまります。安定した kind と、挙動に影響するすべてのパラメータをエンコードしてください。

## 6. ソースリビジョンと環境リビジョンは交換可能ではない

`codeRevision` はリポジトリの内容を識別し、通常は完全なコミット SHA と clean/dirty 状態のポリシーです。`environmentRevision` は Nix の flake lock、または他の不変な環境定義を識別します。

Scala の依存関係をピン留めしてもソースを記録しなければ不十分です。ソースを記録してもピン留めされていないコンパイラ/JDK を使うのであれば、これも不十分です。両方のフィールドが必須で、空であってはなりません。

ラボでは、呼び出し側が JVM プロパティを与えない限り、これらのデフォルトは `unrecorded` になります:

```console
$ nix develop -c sbt \
  -Dlearnai.codeRevision=<commit> \
  -Dlearnai.environmentRevision=<flake-lock-hash> \
  'runMain learnai.training.runMiniGptTrainingLab'
```

`unrecorded` は正直な警告であり、再現可能なリビジョンではありません。将来のラン起動ツールは、訓練の前にこれらの値を導出し検証します。

## 7. 実行マニフェストの内容

`ExperimentManifest` は次をレンダリングします:

```text
schema_version
experiment_id
specification
runtime
```

実行時情報には、Java ランタイムのバージョン、VM の名前/バージョン、オペレーティングシステム、アーキテクチャ、利用可能なプロセッサ数が含まれます。第 22 章のベンチマークも同じ型を使うため、品質の証拠とパフォーマンスの証拠が環境に関する語彙を共有します。

これはまだ完全な実行成果物ではありません。開始/終了時刻、ホスト同一性のポリシー、メトリクス、チェックポイントのハッシュ、コマンドライン引数、終了ステータス、失敗理由はまだ含まれていません。それらは不変な論理仕様ではなく、追記専用の実行記録に属します。

## 実装ウォークスルー

### `CorpusFingerprint`

構築時に、空でない名前、小文字 64 桁のハッシュ、非負のカウントを検証します。`fromText` は正確な Java の `String` を UTF-8 バイト列に変換し、プライベートな `Sha256` ヘルパーに渡してから、検証済みの値を構築します。

`MessageDigest.getInstance("SHA-256")` がダイジェストを計算し、JDK の `HexFormat` が小文字の 16 進表現を生成します。プラットフォーム既定のテキストエンコーディングは使いません。

### `ExperimentSpecification`

ケースクラスの構築時に、人間向けの識別情報と両方のリビジョンを検証します。その後 `canonicalJson` を一度だけ構築し、そのコンパクトなレンダリングをハッシュします。モデルと訓練のオブジェクトはそれぞれのコンストラクタで既に検証されているため、あり得ない負の次元やバッチサイズがマニフェストに入り込むことはありません。

ケースクラスの `copy` は新しいインスタンスを生成するため、JSON と ID が再計算されます。テストはこの挙動を利用して、シード、コーパスのカウント、コードリビジョン、環境リビジョン、バッチ設定の変更が同一性を変えることを示します。

### `ExperimentJson`

プライベートなエンコーダメソッドが各ドメイン型を `JsonObject` に写像します。プライベートに保つことで、任意の呼び出し側が別の正準順序を発明することを防ぎます。Double のフィールドは `JsonNumber` に入る前に `BigDecimal.decimal` を通し、2 進浮動小数点表現の長い 10 進展開を避けます。

`LearningRateSchedule` に対するパターンマッチは、trait が sealed であるため網羅的です。新しいスケジュールを追加すると、コンパイル時にこのエンコーダの更新が強制されます。

### `ExperimentManifest`

マニフェストは仕様と実行時情報を型付きの値として保持します。`experimentId` は仕様に委譲します。`json` は正準な仕様を再エンコードせずにラップするため、ID のペイロードと表示されるペイロードがずれることを防ぎます。

`render` は決定的ですが、現時点ではメモリ内にとどまります。アトミックな永続化、チェックサム、安全な読み込み、スキーママイグレーションは、第 25 章の成果物の原則を再利用します。

## テストの読み方

`ExperimentManifestSuite` は、`abc` に対する公開の SHA-256 オラクルから始まります。これは実験モデルとは独立に、バイトエンコーディングと 16 進レンダリングをテストします。

等価性テストは 2 つの仕様を独立に構築し、それらの JSON と ID を比較し、レンダリングされた JSON を厳密なパーサでパースします。バリアントテストは論理的な境界を 1 つずつ変更し、新しい ID になることを要求します。

実行時テストは意図的に VM 名だけを変更します。ID は等しいまま、完全なマニフェストは異なることで、論理と実行の分離が証明されます。必須フィールドのテストは、レンダリングされた出力の中からすべての主要な再現性境界を探します。一方、不正なハッシュや空のリビジョンのテストは、マニフェストが存在する前に失敗します。

これらのテストは決定的な同一性のセマンティクスを確立します。呼び出し側が真実のコミット、完全なコーパス、正しいカウントを供給したことは証明しません。取得と証明 (attestation) は別の責務です。

## デバッグチェックリスト

- 等しい実験が異なる ID になる場合は、正準 JSON をバイト単位で差分し、まずフィールド順と 10 進表記を調べます。
- 設定を変更しても ID が同じままの場合は、そのフィールドが Scala のケースクラスに保存されているだけでなく、`ExperimentJson` でエンコードされていることを確認します。
- コーパスのハッシュがマシン間で変わる場合は、ディレクトリ名ではなく、正確なバイト列、正規化、改行の規約、シャード順、圧縮を比較します。
- JSON のラウンドトリップで数値が変わる場合は、`BigDecimal` の構築とレンダラの正規化を調べます。
- 実行時環境の変更が実験 ID を変えてしまう場合は、実行時情報が `ExperimentManifest` によってのみ付加されていることを確認します。
- ID から結果を再現できない場合は、欠けている論理的入力を特定します: トークナイザ、データ変換、精度、ソース、環境、シード。
- `unrecorded` が本格的な訓練に到達した場合は、プレースホルダを有効な来歴として扱うのではなく、計算資源を費やす前にランチャーを失敗させます。

## 演習

1. トークナイザ成果物の同一性と、ソート済みシャードマニフェストのハッシュを追加する。
2. アトミックなマニフェストライタと、厳密なスキーマバージョン付きリーダを実装する。
3. 無制限の diff を埋め込まずに、dirty なワークツリーの状態を記録する。
4. 論理的な実験同一性を保ったまま、時刻、ステータス、メトリクス、成果物ハッシュを含む追記専用の実行記録を追加する。
5. スキーマ 1 からスキーマ 2 へのマイグレーションを定義し、古い ID が解釈可能なままであることを証明する。
6. マニフェストに署名し、署名検証が何を証明し、何を証明しないかを説明する。

## 完了基準

- 論理的な実験同一性、実行時の観測、実行状態、結果を区別できる。
- 正準なシリアライゼーションがハッシュプロトコルの一部である理由を説明できる。
- 現在の実験 ID を変えるすべてのフィールドを述べられる。
- SHA-256 によるコーパスハッシュが何を証明し、何を含まないかを説明できる。
- 同じ型付き入力からマニフェストと ID を再現できる。
- 実行時環境だけを変えても論理的同一性が保たれることを示せる。
- 実行途中からの厳密再開 (exact resume) に必要な追加の状態を特定できる。
