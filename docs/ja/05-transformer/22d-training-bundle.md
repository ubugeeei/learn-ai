# 22d — 永続化された訓練バンドル

## この章で構築するもの

第 22c 章では、訓練 (training run) の完全な状態が 1 つの値に収まり、そこからの再開 (resume) がビット単位 (bitwise) で厳密であることを証明しました——ただし 1 つのプロセスの内部で、です。現実の中断はプロセスを殺します。この章では、契約を弱めずに状態をディスクへ移します。ファイルから再開した実行は、中断のない実行と依然としてビット単位で等しくなければならず、バンドル (bundle) は自分が属していない実験の再開を拒否しなければなりません。

この章では次を行います:

- `MiniGptTrainingState` をビットレベルで厳密な double でシリアライズする
- 第 25 章のフォーマット規律を再利用する: マジック、バージョン、境界付き読み込み、末尾の SHA-256、アトミックな置き換え
- すべてのバンドルを第 22a 章の `experimentId` に結び付け、一致しない再開を明示的に拒否する
- 一気通貫の実行とディスクからの再開を比較するビット単位のオラクルで、経路全体を検証する

## 前提知識

第 22a 章 (正準な実験同一性 (experiment identity))、第 22c 章 (状態が何を含み、なぜ含むのか)、第 25 章 (このファイルレイアウトが踏襲するチェックポイントフォーマットの規律) を理解している必要があります。

## 1. 厳密さがシリアライゼーションを生き延びるのは意図した場合だけ

再開の契約はビット単位なので、ファイルフォーマットもそうでなければなりません。`DataOutputStream.writeDouble` は正確な IEEE 754 のビット (内部では `doubleToLongBits`) を書き込み、`readDouble` がそれを復元するため、すべての損失 (loss)、モーメント、パラメータが変わらずに生き残ります。魅力的に見える代替——JSON や CSV のテキストダンプ——は、契約を黙って破ります。10 進テキストは丸めを生じ、1 ulp だけ丸められた値は、数回の更新の後に一気通貫の実行から発散する訓練軌跡を生み出します。状態を人間可読にしたいなら、バンドルの隣に*レポート*をレンダリングしてください。レポートを真実の源にしてはいけません。

## 2. バンドルは実験に属する

「問題なく読み込める」状態ファイルでも、これから続けようとしている実行にとっては誤りであり得ます。同じアーキテクチャで異なるコーパス、同じコーパスで異なるスケジュール、すべて同じで異なるコードリビジョン。第 22a 章が同一性を既に解決しています——`experimentId` は、正準なモデル/データ/訓練/コード/環境仕様の SHA-256 です。バンドルはその id を保存し、`loadForResume(path, expectedExperimentId)` はそれを、*呼び出し側が現在の設定から導出した*同一性と比較します:

```text
caller: id = ExperimentSpecification(...).experimentId   (from live config)
bundle: id stored at save time
equal  -> resume
differ -> explicit refusal naming both ids
```

拒否は警告ではなくエラーです。一致しない再開はそれ自体では大きな音を立てて失敗しません——もっともらしく見えるものの比較不能な実行を訓練してしまい、それは最も高くつく種類の誤りだからです。

## 3. フォーマット規律の再確認

このレイアウトが第 25 章のチェックポイント規則に従うのは、失敗モードが同一だからです:

```text
magic "LAIBND01" | version | experiment id
| completedUpdates | tokensSeen | best/initial validation | RNG counter
| parameter value groups | optimizer step | first moments | second moments
| SHA-256 over everything above
```

ファイルから読み取られるすべてのカウントは、割り当ての前に境界チェックされます (テンソル数、テンソルごとのスカラー数、文字列長、ファイル全体のサイズ)。そのため、壊れたファイルや悪意のあるファイルが途方もない割り当てを要求することはできません。チェックサムは、あらゆるパースの前に検証されます。書き込みはまず一時ファイルに行われ、アトミックに所定の位置へ移動されるため、保存途中のクラッシュは、書きかけのバンドルではなく、以前のバンドルを無傷のまま残します。読み込まれた値は `MiniGptTrainingState` と `AdamWSnapshot` のコンストラクタを再び通過し、その不変条件 (有限性、非負の 2 次モーメント、記録の範囲) は、チェックサム単体では確率的にしか捕捉できない破損を拒否します。

## 4. 実験の実行

```console
$ nix develop -c sbt 'runMain learnai.testing.AllTests'
```

`TrainingBundle` のスイートは、実際の一時ファイルを通して訓練・保存・再開を行い、破損と拒否の経路も含みます。

## 5. 実装ウォークスルー

`save` はまず、1 バイトも書き込まれる前に実験 id のフォーマットを検証します。そのため、拒否された保存がファイルを一切残さないことが証明できます。その後、状態を固定のフィールド順で `DataOutputStream` を通してストリームし、ペイロードの SHA-256 を付加し、結果を `writeAtomically` に渡します。これは第 25 章が使うのと同じ「一時ファイル+アトミック移動」の手順であり、アトミック移動をサポートしないファイルシステムのためのフォールバックも含みます。

`load` はその鏡像に、さらに疑い深さを加えたものです: 読み込みの前のサイズ境界、パースの前のチェックサム、フィールドの前のマジックとバージョン、各割り当ての前の境界付きカウント、そして最後の末尾バイトのチェックにより、付け足されたペイロードが気付かれずに紛れ込むことはできません。状態は、信頼したフィールド代入ではなく、検証を行うコンストラクタを通して再構築されます。

`loadForResume` は `load` と同一性の比較を合成し、状態だけを返します。再開するトレーナーが必要とするのはそれだけだからです。より広い `load` は検査ツールのために存在します。その後の再開は、まさに第 22c 章*そのもの*です: `ResumableMiniGptTraining.train(freshModel, split, config, state, n)`。

値グループのエンコーディング (カウント、続いてグループごとのカウントと double) はパラメータ値と両方のモーメント配列で共有されるため、1 組の境界付きリーダ/ライタが 3 つのセクションすべてをカバーします。

## 6. テストの読み方

- ラウンドトリップは、読み込まれた状態を保存された状態とケースクラスの等価性で比較します——すべての double、モーメント、カウンタをビット単位で。
- ディスク再開のオラクルは、4 回更新の訓練を行い、保存し、*ファイルと呼び出し側の同一性以外のすべてを忘れ*、読み込み、異なるシードのモデルへさらに 4 回の更新を再開し、一気通貫の 8 回更新の実行のメトリクスと最終的な重みに厳密に一致させます。
- 拒否テストは、ある同一性で保存し、別の同一性で再開を試み、メッセージが両方の id を名指しして拒否することをアサートします。
- 破損テストは、ペイロードの 1 バイト反転 (チェックサム)、ファイルの切り詰め、マジックの破壊を行います。すべて、いかなる状態も漏れ出す前に失敗しなければなりません。
- フォーマットテストは、不正な形式の実験 id を拒否し、拒否された保存がファイルを作らないことを証明します。

## 7. デバッグチェックリスト

1. ディスクから再開した実行が一気通貫の実行から発散する場合は、まず第 22c 章のメモリ内スイープを再実行してください。それが通れば、バグは訓練ではなくシリアライゼーションにあります。
2. 読み込まれた状態を、保存された状態とフィールドごとに差分してください。最初に等しくないフィールドが、順序を誤ったか型を誤った書き込みを名指しします。
3. 厳密さを、出力された 10 進数で決してデバッグしないでください——`doubleToLongBits` の値を比較してください。
4. 拒否が予期せず発火する場合は、両方の正準な仕様 (第 22a 章) をレンダリングして JSON を差分してください。異なっているフィールドがドリフトです。
5. 読み込めるのにコンストラクタの不変条件で失敗するバンドルは、フォーマットバージョンがバージョンを上げずにセマンティクスを変えたことを意味します。

## 8. テストすべき失敗モード

- double がテキストを経由して丸められる
- フィールドが異なる順序で書き込まれ、読み込まれる
- チェックサムがパースの前ではなく後に検証される
- 壊れたヘッダから境界のないカウントが割り当てられる
- 宛先ファイルの切り詰めと書き直しの間のクラッシュ
- 異なる実験同一性の下で再開が受け入れられる
- 古いファイルを黙って再解釈するバージョンアップ

## 演習

1. `ExperimentSpecification.experimentId` をエンドツーエンドで接続する: 保存と再開の両方で、生きた設定から id を導出し、コーパスのテキストだけを変えて拒否を実証する。
2. 再開可能なトレーナーに、$m$ 回の更新ごとの定期的なバンドル書き込みを追加し、そのオーバーヘッドを正直に計測する (第 22 章のルール)。
3. 保持ポリシーで直近 $k$ 個のバンドルを保持し、部分書き込みをシミュレートした後に、最新の検証済みバンドルが選ばれることを証明する。
4. パラメータ値を読み込まずに、バンドルのメタデータと要約統計を出力する小さな検査コマンドを書く。

## 完了基準

次のことができれば完了です:

- ビット単位の再開が 10 進テキストのシリアライゼーションを禁じる理由を説明できる。
- 検証の順序 (サイズ、チェックサム、マジック、バージョン、境界付きフィールド、末尾バイト) と、各ステップが何から守るのかを列挙できる。
- 同一性の拒否が警告ではなくエラーでなければならない理由を説明できる。
- アトミックな置き換えがクラッシュをまたいで何を保証するかを記述できる。
- 検証がどこに存在するか (コンストラクタ)、そしてチェックサムがあっても読み込み時の再検証が重要である理由を述べられる。
