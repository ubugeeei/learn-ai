# 22 — 正しさの診断とパフォーマンス測定

## この章で作るもの

コンパイルが通り、訓練 (training) の損失 (loss) が下がるモデルでも、なお
間違っていることがあります。期待どおりのトークンを返すモデルでも、なお
不必要に遅いことがあります。この章では、再利用可能な 3 つの診断レイヤーを
追加します。

1. 明示的なバイト集計を伴う密なパラメータインベントリ
2. 決定的に選ばれたパラメータ座標に対する中心差分 (central difference) の
   勾配 (gradient) チェック
3. ウォームアップ、バッチ化、生サンプル、記述統計、観測可能なチェックサム、
   ランタイムフィンガープリントを備えたローカルなベンチマーク (benchmark)
   ハーネス

本体のソース:

- `src/main/scala/learnai/diagnostics/GradientCheck.scala`
- `src/main/scala/learnai/diagnostics/Benchmark.scala`
- `src/main/scala/learnai/transformer/MiniGptDiagnosticsLab.scala`

実行可能な仕様:

- `src/test/scala/learnai/diagnostics/GradientCheckSuite.scala`
- `src/test/scala/learnai/diagnostics/BenchmarkSuite.scala`

目的は、緑のスコアをでっち上げることではありません。*何と何を比較したのか、
どこが異なったのか、タイミングはどれほどばらついたのか、どの環境がその観測を
生んだのか*に答えられるだけの証拠を保持することが目的です。

## 1. 3 種類の正しさを区別する

### 数学的な正しさ

実装は、意図した関数と導関数を計算しなければなりません。例:

- 行列積が正しい内側の次元を縮約する
- causal attention (causal attention) が未来の位置に確率も勾配も割り当てない
- 交差エントロピーの微分が `softmax - oneHot` になる
- 共有された出力射影が勾配を共有埋め込み (embedding) に累積する

このレイヤーは、手計算、有限差分、代数的性質、独立したリファレンスパスに
よってテストします。

### 数値的な正しさ

代数的には正しい式でも、有限精度では破綻することがあります。例としては、
`exp(logit)` のオーバーフロー、ほぼゼロのノルムによる除算、ほぼ等しい 2 つの
値の減算などがあります。数値テストには、現実的なスケール、明示的な許容誤差、
`NaN` と無限大のチェックが必要です。

### システムとしての正しさ

正しい数値が、誤った動作特性で計算されていることもあります。キャッシュが
等価なロジットを返しながら、トークンごとにメモリ確保をしているかもしれま
せん。リトライが書き込みを重複させた末に成功しているかもしれません。
ベンチマークが見事な時間を報告しているのは、結果が使われず最適化で消えた
からかもしれません。

システムのテストは、最終的な値だけではなく、所有権、作業カウンタ、バイト数、
呼び出し回数、イベントの順序、レイテンシのサンプル、環境の状態を検査します。

単一のテストで 3 つのレイヤーすべてをカバーすることはできません。

## 2. 正しさのはしごを使う

新しいカーネルやモデルのパスを実装するときは、小さいものから大きいものへと
順に検証します。

```text
hand-computable scalar/vector case
  -> operation property
  -> finite-difference local gradient
  -> reference/optimized equivalence
  -> full-block causality and shape
  -> one-batch overfit
  -> deterministic end-to-end run
  -> statistical evaluation on held-out data
```

最初に end-to-end の実行が失敗すると、探索空間はシステム全体になります。
要素 3 個の性質が失敗するなら、原因の演算はたいてい明らかです。
プロフェッショナルなデバッグは、下位の不変条件が通ってからはじめて上へ
進みます。

## 3. 中心差分を導出する

スカラー損失 $L(\theta)$ と 1 つのパラメータ座標 $\theta_i$ について、現在の
点のまわりでテイラー展開すると次を得ます。

$$
L(\theta_i+h)
=L(\theta_i)+hL'(\theta_i)+\frac{h^2}{2}L''(\theta_i)
+\frac{h^3}{6}L'''(\theta_i)+O(h^4)
$$

また、

$$
L(\theta_i-h)
=L(\theta_i)-hL'(\theta_i)+\frac{h^2}{2}L''(\theta_i)
-\frac{h^3}{6}L'''(\theta_i)+O(h^4)
$$

両者を引き算すると、定数項と偶数次の項が打ち消されます。

$$
L'(\theta_i)
\approx
\frac{L(\theta_i+h)-L(\theta_i-h)}{2h}
$$

打ち切り誤差は $O(h^2)$ であり、片側差分の誤差 $O(h)$ より優れています。
しかし `h` を無制限に小さくすることはできません。ほぼ等しい損失同士の浮動
小数点の減算は、おおよそ $O(\epsilon_{machine}/h)$ のように振る舞う丸め誤差を
持ち込みます。したがって勾配チェックには、適度なステップ幅——このような
小規模な `Double` 計算では `1e-5` 付近が一般的です——と、厳密な一致を仮定
するのではなくスケールを考慮した許容誤差が必要です。

## 4. 勾配プローブを 1 つ手で解く

次の関数を考えます。

$$
L(x_0,x_1)=x_0^2+x_1^2
$$

点 $(x_0,x_1)=(1.5,-2)$ において、リバースモード自動微分は次を報告するはず
です。

$$
\nabla L=(2x_0,2x_1)=(3,-4)
$$

$x_0$ について $h=10^{-5}$ を使うと、

$$
g_{num}
=\frac{(1.5+h)^2+(-2)^2-((1.5-h)^2+(-2)^2)}{2h}
\approx3
$$

テストではまさにこのパラメータ Tensor を構築し、`backward` を実行し、3 つの
座標すべてをプローブし、解析的な値と数値的な値を比較し、最後に元のパラメータ
ベクトルが変化していないことをアサートします。

この復元のアサーションは重要です。モデルを黙って摂動させる診断ツールは、
本来調査すべき訓練そのものを壊しかねません。

チェッカーは、自分が作った解析的勾配もクリアします。実行は訓練ステップの
合間に行ってください。チェック開始時にすでに存在していた勾配は意図的に保存
されません。訓練の勾配と診断の勾配を黙って混合するほうが、空の境界を確立
するよりも危険だからです。

## 5. スケールを考慮した合格条件を定義する

解析的勾配 $g_a$ と数値的勾配 $g_n$ に対して、

$$
e_{abs}=|g_a-g_n|
$$

$$
s=\max(|g_a|,|g_n|)
$$

プローブが合格するのは次のときです。

$$
e_{abs}\le a_{tol}+r_{tol}s
$$

絶対誤差の項はゼロ付近の勾配を扱います。相対誤差の項は、大きな勾配に応じて
誤差がスケールすることを許容します。相対誤差だけを使うと、無害な `1e-10` と
`0` の比較が無限大に見えてしまいます。絶対誤差だけを使うと、大きさ 100 万の
勾配に対して厳しすぎることがあります。

`GradientProbe` は、両方の勾配、フラットな座標と多次元の座標、絶対誤差、
相対誤差、許容誤差、そして観測誤差と許容誤差の比を保持します。そのため失敗
したレポートは、Boolean を 1 つ返すのではなく、問題のパラメータ要素を正確に
特定します。

## 6. 所有権を隠さずに座標をサンプリングする

すべての座標をチェックすると、要素ごとに 2 回の順伝播評価が必要になります。
この小さな MiniGPT でさえ、全数チェックを遅くするのに十分なパラメータ数が
あります。そこでチェッカーは最大 `maximumCoordinatesPerParameter` 個の
インデックスを、先頭と末尾の座標を含めて等間隔に選択します。

サイズ 10 のパラメータで最大数が 3 の場合、選択されるフラットインデックスは
次のとおりです。

```text
0, 4, 9
```

これは決定的であり、オフセットや形状境界のバグがよく現れる両端をカバー
します。ランダムな統計的カバレッジではなく、すべての座標を証明するものでも
ありません。開発中は、サンプリングによるモデル全体のチェックと、各小規模演算
の全数チェックを組み合わせてください。

渡されるすべてのパラメータ参照とラベルは一意でなければなりません。参照が
重複すると、共有されたストレージを二重にプローブし、二重に数えてしまいます。
MiniGPT の `parameters` メソッドは、共有されたトークン埋め込みを意図的に
1 回だけ返します。

## 7. 損失を毎回新しい決定的なグラフとして扱う

チェッカーが受け取るのは `lossFactory: () => Tensor` であって、評価済みの
損失 Tensor ではありません。すべての摂動は、変更後のパラメータ値を使って
新しいグラフを構築しなければなりません。

```scala
GradientChecker.check(
  model.parameters,
  () => model.loss(inputs, targets),
  GradientCheckConfig(maximumCoordinatesPerParameter = 2)
)
```

このファクトリは決定的でなければなりません。ドロップアウト、ランダムな
データサンプリング、可変なバッチ順序、非同期の状態変化があると、プラス側と
マイナス側の損失差が 2 つの変動源を同時に測ることになります。勾配の不一致を
解釈する前に、乱数シードとデータを固定してください。

非平滑な点にも注意が必要です。ReLU はゼロにおいて一意な導関数を持ちません。
摂動が境界をまたぐと、有限差分は両側の挙動を平均しますが、この Tensor
エンジンはゼロより大きい入力に対してのみ正側の導関数を選びます。その不一致は
数学的な点の性質を記述しているのであって、実装の欠陥ではないかもしれません。

## 8. メモリを見積もる前にパラメータを棚卸しする

`ParameterInventory` は、所有するすべての学習可能な Tensor について、ラベル、
形状、要素数、密なペイロードのバイト数を記録します。`Double` の要素数を
$P$ とすると、

$$
M_{parameters}=8P\text{ bytes}
$$

これは訓練メモリの総量では*ありません*。現在のすべて `Double` の AdamW パス
では、永続状態のおおまかな内訳は次のようになります。

```text
parameters       8P
gradients        8P
Adam first       8P
Adam second      8P
-------------------
subtotal        32P bytes
```

これでも、逆伝播のために保持される活性化、一時的な演算バッファ、オブジェクト
や配列のヘッダ、グラフのノード、トークナイザやデータのメモリ、JVM ランタイム、
アロケータの断片化は含まれていません。ペイロードのインベントリが有用なのは、
まさに何を数えているのかを明示するからです。これを「モデルのメモリ」と呼ぶ
のは誤解を招きます。

後の分散処理の章では、同じ内訳を使って、データ並列がどの状態を複製し、
ZeRO の各ステージがどの状態を分割するのかを示します。

## 9. 物語ではなく演算をベンチマークする

反論に耐えるベンチマークは次を定義します。

- 対象の演算と入力の形状
- 計測区間の外で行うセットアップ
- ウォームアップの挙動
- 1 回の計測あたりの演算数
- 生サンプルの数
- 正しさとチェックサムの証拠
- ランタイムとハードウェアの識別情報
- 統計量とばらつき
- その実験では支持できない主張

`Benchmark.measure` は、計測対象のバッチと同じ形のウォームアップバッチを
実行します。ウォームアップは統計から除外され、サンプリングの前に JVM の
クラスロードと just-in-time コンパイルを開始させます。ウォームアップは
コンパイルが安定したことを保証するものではありません。本格的なパフォーマンス
の主張をするときは、より長い実行を調べてください。

1 つの計測区間には複数の演算を含められます。3 回の順伝播からなるバッチに
900,000 ns かかった場合、保存されるサンプルは次のとおりです。

$$
900{,}000/3=300{,}000\text{ ns/op}
$$

バッチ化はタイマー分解能のオーバーヘッドを減らしますが、バッチ化しすぎると
ガベージコレクションの停止や熱による挙動が隠れることがあります。生サンプルは
引き続き参照できます。

## 10. 使われない計算が消えるのを防ぐ

ベンチマーク対象の演算は `Long` を返します。MiniGPT のラボはロジットを計算
し、その値を合計し、`Double` のビットを `Long` に変換し、その値をハーネスが
チェックサムに混ぜ込むようにします。

```scala
val logits = model.logits(inputs)
java.lang.Double.doubleToRawLongBits(logits.values.sum)
```

これにより結果が観測可能になり、ワークロードの意図しない変化を捕捉できます。
ただし、あらゆる JVM 最適化を無効化できたことを証明するわけではありません。
JMH はより洗練された仕組みと分離されたフォークを使います。このコースの
ハーネスは、そのツールを導入する前に、必要な概念を見えるようにするために
存在します。

## 11. タイミングの統計を正しく読む

ハーネスは正規化されたサンプルを保持し、次を報告します。

- 最小値と最大値
- 中央値
- nearest-rank 法による p95
- 算術平均
- 母標準偏差
- 変動係数
- そこから導出される 1 秒あたりの演算数

サンプルが `[1,2,3,4]` ns/op のとき、

$$
\operatorname{median}=\frac{2+3}{2}=2.5
$$

nearest-rank 法による p95 のインデックスは $\lceil0.95\times4\rceil-1=3$
なので、p95 は `4` です。平均は `2.5`、母標準偏差は $\sqrt{1.25}$ です。

変動係数は次のとおりです。

$$
CV=\frac{\sigma}{\mu}
$$

大きな `CV` は、環境またはワークロードが不安定であることを警告します。理由
までは説明しません。考えられる原因には、JIT コンパイル、ガベージ
コレクション、他のプロセス、CPU 周波数の変化、突発的なメモリ確保、入力依存の
分岐、不十分なバッチサイズなどがあります。

30 サンプルに対するローカルな p95 の 1 つの値は、その実行に関する記述的な
証拠にすぎません。サービスレベルのパーセンタイルでも、信頼区間でも、マシンを
またぐ主張でもありません。

## 12. ランタイムフィンガープリントを記録する

`RuntimeFingerprint.current` は、Java ランタイムのバージョン、VM の名前と
バージョン、オペレーティングシステムとそのバージョン、アーキテクチャ、利用
可能なプロセッサ数を記録します。これは、ラベルのない数値同士を比較しないため
に最低限必要なコンテキストです。

より強力なパフォーマンス成果物は、さらに次も記録します。

- リポジトリのコミット
- Nix flake のリビジョン
- 正確なモデル設定とシード
- CPU のモデル、周波数ポリシー、メモリ
- JVM フラグとヒープサイズ
- プロセスのアフィニティと競合する負荷
- ウォームアップと計測のサンプルベクトル
- プロファイラまたはハードウェアカウンタの証拠

この章のラボは、実際に観測したものだけを出力します。GPU やフロンティア級
スループットに関する主張は行いません。

## 実装ウォークスルー

### `BenchmarkStatistics.from`

コンストラクタは、空のサンプル、非正のサンプル、無限大のサンプル、`NaN` の
サンプルを拒否します。順序統計量のためにコピーをソートしますが、平均と分散は
元のベクトルから計算します。レポートは元の計測順序を保持するため、読み手は
ウォームアップのドリフトや後半の停止に気づくことができます。

中央値は、要素数が奇数なら中央の要素を選び、偶数なら中央の 2 要素を平均
します。nearest-rank 法の p95 は `ceil(0.95 * n) - 1` を使い、すべての正の
`n` に対して有効なゼロ始まりのインデックスを生成します。分散を `n` で割る
のは、この値が観測されたベクトルそのものを記述するためです。ハーネスは
不偏な母集団推定量を主張しません。

### `Benchmark.measure`

名前渡しの `operation` は、繰り返しのたびに `runBatch` の内部で評価されます。
ウォームアップは時計を読まずに `runBatch` を呼びます。各計測では、1 バッチの
直前と直後に単調増加クロックを読み、ゼロまたは負の経過時間を拒否し、バッチ
サイズで正規化して、サンプルを 1 つ追加します。

`NanoClock` は、静的呼び出しを直接使うのではなく境界として設計されています。
テストは正確なタイムスタンプを与え、ウォームアップが計測されないこと、
バッチ化が期待どおりの演算回数を実行すること、経過時間が正しく正規化される
ことを証明します。

### `GradientChecker.check`

チェッカーは、作業を始める前にすべてのパラメータベクトルのスナップショットを
取ります。`try` ブロックの内部で勾配をクリアし、スカラー損失を 1 つ構築し、
`backward` を呼び、選択された座標を訪れながら解析的勾配を取得します。

各座標について、`evaluatePerturbation` は 1 要素だけ変更した新しいベクトルを
構築し、Tensor の検証付き変更境界を通して代入し、損失を新しく構築してその
スカラー値を読み取り、自身の `finally` ブロックでパラメータを復元します。
外側の `finally` は、ファクトリや検証のステップが失敗した場合の第二の防衛線
として*すべての*スナップショットを復元し、その後、診断用の勾配をすべて
クリアします。

数値微分、誤差、許容誤差、座標、ラベルは `GradientProbe` にまとめられます。
`GradientCheckReport.worstProbe` は、観測誤差と許容誤差の比が最大のものを
選びます。これは、絶対差が最大のものだけを選ぶより多くの情報を与えます。

### `ParameterInventory.from`

インベントリは、空の入力、非正の要素サイズ、重複した Tensor 参照、重複した
ラベルを拒否します。ペイロードの乗算には `Math.multiplyExact` を使うため、
整数オーバーフローはラップされた負のバイト数ではなく明示的な失敗になります。

### `runMiniGptDiagnostics`

このラボは、シード固定の MiniGPT を 1 つと、固定のトークンとターゲットのペア
を 1 組構築します。パラメータの所有権を出力し、Tensor ごとに 2 つのプローブを
実行し、その後、決定的な順伝播をベンチマークします。勾配チェックは計測の前に
行われます。チェッカーの最終的な復元により、計測対象のモデルが元の重みを保持
していることが保証されます。

## テストの読み方

`GradientCheckSuite` は手計算できる二次関数から始まります。これは独立した
オラクルです。その勾配は Tensor の自動微分を信用しなくてもわかります。別の
テストでは、三次関数に対して意図的に大きな `h` とゼロの許容誤差を使います。
期待される `0.01` の不一致は、失敗したレポートが例外を投げたり false を返し
たりするだけでなく、証拠を保持することを証明します。

失敗注入のテストは、最初の摂動された損失の構築中に例外を投げ、その後
パラメータが厳密に復元されていることをアサートします。サンプリング版の
MiniGPT テストは、埋め込み、attention、フィードフォワード、
正規化、共有ヘッド、交差エントロピーからなる完全なグラフを走査します。これは
以前のスイートの全数演算チェックを補完するものであり、置き換えるものでは
ありません。

`BenchmarkSuite` は `[4,1,3,2]` から統計量を手計算します。スクリプト化された
時計は、それぞれ 2 演算を含む 60 ns と 80 ns の計測バッチを生成するため、
サンプルは `[30,40]` でなければなりません。演算カウンタは、計測されない
ウォームアップバッチ 1 回と計測バッチ 2 回で、ちょうど 6 回の演算が実行された
ことを証明します。

失敗系のテストは、不正な反復回数と増加しない時計を拒否します。経過時間ゼロの
結果が黙って無限のスループットに変換されることはありません。

## ラボを実行して解釈する

```console
$ nix develop -c sbt 'runMain learnai.transformer.runMiniGptDiagnostics'
```

次のフィールドを順に確認してください。

1. **パラメータエントリ:** ラベルと形状が第 21 章のアーキテクチャに対応し、
   共有された埋め込みが 1 回だけ含まれていること。
2. **勾配プローブ:** `passed=true`、最大絶対誤差、誤差と許容誤差の比の最大値
   は、出力された設定のもとでサンプリングされた一致を示すものであり、すべて
   の入力を証明するわけではありません。
3. **最悪のプローブ:** 許容誤差を変更する前に、そのパラメータと座標を特定
   すること。
4. **中央値/p95/CV:** 1 回のローカルなタイミング観測として扱うこと。性能の
   退行を主張する前に、再実行して生サンプルを比較すること。
5. **チェックサム:** 同じセットアップであれば、同じ計算量と出力の同一性が
   保たれるはずです。
6. **ランタイム:** このコンテキストを捨てたままタイミングを比較しないこと。

勾配チェックの合格が確立するのは、サンプリングされた座標におけるリバース
モードと中心差分の局所的な一致です。モデルの品質、多数ステップにわたる
オプティマイザの正しさ、データの妥当性、本番のパフォーマンスを確立するもの
ではありません。

## デバッグチェックリスト

### 勾配の不一致

1. 問題を、そのパラメータを含む最小の演算まで縮小する。
2. 解析的勾配、プラス側の損失、マイナス側の損失、数値的勾配を出力する。
3. `h` を `1e-2` から `1e-7` まで振る。表示上の誤差が最小のステップを選ぶ
   のではなく、安定した領域を探す。
4. ReLU などの非平滑な境界がないか確認する。
5. シード、ドロップアウト、データ順序、可変キャッシュを固定する。
6. 摂動させたパラメータが新しいグラフに参加していることを検証する。
7. 共有された経路が勾配を上書きせず累積していることを検証する。
8. 許容誤差を変更するのは、数値誤差の原因を特定してからにする。

### 疑わしいタイミング

1. チェックサムとモデル設定がリファレンス実行と一致することを確認する。
2. サンプルがタイマー分解能に近づく場合は、1 回の計測あたりの演算数を増やす。
3. ウォームアップを増やし、サンプルを実行順にプロットする。
4. JFR や専用プロファイラでメモリ確保とガベージコレクションを調べる。
5. セットアップ、トークン化、モデルの順伝播、サンプリング、出力の描画を
   分離する。
6. 実時間より先に決定的な作業カウンタを比較する。
7. 公表に耐える JVM マイクロベンチマークにはフォーク付きの JMH を使う。
8. 必要な環境の証拠がない場合は `No measurements found` と報告する。漸近的な
   計算量だけから速度を推測しない。

### 誤ったメモリ合計

1. 問いがパラメータのペイロード、永続的な訓練状態、生存テンソルのピーク、
   プロセスの RSS、デバイスアロケータの予約のどれに関するものかを決める。
2. 合計する前に、共有パラメータの同一性を確認する。
3. dtype ごとの要素あたりバイト数を確認する。
4. 勾配、オプティマイザのモーメント、マスター重み、活性化を、名前を付けた
   別々の項として加算する。
5. オブジェクトとアロケータのオーバーヘッドは、ペイロードの計算に紛れ
   込ませず測定する。

## プロフェッショナルなワークフローとのつながり

後で登場する最適化パスや分散パスはすべて、2 つの実装を維持しなければ
なりません。

```text
clear reference path
  + independent correctness oracle
  + optimized path
  + equivalence tests
  + workload counters
  + environment-labeled measurements
```

FlashAttention、量子化カーネル、grouped-query のキャッシュ、テンソル並列の
行列積、投機的デコーディングは、実行の構造を変えます。新しいパスが速く見える
というだけの理由で、リファレンスパスを削除してよいものは 1 つもありません。

継続的インテグレーションでは決定的な正しさのチェックを実行すべきです。
パフォーマンスのゲートは、安定した専用ランナーが所有すべきです。ノイズの
多いラップトップのタイミングを通常のユニットテストに混ぜると、不安定な
ビルドを生み、誤った証拠のモデルを教えることになります。

## 演習

1. 前進差分モードを追加し、`h` を 10 のべき乗で変えながら、その誤差を中心
   差分と比較してプロットする。
2. 明示的なシードを持つランダム座標サンプラーを追加し、サンプリングされた
   インデックスをレポートに保持する。
3. 活性化を分けたまま、SGD と AdamW の訓練状態の物理バイト数見積もりを
   追加する。
4. ベンチマークの生サンプルとランタイムのメタデータを、バージョン付き JSON
   としてエクスポートする。
5. ラボの実行を Java Flight Recorder で囲み、計測対象の演算を変えずに最大の
   アロケーション箇所を特定する。
6. トークンとロジットの等価性をアサートしてからはじめて、完全プレフィックス
   とキャッシュ版の順伝播レイテンシを比較する。
7. 独立した分析レイヤーにブートストラップ信頼区間を実装し、相関を持つ JVM の
   サンプルが単純な独立性の仮定に反する理由を説明する。

## 完了基準

次のことができれば完了です。

- 数学的な正しさ、数値的な正しさ、システムとしての正しさを区別できる
- 中心差分を導出し、競合する誤差の要因を説明できる
- スケールと精度から絶対許容誤差と相対許容誤差を選択できる
- 保持された座標レベルの証拠から失敗したプローブを診断できる
- 成功時と失敗時のパラメータ復元を証明できる
- 密なパラメータのペイロードと訓練メモリの総量を区別できる
- ウォームアップ、バッチ化、観測可能な結果、生サンプル、中央値、p95、
  ばらつきを説明できる
- ローカルベンチマークが証明しないことを述べられる
- 最適化を受け入れる前にリファレンスとの等価性を要求できる
