# 03 — 依存ライブラリなしのテストとデバッグ

## この章で作るもの

関数を実行し、値を比較し、失敗を報告し、ビルドを失敗させる、最小限のテスト
ランナーを作ります。ソースは `src/test/scala/learnai/testing` 以下にあります。

本番向けの Scala プロジェクトでは通常テストライブラリを使います。ここで最初の
ランナーを自作するのは、テストが入力・実行・比較・報告からなる普通のプログラムで
あることを示すためです。

## Arrange・Act・Assert

```scala
test("mean averages non-empty values") {
  // Arrange
  val values = Vector(2.0, 1.0, 0.5)

  // Act
  val result = ScalaTour.mean(values)

  // Assert
  Assert.close(Assert.right(result), 7.0 / 6.0)
}
```

各テストは一つの振る舞いを検査します。テスト名は、本体を読まなくてもどの性質が
壊れたのか分かるように付けるべきです。

## テスト本体を関数として保持する

```scala
final case class TestCase(name: String, run: () => Unit)
```

`() => Unit` は、副作用を実行しうる引数なしの関数です。ヘルパーの
`body: => Unit` は名前渡し (by-name) 引数なので、`TestCase` を構築してもテストは
実行されません。いつ呼び出すかはランナーが決めます。

したがってランナーは次のことができます。

1. テストを選択する。
2. `run()` を呼び出す。
3. 正常に戻れば成功として記録する。
4. 例外が出れば失敗として記録する。
5. すべての結果を集計する。

## 浮動小数点の結果を盲目的に比較しない

多くの実数は有限の二進表現を持ちません。

```scala
0.1 + 0.2 == 0.3 // may be false
```

絶対差を、根拠のある許容誤差 (tolerance) と比較します。

$$
|\text{actual}-\text{expected}|\leq\text{tolerance}
$$

```scala
Assert.close(actual, expected, tolerance = 1e-9)
```

許容誤差が大きすぎると欠陥を隠してしまいます。値のスケール、演算の回数、想定
される誤差の発生源から選んでください。後の章では絶対許容誤差と相対許容誤差を
組み合わせます。

## 失敗パスをテストする

```scala
test("mean rejects an empty collection") {
  val result = ScalaTour.mean(Vector.empty)
  Assert.equal(Assert.left(result), "mean requires at least one value")
}
```

数値計算の章では、形状 (shape) の不一致、不正なインデックス、空のテンソル、
非有限値を意図的にテストします。エージェントの章では、不正なツール引数、
タイムアウト、使い切った予算、破損したモデル応答をテストします。

## 明示的なスイート登録

`AllTests` には完全なスイート一覧が含まれています。

```scala
private val suites: Vector[TestSuite] = Vector(
  ScalaTourSuite
)
```

リフレクションによる自動発見はありません。新しいスイートは明示的に登録する必要が
あり、実行対象の集合が一つのファイルで見渡せるようになっています。

## テストを実行する

```console
$ nix develop -c sbt test
[pass] ScalaTour / square multiplies a value by itself
...
5 passed, 0 failed, 5 total
```

クリーンなコンパイルと全テストの実行には次を使います。

```console
$ nix develop -c sbt check
```

## 意図的な失敗を観察する

square の期待値を一時的に `9.0` から `8.0` に変更してスイートを実行し、次を
確認してください。

- スイート名とテスト名。
- 期待値と実際の値。
- 非ゼロの sbt 実行結果。

その後 `9.0` に戻してください。テストが失敗する様子を見ることで、それが成功して
いる状態が何を保証しているのかが明確になります。

## 実装ウォークスルー

`TestCase` は名前と引数なしの関数を保持します。

```scala
final case class TestCase(name: String, run: () => Unit)
```

`test` ヘルパーは本体を名前渡しで受け取ります。つまりアサーションの本体は
スイートオブジェクトの初期化中には実行されず、`() => body` に包まれて、後で
ランナーのエラー処理のもとで実行されます。ここで即時実行してしまうと、
`AllTests` がスイート名とケース名を報告する前に例外が投げられてしまいます。

`Assert.equal` がジェネリックなのは、等価性の比較が数値・ベクトル・列挙型・
ドメイン値のいずれにも有用だからです。`Assert.close` は意図的に `Double` 専用に
しています。許容誤差の契約を検査し、`NaN` を拒否し、一致しない無限大を区別し、
観測された差を出力します。これは `math.abs(a-b) <= tolerance` より厳密です。
後者は一部の不正な値をうっかり成功した比較として扱ってしまいます。

`Assert.throws[E]` は `ClassTag` から実行時クラスを取得します。その 3 つの経路が
重要です。

1. 例外なし: 不正な入力が受理されたため失敗。
2. 期待した例外型: テストがメッセージを検査できるよう、その例外を返す。
3. 異なる例外型: 失敗とし、診断用に型とメッセージを保持する。

`AllTests` はスイートとケースの直積を走査して `runOne` を呼び、Boolean の結果を
数えます。失敗したケースは即座に出力されますが、残りのケースも実行され続けます。
完全なレポートを出し終えた後で初めてランナーが例外を投げ、sbt に非ゼロのプロセス
ステータスを渡します。

## 回帰テストの読み方と書き方

メソッド名ではなく、バグの記述から始めてください。

> 再利用された計算グラフのノードが、勾配の寄与を一つ失う。

次に、正しい実装とバグのある実装で結果が異なる最小の入力を作り、答えを独立に
計算し、その振る舞いに名前を付けます。

```scala
test("a reused node accumulates gradient from every path") { ... }
```

有用なテストは、ただ一つの特定の理由で失敗します。大きなモデルを初期化し、
ランダムに訓練 (training run) し、シリアライズして一つの Boolean を検査するような
テストでは、失敗しても原因の特定にほとんど役立ちません。焦点を絞ったユニット
テストを保ち、統合テストは別に追加してください。

## デバッグチェックリスト

1. スイートが `AllTests` の出力に現れることを確認する。
2. 最小の失敗ケースを、手計算できる値で頭の中で実行してみる。
3. `Assert.close` が失敗したら、スケール、想定される誤差の発生源、許容誤差を
   出力する。闇雲に許容誤差を広げない。
4. `Assert.throws` が異なる例外で失敗したら、バリデーションを公開境界の近くに
   移動する。
5. テスト同士が影響し合うなら、共有された可変状態や再利用された RNG 状態を探す。
6. タイムアウトのテストがハングするなら、ワーカースレッドが割り込まれ、
   executor が `finally` でシャットダウンされることを確認する。

## 演習

1. `Assert.isTrue` を使うテストを一つ追加してください。
2. 空文字列で `normalizeLabel` をテストしてください。
3. レジストリからスイートを一つ削除し、実行数の変化を観察してください。
4. `Assert.close(Double.NaN, Double.NaN)` が失敗する理由を説明してください。

## 完了基準

- 既存のテストの中で Arrange・Act・Assert を識別できる。
- 浮動小数点の比較に許容誤差が必要な理由を説明できる。
- 正常系のテストと失敗パスのテストを一つずつ追加できる。
- `sbt check` が成功する。
