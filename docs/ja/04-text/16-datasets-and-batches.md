# 16 — 因果的データセット、分割、バッチ

## この章で作るもの

固定長の次トークン予測サンプル、重なりのない訓練/検証領域、再現可能な
ランダムバッチ、決定的な評価バッチを作ります。ソース:
`src/main/scala/learnai/data/CausalDataset.scala`。

## 因果的言語モデリング

次のトークン列が与えられたとします。

```text
[t0,t1,t2,t3,t4]
```

コンテキスト長 3 のサンプルは次のようになります。

```text
input:  [t0,t1,t2]
target: [t1,t2,t3]
```

`input(i)` の後の出力が `target(i)` を予測します。したがって 1 つの
ウィンドウは、時刻位置ごとに 1 つの次トークンターゲットを提供します。
次のスライディングウィンドウは:

```text
input:  [t1,t2,t3]
target: [t2,t3,t4]
```

長さ $N$ の系列とコンテキスト $T$ からは
$\max(0,N-T)$ 個の完全なウィンドウが得られます。

## 形状

1 つのサンプル:

```text
inputs:  [time]
targets: [time]
```

$B$ 個のサンプルからなるバッチ:

```text
inputs:  [batch,time]
targets: [batch,time]
```

`CausalBatch` は単一のコンテキスト長を強制します。可変長のバッチ処理を
後で行うには、パディングと明示的なマスクが必要になります。

## ウィンドウ化の前に生トークンを分割する

ランダム分割の前に重なり合うウィンドウを作ってしまうと、訓練と検証が
ほぼすべてのトークンを共有しかねません。

```text
train:      [t0,t1,t2,t3]
validation:    [t1,t2,t3,t4]
```

このリーク (leakage) は汎化性能を過大評価させます。まず生の系列を連続した
領域に分け、その後で各側を独立にウィンドウ化します。

```text
tokens -> [training region | validation region]
             -> windows       -> windows
```

境界付近のサンプルはいくらか失われますが、分割をまたぐサンプルは
存在しません。実際のデータセットでは、多くの場合、文書やソースの同一性で
分割し、パーティション間の重複を除去すべきです。

## バッチは全体の勾配を推定する

すべてのサンプルを毎ステップ計算するのは高コストです。バッチは平均損失
(loss) を推定します。

$$
\hat L=\frac{1}{B}\sum_{i\in\text{batch}}L_i
$$

- 小さいバッチ: メモリが少なく、勾配 (gradient) のノイズが大きい。
- 大きいバッチ: 推定が安定するが、メモリとステップあたりの計算が増える。

バッチサイズはスループットだけでなく、訓練のダイナミクスも変えます。

## 復元抽出によるサンプリング

`sampleBatch` は各要素を独立に選ぶため、同じサンプルが複数回現れることが
あります。エポック方式のシャッフル反復も、もう 1 つの一般的な方針です。

どの方針であれ、再開した訓練でデータ順序を保つ必要があるなら、シードと
サンプラの状態をチェックポイントに保存してください。

## 決定的な検証

検証は `sequentialBatches` を使い、パラメータ更新は行いません。短い最終
バッチを残すか捨てるかは呼び出し側が選びます。ドロップアウトや訓練依存の
正規化を持つモデルには、明示的な評価モードも必要です。

## パイプラインの不変条件

テストは次を検証します。

- すべてのターゲットがちょうど 1 トークン先である。
- 不完全なウィンドウは除外される。
- 分割境界をまたがない。
- 同じシードは同じバッチを生む。
- 空のデータセットからのサンプリングは明示的に失敗する。
- バッチ内のすべてのサンプルが同じ形状を持つ。

データの欠陥はモデルの欠陥のように見えることがあり、リークは無効な評価の
まま損失を下げ続けることがあります。データパイプラインはモデルのコードと
同じ厳格さでテストしてください。

## 実装ウォークスルー

`CausalDataset.fromTokens` は、1 つのトークン列の上を `contextLength + 1`
のウィンドウでスライドします。先頭の `contextLength` 個の ID が入力で、
同じウィンドウを 1 つずらしたものがターゲットです。

```text
tokens:  [10, 11, 12, 13]
context: 3
input:   [10, 11, 12]
target:  [11, 12, 13]
```

`N` 個のトークンとコンテキスト `T` に対して、完全なサンプルは
`max(0, N-T)` 個あります。最後の有効な開始位置は `N-T-1` です。この個数を
先に書き出しておくことが、最もよくある off-by-one バグを防ぎます。

`sampleBatch` は呼び出し側の RNG を受け取り、サンプルのインデックスを
復元抽出で選びます。復元抽出は呼び出しを単純に保ち、要求されたバッチ
サイズがデータセットサイズを超えた場合の挙動も定義します。空のデータ
セットは、有効なインデックスが存在しないため `Left` を返します。
`sequentialBatches` は代わりに宣言順のスライスを行い、短い最終バッチを
残すかどうかを明示的に選びます。

`contiguousSplit` は、ウィンドウを構築する前に元のトークンストリームを
分割します。作成済みのサンプルを分割すると、境界付近で検証サンプルの
入力やターゲットが訓練トークンと重なる可能性があります。分割は 2 つの
独立したデータセットを所有し、トークンの境界位置を報告します。

## テストの読み方

小さな系列 1 つで、整列したウィンドウを正確に列挙します。コンテキスト
以下の長さの系列で、サンプル数ゼロの境界を検証します。分割のテストは
ソース範囲を調べ、ウィンドウが境界をまたがないことを保証します。同じ
RNG シードは同じバッチを選ばなければなりません。空のサンプリングは
`Either` のエラーを確認し、逐次バッチ処理は両方の `dropLast` モードを
カバーします。

## デバッグチェックリスト

1. ループを書く前に `N`、`T`、期待される `N-T` 個のサンプル数を書き出す。
2. 最初と最後の開始インデックスについて入力/ターゲットのペアを出力する。
3. ウィンドウ構築の前にトークンを分割する。
4. シード付きバッチが一致しない場合は、RNG の所有権とそれまでの呼び出しを比較する。
5. サンプリングが復元抽出か非復元抽出かを決めてテストする。

## 演習

1. 長さ 10、コンテキスト 4 の最初と最後のウィンドウを列挙する。
2. エポックベースのシャッフルサンプラを実装する。
3. 文書をまたぐウィンドウを作らない `fromDocuments` を設計する。
4. トークン予算に基づく動的バッチ処理を設計する。
5. 分割境界で失われるトークン数とサンプル数を報告する。
6. ウィンドウをランダム分割した場合の重複リークを測定する。

## 完了基準

- 入力とターゲットの 1 トークンずれを説明できる。
- `[batch,time]` の両方の軸の名前を言える。
- ウィンドウ化の前に分割することがリークを防ぐ理由を実演できる。
- バッチサイズと勾配ノイズの関係を説明できる。
- 訓練時のサンプリングと検証時の反復を区別できる。
- `CausalDatasetSuite` が通る。
