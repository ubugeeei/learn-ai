# 14 — Unicode、UTF-8、バイトトークナイザ

## この章で作るもの

Scala の `String` を UTF-8 バイトのトークン ID に変換し、不正なバイト列を
黙って置換するのではなく明示的に拒否する処理を実装します。ソース:
`src/main/scala/learnai/text/Utf8.scala`。

## モデルは文字列を直接消費しない

ニューラルネットワークが消費するのは数値です。トークナイザ (tokenizer) は、
テキストとトークン ID の境界に位置します。

```text
text --encode--> token IDs --model--> token IDs --decode--> text
```

トークン化は、語彙サイズ、系列長、訓練効率、多言語での挙動、生成される
テキスト、そしてモデルのパラメータ数に影響します。

## 「文字」とは何か

混同しやすい単位がいくつかあります。

- **書記素クラスタ (grapheme cluster)**: 読み手が 1 つの記号として知覚するもの。
- **Unicode コードポイント (code point)**: `U+03B1` のような番号。
- **UTF-16 コードユニット**: JVM の文字列 API が公開する 16 ビット単位。
- **UTF-8 バイト**: ファイルやネットワークで広く使われる 8 ビット単位。

目に見える 1 つの記号が、複数のコードポイントを含むことがあります。また、
見た目が同じテキストでも、正規化されたコードポイント列が異なる場合が
あります。`String.length` は普遍的な文字数ではありません。

## Unicode と UTF-8

Unicode はコードポイントを割り当てます。UTF-8 はコードポイントの列を
バイトとして符号化します。1 つのコードポイントは 1〜4 バイトの UTF-8
バイトを使います。

| テキスト | コードポイント | UTF-8 バイト | バイトトークン数 |
| --- | --- | --- | --- |
| `A` | `U+0041` | `41` | 1 |
| `α` | `U+03B1` | `CE B1` | 2 |
| `🚀` | `U+1F680` | `F0 9F 9A 80` | 4 |

JVM では `Byte` は符号付きです。`byte & 0xff` によって、符号なしのトークン
範囲 `[0,255]` に変換します。

## バイトトークン化があらゆる UTF-8 テキストで機能する理由

トークン ID `0..255` をバイト値にそのまま割り当てます。

- 固定された小さな語彙。
- 有効な UTF-8 に対して未知トークンが存在しない。
- 可逆な符号化。
- 単純な実装。

代償は系列長です。多くの文字体系や絵文字は、目に見える記号 1 つあたり
複数のバイトを必要とします。attention の計算コストは系列長の
2 乗で増えるため、次の章では頻出するバイト列を BPE でマージします。

## 特殊トークン

構造を表すトークンは、符号化されたテキストと衝突してはいけません。

```text
0..255: byte values
256:    beginning of text
257:    end of text
```

本番のプロトコルには、ロール、パディング、ツール呼び出しのトークンが
存在することもあります。トークンを 1 つ追加すると語彙サイズが変わり、
その結果として埋め込み (embedding) と出力の形状も変わります。

## 厳格なデコード

すべてのバイト列が有効な UTF-8 とは限りません。`0x80` 単独は、先頭バイトの
ない継続バイトです。置換文字を使うデコーダでは、モデルが不正な列を生成した
事実が隠れてしまいます。

このコースでは `CodingErrorAction.REPORT` を設定し、
`Either[String,String]` を返します。ストリーミング UI は不完全な末尾を
次のトークンまでバッファしてもかまいませんが、生のトークンとエラーは
観測可能なままでなければなりません。

## 往復変換の不変条件

有効なテキストに対して:

$$
\operatorname{decode}(\operatorname{encode}(text))=text
$$

ASCII、複数の文字体系、絵文字、結合記号、改行、空文字列をテストします。
このトークナイザは正規化を一切行わないため、元の UTF-8 バイトをそのまま
保持します。

## 不透明なトークン ID

```scala
opaque type TokenId = Int
```

実行時の表現は `Int` ですが、API 上は独立した意味を持つ型です。長さや
オフセットを誤って渡してしまう可能性が下がり、構築時に負の ID を拒否
します。

## 実装ウォークスルー

`Utf8.encodeBytes` は文字からバイトへの変換を JDK の UTF-8 実装に委譲し、
その後 `byte & 0xff` で符号付きの JVM バイトを符号なし整数に写します。
Java の `Byte` の範囲が `-128..127` であるにもかかわらず、基本トークン ID
がちょうど `0..255` を占めるのはこのためです。

`decodeBytes` は、すべての整数がバイト範囲に収まることを検証した後で
初めて逆変換を行います。`CharsetDecoder` は、置換文字を挿入する代わりに
不正な入力とマップ不能な入力を報告するよう設定されています。黙った置換を
許すと、不正なトークン列を壊しているのに `decode(encode(text))` が成功
しているように見えてしまいます。

具体例を 1 つ手で追ってみましょう。ASCII の `A` は 1 バイト `0x41` に
なります。ユーロ記号は 3 バイト `E2 82 AC` に、ロケットの絵文字は
4 バイト `F0 9F 9A 80` になります。したがって、目に見える文字数、Unicode
コードポイント数、UTF-16 コードユニット数、UTF-8 バイト数はそれぞれ
異なる量です。

`ByteTokenizer.encode` は各符号なしバイトを `TokenId` に写し、必要に応じて
`0..255` の外にある開始・終了の特殊 ID を追加します。デコードは、明示的な
フラグに応じて、既知の特殊トークンをスキップするか拒否するかを選びます。
`TokenId` は構築時に負の ID を防ぎますが、語彙固有の上限はモデルまたは
トークナイザの責任のままです。

## テストの読み方

往復変換のテストには ASCII、アクセント付きラテン文字、絵文字が含まれて
おり、1 バイト文字しか扱えないコードは通過できません。明示的なバイト数の
テストは 1 バイト、3 バイト、4 バイトのケースを検証します。単独の継続
バイトは、不正な UTF-8 を検出するコンパクトなオラクルです。特殊トークンの
テストはスキップと厳格の両方の挙動を網羅します。負の ID の拒否は、後続の
すべての埋め込みルックアップを守ります。

## デバッグチェックリスト

1. バイトは符号付き 10 進ではなく、符号なし 16 進で出力する。
2. UTF-16 の `String.length` を、Unicode コードポイントおよび UTF-8 バイトと区別する。
3. 信頼できないバイトをデコードする前に、デコーダのエラー動作を設定する。
4. 往復変換が失敗したら、トークナイザの ID より前に生のバイトを比較する。
5. 特殊トークンの ID はバイト語彙の外に置き、その意味をバージョン管理する。

## 演習

1. 自分の名前を符号化し、各バイトを 16 進で出力する。
2. 空のテキスト、改行、NUL の往復変換テストを追加する。
3. `String.length`、コードポイント数、UTF-8 バイト数を比較する。
4. マルチバイト記号の不完全な先頭部分をデコードしてみる。
5. 特殊トークンを構造として保持するデコーダの戻り値を設計する。

## 完了基準

- 書記素、コードポイント、UTF-8 バイト、トークンを区別できる。
- バイトトークナイザに未知トークンが存在しない理由を説明できる。
- 一部の文字体系が ASCII より長いバイト列を使う理由を説明できる。
- 厳格なデコードがインシデント分析に役立つ理由を説明できる。
- 多言語の往復変換を検証できる。
- `Utf8Suite` が通る。
