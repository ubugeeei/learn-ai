# 15 — バイトペア符号化

## この章で作るもの

最も頻度の高い隣接トークンペアを繰り返しマージすることでバイトレベルの
BPE 語彙を訓練し、学習したマージテーブルで符号化と復号を行います。
ソース: `src/main/scala/learnai/text/BpeTokenizer.scala`。

## 語彙サイズと系列長のトレードオフ

バイト語彙のエントリはわずか 256 個ですが、しばしば長い系列を生みます。
反対の極端として、文書ごとに 1 トークンを割り当てれば系列は短くなりますが、
未知の文書には一般化できません。

- 語彙が小さい: 埋め込み (embedding) と出力のパラメータが少ない、系列が長い。
- 語彙が大きい: 系列が短い、パラメータと希少トークンが多い。

BPE は、あらゆる文字を扱えるバイトのカバレッジから出発し、コーパスで
繰り返されるパターンだけを追加します。

## 訓練アルゴリズム

初期の系列は UTF-8 バイトの ID です。

```text
"banana" -> [98,97,110,97,110,97]
```

目標の語彙サイズに達するまで次を繰り返します。

1. すべての系列で隣接ペアを数える。
2. 最も頻度の高いペアを選ぶ。
3. 新しいトークン ID を割り当てる。
4. すべての系列で、重ならない出現を置き換える。

`an` のバイト列である `(97,110)` が最も頻度が高い場合:

```text
[98,97,110,97,110,97]
 -> [98,256,256,97]
```

後のマージは以前のマージ結果のトークンを参照できるため、`ban` のような
断片、単語、さらには複数語のまとまりが自然に現れます。

## マージテーブルはモデル成果物の一部である

```text
merge 0: (97,110) -> 256
merge 1: (98,256) -> 257
```

マージの順序が符号化を定義します。結果 ID `256+i` はバイトか、それより
古いマージ結果のみを参照するため、有向非巡回の展開グラフが構成されます。

異なるマージテーブルの下では重みは無意味です。トークン ID がもはや同じ
バイト列を表さないからです。デプロイ可能なバンドルには、トークナイザの
種類、マージ、特殊トークン、正規化ルール、バージョン、チェックサムが
含まれます。

## 符号化

```text
UTF-8 bytes
 -> apply merge 0 everywhere
 -> apply merge 1 everywhere
 -> ...
 -> token IDs
```

この教育用実装は、系列とマージテーブルを直接走査します。本番の
トークナイザは、マージランク、ペアのインデックス、優先度付きキューを
使います。

## 復号

バイトトークンの展開はそのバイト自身です。マージされたトークンは、
左右の展開結果を連結して展開します。

```text
bytes(256) = bytes(97) ++ bytes(110) = [97,110]
```

すべてのトークンを展開した後、厳格な UTF-8 デコードでテキストを復元
します。

## 決定的なタイブレーク

ペアの頻度が等しい場合の選択は、ハッシュマップの反復順に依存しては
いけません。この訓練器は次の基準で選びます。

1. カウントが大きい方。
2. 左の ID が小さい方。
3. 右の ID が小さい方。

コーパスの内容と順序、前処理、目標サイズ、タイブレークの方針は、
すべて再現性を左右する入力です。

## Unicode の境界

バイトレベルの BPE は、コードポイントや書記素の境界を知りません。
1 つのトークンがコードポイントの一部を含むことも、目に見える記号を
複数含むこともあります。それでも完全なトークン列は往復変換できます。
バイトの連結結果が変わらないからです。

ストリーミングデコーダは、各トークンを独立に復号するのではなく、
UTF-8 の状態を保持しなければなりません。

## 計算量と本番運用上の関心事

この参照実装の訓練器は、マージのたびにコーパス全体の再カウントと置換を
行い、$M$ 回のマージと $N$ 個のコーパストークンに対しておよそ $O(MN)$ です。
大規模な訓練には、増分カウント、出現位置のインデックス化、並列化、
外部メモリが必要になります。

本番の設計ではさらに、正規化、事前トークン化、特殊トークンの衝突ルール、
バイトフォールバック、ストリーミング、成果物の互換性も規定します。

## 実装ウォークスルー

訓練は UTF-8 バイトごとに 1 トークンの状態から始まります。各イテレーション
で、`BpeTrainer` はコーパスの各項目ごとに独立して隣接トークンペアを数え
ます。最高頻度のペアを選び、同数の場合はペアのトークン ID を使うため、
頻度が等しいコーパスでもどの JVM 上でも同じマージテーブルが得られます。

`banana` の場合、初期のバイト記号列は `b a n a n a` です。隣接カウントは
次のとおりです。

```text
(b,a): 1
(a,n): 2
(n,a): 2
```

決定的なタイブレークルールは、`(a,n)` と `(n,a)` のどちらかを ID 順で
選びます。選ばれたペアは左から右へ新しいトークン ID に置き換えられます。
重ならない置換であることが重要です。`aaa` の中で `(a,a)` をマージすると、
マージ済みトークン 1 つと `a` 1 つになるのであって、重なった 2 つの結果に
なるわけではありません。

各 `BpeMerge` は左、右、結果の ID を保持します。`fromMerges` は、新しい
結果がトポロジカルに順序付いていること、つまり両方の入力がすでに存在する
ことを検証します。復号は学習済みトークンを再帰的に基本バイトまで展開し、
その後厳格な UTF-8 デコードを行います。符号化は学習済みマージをテーブル
順に適用し、訓練時の語彙構築と一致させます。

語彙は 256 個の基本バイト ID から始まります。訓練は、目標に達するか、
ペアが残っていないか、コーパスが短すぎる場合に停止します。空の項目や
1 トークンの項目は、後で特殊ケースを引き起こすのではなく、単にペアを
提供しません。

## テストの読み方

Unicode の往復変換テストは、マージがテキストを失うことなくバイト上で
動作することを検証します。`banana` の繰り返しは、頻出ペアがトークン数を
減らすことを証明します。頻度が等しい訓練を 2 回繰り返すことで、
非決定的なマップ反復を検出します。ペアのないコーパスは早期終了を確認
します。不正なトークン ID や順序の狂ったマージテーブルは、訓練だけで
なく成果物の検証を確かめます。

## デバッグチェックリスト

1. 最初のマージの前にバイトトークン列を出力する。
2. ペアはコーパス項目ごとに別々に数える。文書境界をまたいではならない。
3. タイブレークの順序を明示的に規定する。
4. 置換が左から右へ、重ならずに行われることを検証する。
5. 復号が失敗したら、学習済み ID を 1 つ再帰的に展開し、最終バイトを調べる。
6. マージテーブルは、その ID を消費するモデルと一緒にバージョン管理する。

## 演習

1. `banana` のすべてのペアを数え、最初のマージを予測する。
2. いくつかの語彙サイズについて平均トークン数を測定する。
3. 自然言語コーパスとコードコーパスで圧縮率を比較する。
4. 順序を定めないタイブレークがどのように非決定性を引き起こすかを説明する。
5. マージテーブルの直列化と往復変換テストを設計する。
6. 単独では有効な UTF-8 にならない学習済みトークンを見つける。

## 完了基準

- 語彙と系列のトレードオフを説明できる。
- 小さな例でカウント、選択、マージを実行できる。
- マージの順序付けが符号化と復号を支える理由を説明できる。
- バイトレベルのトークンが Unicode の境界と一致しなくてよい理由を説明できる。
- トークナイザ成果物と重みを一緒にバージョン管理すべき理由を説明できる。
- `BpeTokenizerSuite` が通る。
